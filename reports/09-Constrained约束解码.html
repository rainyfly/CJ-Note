2.2 BaseGrammarObject 抽象

BaseGrammarObject 是所有语法对象的基类，定义了约束解码过程中语法状态机需要实现的核心接口：

核心方法职责：

方法职责

2.3 BaseGrammarBackend 后端抽象

BaseGrammarBackend 负责语法对象的编译和缓存管理：

class BaseGrammarBackend:
    def __init__(self):
        self.executor = ThreadPoolExecutor()   # 异步编译线程池
        self.cache: Dict[Tuple[str, str], CacheEntry] = {}  # 编译缓存

    def get_cached_or_future_value(self, key, require_reasoning):
        value = self.cache.get(key)
        if value:
            return value.copy(), True   # 缓存命中，返回副本
        # 缓存未命中，提交到线程池异步编译
        value = self.executor.submit(self._init_value_dispatch, key, require_reasoning)
        return value, False

分发机制通过 _init_value_dispatch() 根据 key 类型路由到对应的 dispatch_json/dispatch_regex/dispatch_ebnf/dispatch_structural_tag 方法。

后端注册机制 — SGLang 提供了 GRAMMAR_BACKEND_REGISTRY 字典和 register_grammar_backend() 函数，允许第三方注册自定义 grammar 后端。create_grammar_backend() 优先查找注册表，找不到时回退到内置后端。

3. XGrammar 后端

XGrammar 是 SGLang 的默认约束解码后端，基于高性能 C++ 库 xgrammar，提供快速的语法编译和 token bitmask 机制。

3.1 XGrammarGrammar 对象

核心属性

  • matcher: GrammarMatcher — xgrammar 的核心匹配器，管理语法状态机
  • ctx: CompiledGrammar — 编译后的语法对象，可复用创建新 matcher
  • vocab_size: int — 词表大小
  • override_stop_tokens — 覆盖的停止 token 列表
  • accepted_tokens: List[int] — 已接受的 token 历史，用于调试和回滚
# Token 接受与回滚
def accept_token(self, token: int):
    if not self.is_terminated():
        accepted = self.matcher.accept_token(token)
        if not accepted:
            raise ValueError(f"Token not accepted: {token}")
        self.accepted_tokens.append(token)

def rollback(self, k: int):
    self.matcher.rollback(k)
    self.accepted_tokens = self.accepted_tokens[:-k]

# 复制：共享 CompiledGrammar，创建新 GrammarMatcher
def copy(self):
    matcher = GrammarMatcher(self.ctx, max_rollback_tokens=200,
                             override_stop_tokens=self.override_stop_tokens)
    return XGrammarGrammar(matcher, self.vocab_size, self.ctx, ...)

Structural Tag 处理

XGrammar 支持两种 Structural Tag 格式：

  • Legacy 格式：包含 structures（含 begin/schema/end）和 triggers 字段。通过 is_legacy_structural_tag() 检测
  • 新格式：包含 format 字段，支持嵌套的 tag/sequence/or/triggered_tags 等结构类型。_sanitize_structural_format() 递归补全缺失的 json_schema

3.3 Token Bitmask 机制

XGrammar 使用压缩 bitmask（而非 bool mask）来表示合法 token 集合，大幅节省内存和带宽：

Bitmask 工作流

  1. 分配：allocate_token_bitmask(batch_size, vocab_size) 分配压缩 bitmask 张量
  2. 填充：matcher.fill_next_token_bitmask(vocab_mask, idx) 根据当前状态设置合法 token 位
  3. 移动：vocab_mask.to(device, non_blocking=True) 异步传输到 GPU
  4. 应用：通过 Triton 或 CUDA 内核原地修改 logits
# 应用掩码 - 根据平台选择实现
def apply_vocab_mask(self, logits, vocab_mask):
    if logits.device.type in ("cuda", "npu", "xpu"):
        if _is_hip:
            apply_token_bitmask_inplace_cuda(logits, vocab_mask)  # AMD HIP
        else:
            apply_token_bitmask_inplace_triton(logits, vocab_mask)  # Triton
Bitmask vs Bool Mask — XGrammar 的 bitmask 将每个 token 用 1 bit 表示（词表 128K 仅需 16KB），而 Outlines 的 bool mask 每个 token 用 1 byte（需 128KB）。bitmask 方案内存效率高 8 倍，但需要专用内核解码。

3.4 Jump Forward 支持

XGrammar 原生支持 jump forward 优化：

def try_jump_forward(self, tokenizer):
    s = self.matcher.find_jump_forward_string()  # xgrammar 原生接口
    if s:
        return [], s
    return None

def jump_and_retokenize(self, old_output_ids, new_output_ids, next_state):
    # 找到最长公共前缀
    k = 0
    for i, old_id in enumerate(old_output_ids):
        if old_id == new_output_ids[i]:
            k = i + 1
        else:
            break
    # 回滚差异部分
    if k < len(old_output_ids):
        self.matcher.rollback(len(old_output_ids) - k)
    # 重新接受新 token
    for i in range(k, len(new_output_ids)):
        self.matcher.accept_token(new_output_ids[i])

4. Outlines 后端

4.1 OutlinesGrammar 对象

Outlines 后端基于 RegexGuide（有限状态机）实现约束解码：

核心差异

  • 状态追踪：维护一个整数 state，通过 guide.get_next_state(state, token) 推进
  • 掩码类型：使用 torch.bool 张量（True = 禁止），而非 bitmask
  • 掩码填充：从 guide.get_next_instruction(state).tokens 获取合法 token 列表，然后反向标记
  • 掩码应用：logits.masked_fill_(vocab_mask, float("-inf"))
# 填充掩码 - 反向逻辑（标记禁止的 token）
def fill_vocab_mask(self, vocab_mask, idx):
    tokens = torch.tensor(
        self.guide.get_next_instruction(self.state).tokens, dtype=torch.int64
    ).to(vocab_mask.device)
    vocab_mask = vocab_mask[idx]
    vocab_mask.fill_(1)           # 先全部标记为禁止
    vocab_mask.scatter_(0, tokens, False)  # 合法 token 取消禁止

# 应用掩码
@staticmethod
def apply_vocab_mask(logits, vocab_mask):
    logits.masked_fill_(vocab_mask, float("-inf"))

5. Jump Forward 优化

Jump Forward（跳跃前进）是 SGLang 独创的约束解码加速技术，源自 Compressed Finite State Machine 论文。

5.1 核心思想

问题

在约束解码中，语法状态机的某些状态可能只有唯一的后继路径（例如在输出 JSON key "name": 时，引号后面的字符是确定的）。逐 token 解码这些确定性前缀浪费了大量计算。

解决方案

通过预分析 FSM，找出所有单出边状态（只有一个可能的下一步），将这些连续的确定性步骤链接成跳跃路径，一次性跳过多个 token 的生成。

单出边: 'a'"] S2 --> S3["状态 3
单出边: 'm'"] S3 --> S4["状态 4
单出边: 'e'"] S4 --> S5["状态 5
多出边"] S0 -.->|"Jump Forward: 'name'"| S5 style S0 fill:#1c2128,stroke:#39d0d8,color:#e6edf3 style S5 fill:#1c2128,stroke:#3fb950,color:#e6edf3 style S1 fill:#1c2128,stroke:#e3b341,color:#e6edf3 style S2 fill:#1c2128,stroke:#e3b341,color:#e6edf3 style S3 fill:#1c2128,stroke:#e3b341,color:#e6edf3 style S4 fill:#1c2128,stroke:#e3b341,color:#e6edf3

5.2 FSM 分析算法

outlines_jump_forward.py 中的 init_state_to_jump_forward() 函数分析 Outlines 的 byte-level FSM：

算法步骤

  1. 解析正则：interegular.parse_pattern(regex) -> FSM
  2. 字节级转换：make_byte_level_fsm() -> byte FSM
  3. 确定化：make_deterministic_fsm() -> deterministic FSM
  4. 出边计数：遍历所有转移边，统计每个状态的出边数量
    • 终止状态默认有 1 条出边（可终止路径）
    • anything_value 通配符边不计入跳跃分析
  5. 构建跳跃表：只保留恰好有 1 条出边的状态作为跳跃点
  6. 字节级处理：分别处理 ASCII 字符和多字节 UTF-8 转移
# JumpEdge 数据结构
@dataclass
class JumpEdge:
    symbol: str = None           # 字符级跳跃符号
    symbol_next_state: int = None
    byte: int = None             # 字节级跳跃
    byte_next_state: int = None

OutlinesJumpForwardMap

OutlinesJumpForwardMap 封装了跳跃表，提供两个核心查询方法：

  • jump_forward_symbol(state)：从 state 开始，沿着单出边状态链前进，返回完整的跳跃字符串和终点状态
  • jump_forward_byte(state)：返回字节级跳跃序列，用于处理 UTF-8 continuation bytes
UTF-8 处理 — Outlines 的 byte-level FSM 将多字节字符拆分为多个状态转移。Jump forward 在返回跳跃结果时，需要先处理 continuation bytes（范围 0x80-0xBF），将它们转换为特殊 token（如 <0xE9>），然后通过 tokenizer 转换为 token ID。

6. LLGuidance 后端

6.1 GuidanceGrammar 对象

LLGuidance 后端基于 Rust 实现的 llguidance 库，提供高效的语法匹配：

核心特点

  • 使用 LLMatcher 作为匹配器，通过 consume_token() 推进状态
  • Bitmask 方案与 XGrammar 类似，使用 llguidance.torch 提供的分配/填充/应用函数
  • 支持 compute_ff_tokens() 实现原生的 fast-forward（jump forward）
  • Bitmask 智能复用：只在 batch 增大时重新分配，避免频繁内存分配
class GuidanceGrammar(BaseGrammarObject):
    def __init__(self, llguidance_tokenizer, serialized_grammar):
        self.ll_matcher = LLMatcher(
            llguidance_tokenizer, serialized_grammar,
            log_level=int(os.environ.get("LLGUIDANCE_LOG_LEVEL", "1"))
        )
        self.bitmask = None  # 惰性分配

    def allocate_vocab_mask(self, vocab_size, batch_size, device):
        if self.bitmask is None or self.bitmask.shape[0] < batch_size:
            self.bitmask = allocate_token_bitmask(batch_size, self.llguidance_tokenizer.vocab_size)
        return self.bitmask[:batch_size]  # 复用已分配的内存

    def try_jump_forward(self, tokenizer):
        ff_tokens = self.ll_matcher.compute_ff_tokens()  # 原生 fast-forward
        return (ff_tokens, "") if ff_tokens else None

7. Reasoner Grammar 推理约束

Reasoner Grammar 是一个装饰器模式的实现，用于支持具有"思考"阶段的模型（如 DeepSeek-R1、Qwen3）在思考结束后才激活语法约束。

7.1 ReasonerGrammarObject

stateDiagram-v2 [*] --> Thinking: tokens_after_think_end = -1 Thinking --> JustEnded: 遇到 think_end_id JustEnded --> Constrained: tokens_after_think_end = 0 Constrained --> Constrained: accept_token 传递给内部 grammar Thinking --> Thinking: accept_token 不传递 note right of Thinking: 思考阶段\n不应用语法约束 note right of Constrained: 约束阶段\n语法完全生效

状态追踪

tokens_after_think_end 字段追踪思考阶段的状态：

  • -1：仍在思考阶段，语法约束不生效
  • 0：刚刚结束思考（上一个 token 是 think_end_id）
  • > 0：思考结束后已生成的 token 数，语法约束完全生效
def accept_token(self, token: int):
    if self.tokens_after_think_end >= 0:
        self.grammar.accept_token(token)  # 仅在约束阶段传递给内部语法
    self.transfer_state(token)

def fill_vocab_mask(self, vocab_mask, idx):
    if self.tokens_after_think_end >= 0:
        self.grammar.fill_vocab_mask(vocab_mask, idx)  # 仅在约束阶段填充掩码

def rollback(self, k):
    steps_after_think = min(k, self.tokens_after_think_end)
    if steps_after_think > 0:
        self.grammar.rollback(steps_after_think)  # 仅回滚约束阶段的步骤
    for _ in range(k):
        self.rollback_state()

7.2 ReasonerGrammarBackend

Reasoner 后端是一个纯粹的装饰器，包装任意一个基础 grammar 后端：

class ReasonerGrammarBackend(BaseGrammarBackend):
    def __init__(self, grammar_backend, think_end_id):
        self.grammar_backend = grammar_backend  # 被装饰的基础后端
        self.think_end_id = think_end_id

    def _init_value_dispatch(self, key, reasoning):
        ret = self.grammar_backend._init_value_dispatch(key, reasoning)
        if ret is None or ret is INVALID_GRAMMAR_OBJ:
            return ret  # 不包装无效语法
        obj = ReasonerGrammarObject(ret, self.think_end_id)
        obj.maybe_init_reasoning(reasoning)
        return obj
自动激活 — 在 create_grammar_backend() 中，如果 server 配置了 reasoning_parser 且 tokenizer 有 think_end_id，系统会自动将基础后端（XGrammar/Outlines/LLGuidance）包装为 ReasonerGrammarBackend。用户无需手动配置。

8.3 多 Rank 同步

在张量并行（TP）场景下，多个 Rank 需要对语法编译的就绪状态达成一致：

同步协议

  1. 每个 Rank 独立轮询自己的 grammar_queue，收集 ready_req_idxs 和 failed_req_idxs
  2. 通过 torch.distributed.all_gather_object() 收集所有 Rank 的结果
  3. 就绪集合：取所有 Rank 的交集 — 只有所有 Rank 都就绪的请求才能继续
  4. 失败集合：取所有 Rank 的并集 — 任何一个 Rank 超时的请求都标记为失败
synced_ready = set.intersection(*[x[0] for x in all_gather_output])
synced_failed = set.union(*[x[1] for x in all_gather_output])

8.4 超时处理

语法编译可能因为复杂的 schema 而耗时很长。系统通过计数机制实现超时：

  • 每次轮询未就绪时，grammar_wait_ct += 1
  • 当 grammar_wait_ct >= SGLANG_GRAMMAR_MAX_POLL_ITERATIONS 时触发超时
  • 超时的请求：grammar.cancel()，缓存 INVALID_GRAMMAR_OBJ，向用户返回错误
  • 实际等待时间 = MAX_POLL_ITERATIONS * max(POLL_INTERVAL, GPU_forward_latency)

11.3 设计亮点

多后端可插拔

统一的 BaseGrammarBackend/BaseGrammarObject 抽象允许无缝切换 XGrammar、Outlines、LLGuidance 后端。注册机制支持第三方扩展。

异步编译不阻塞

语法编译可能耗时数秒（复杂 JSON Schema）。通过 ThreadPool + Future 模式，编译过程不阻塞 GPU 推理。已编译的语法被缓存复用。

Bitmask 高效掩码

XGrammar 和 LLGuidance 使用压缩 bitmask（1 bit/token），配合 Triton/CUDA kernel 原地修改 logits。相比 bool mask 节省 8 倍内存带宽。

Reasoner 装饰器

透明地包装任何后端，使"思考"模型在推理阶段跳过语法约束，结束思考后才激活。自动检测配置，用户无需手动设置。

Jump Forward 加速

通过 FSM 分析预识别确定性路径，跳过多个 token 的逐一生成。XGrammar 和 LLGuidance 提供原生实现，Outlines 通过 JumpForwardMap 实现。

多 Rank 一致性

GrammarManager 通过 all_gather + 交集/并集策略确保分布式环境下所有 Rank 对语法就绪状态达成一致，避免 TP desync。
