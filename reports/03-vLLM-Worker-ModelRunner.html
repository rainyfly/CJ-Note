vLLM Worker 与 GPUModelRunner 深度解析

TYPE: REPO ANALYSIS DATE: 2026-04-10 TOPIC: Worker / ModelRunner / CUDAGraph / Attention BASE: vLLM v1 架构

TL;DR

  • 每个 GPU 对应一个 GPUWorker 进程，由 Executor（Multiproc/Ray/UniProc）统一管理，通过 collective_rpc 广播推理任务。
  • GPUModelRunner 是核心推理引擎，执行链为：_update_states → _prepare_inputs → model.forward → _sample → _bookkeeping，每步都高度优化。
  • InputBatch 用 CPU pinned memory 维护所有请求的 token_ids、block_table 和采样参数，避免频繁 GPU 内存分配，同时支持 CPU→GPU 异步传输。
  • Decode-only 批次使用预捕获的 CUDAGraph 执行，将数百次 Kernel Launch 压缩为一次 cudaGraphLaunch，大幅降低 CPU 开销（尤其对小 batch 的 decode 阶段）。
  • 异步调度（use_async_scheduling=True）将下一步的调度/准备与当前步的 GPU 执行重叠，几乎消除 CPU 空洞，进一步提升吞吐。

class ExecutorBase(ABC):

    @abstractmethod
    def execute_model(
        self,
        scheduler_output: SchedulerOutput,
        non_block: bool = False,
    ) -> ModelRunnerOutput | Future[ModelRunnerOutput]:
        """
        在所有 Worker 上执行一步模型推理。
        non_block=True: 立即返回 Future（用于与 CPU 工作重叠）
        """

    @abstractmethod
    def collective_rpc(
        self,
        method: str,
        timeout: float | None = None,
        args: tuple = (),
        kwargs: dict = {},
    ) -> list[Any]:
        """向所有 Worker 广播 RPC 调用"""

4.2 _update_states() 详解

def _update_states(self, scheduler_output: SchedulerOutput):
    """
    根据调度结果更新 ModelRunner 内部的请求状态缓存。
    避免每次都从 SchedulerOutput 重新重建所有请求的状态。
    """
    # 1. 处理新请求
    for req_data in scheduler_output.scheduled_new_reqs:
        req_id = req_data.req_id
        self.requests[req_id] = CachedRequestState(
            req_id=req_id,
            prompt_token_ids=req_data.prompt_token_ids,
            prompt=req_data.prompt,
            mm_inputs=req_data.mm_inputs,
            sampling_params=req_data.sampling_params,
            # 初始状态
            output_token_ids=[],
            generator=req_data.generator,
        )
        # 设置 block_table 到 input_batch
        self.input_batch.add_request(...)

    # 2. 更新已有请求（新的 block_ids）
    for req_data in scheduler_output.scheduled_cached_reqs:
        req_id = req_data.req_id
        req_state = self.requests[req_id]
        # 更新 block_ids（新分配的块）
        req_state.block_ids = req_data.new_block_ids
        # 更新 input_batch 中的 block_table
        self.input_batch.update_request(req_id, req_data)

    # 3. 移除完成的请求
    for req_id in scheduler_output.finished_req_ids:
        del self.requests[req_id]
        self.input_batch.remove_request(req_id)

4.3 _prepare_inputs() 详解

这是最复杂的步骤，需要构建多个张量：

def _prepare_inputs(self, scheduler_output: SchedulerOutput):
    """构建模型推理所需的所有 GPU 张量"""

    total_num_scheduled_tokens = scheduler_output.total_num_scheduled_tokens
    num_reqs = self.input_batch.num_reqs

    # ── 1. 构建 input_ids ──────────────────────────────────────────
    # 对于每个请求，取 [num_computed_tokens : num_computed_tokens + num_scheduled]
    # 形状: [total_num_scheduled_tokens]
    input_ids = torch.zeros(total_num_scheduled_tokens, dtype=torch.int32, device="cuda")
    # 从 input_batch.token_ids_cpu 按请求拼接填充

    # ── 2. 构建 positions ─────────────────────────────────────────
    # 每个 token 的绝对位置（用于 RoPE）
    # 对请求 r，scheduled token k 的 position = num_computed_tokens_r + k
    positions = torch.zeros(total_num_scheduled_tokens, dtype=torch.int64, device="cuda")

    # ── 3. 构建 block_table ───────────────────────────────────────
    # 形状: [num_reqs, max_num_blocks_per_req]
    # block_table[i, j] = req i 的第 j 个逻辑块对应的物理块 ID
    block_table = self.input_batch.block_table_cpu_tensor[:num_reqs].to("cuda")

    # ── 4. 构建 slot_mapping ─────────────────────────────────────
    # 形状: [total_num_scheduled_tokens]
    # slot_mapping[i] = 第 i 个 token 的 KV Cache 写入位置
    # = block_id × block_size + position_within_block
    slot_mapping = compute_slot_mapping(
        block_table, num_computed_tokens_per_req, num_scheduled_tokens_per_req
    )

    # ── 5. 构建 AttentionMetadata ─────────────────────────────────
    attn_metadata = attn_backend.build_metadata(
        num_prefills=...,
        num_prefill_tokens=...,
        num_decode_tokens=...,
        query_start_loc=...,     # 每个请求 query 的起始位置
        seq_lens=...,             # 每个请求的完整序列长度（含历史）
        max_query_len=...,
        block_table=block_table,
        slot_mapping=slot_mapping,
    )

    # ── 6. 多模态特征（如有）──────────────────────────────────────
    mm_kwargs = build_multimodal_kwargs(scheduled_reqs)

    # ── 7. LoRA 设置（如有）──────────────────────────────────────
    if self.lora_enabled:
        set_active_loras(scheduled_reqs)

    return input_ids, positions, attn_metadata, mm_kwargs

5. InputBatch（批次管理）

文件：vllm/v1/worker/gpu_input_batch.py

5.1 数据结构

class InputBatch:
    """维护当前 batch 中所有请求的状态"""

    # 请求列表（有序，用于索引）
    req_ids: list[str]
    req_id_to_index: dict[str, int]

    # CPU 侧张量（pinned memory，支持异步 CPU→GPU 传输）
    token_ids_cpu: np.ndarray        # [num_reqs, max_model_len]  token ID
    num_tokens_no_spec: np.ndarray   # [num_reqs]  当前 token 数（不含推测）
    block_table_cpu: np.ndarray      # [num_reqs, max_blocks]  block 映射

    # GPU 侧张量（常驻，减少分配）
    token_ids_gpu: torch.Tensor      # [num_reqs, max_model_len]
    block_table_gpu: torch.Tensor    # [num_reqs, max_blocks]

    # 采样参数（批量化，避免逐请求循环）
    sampling_metadata: SamplingMetadata

    # 推测解码
    prev_sampled_token_ids: torch.Tensor | None

5.2 CachedRequestState

class CachedRequestState:
    """ModelRunner 侧的请求状态（轻量缓存）"""
    req_id: str
    prompt_token_ids: list[int]
    sampling_params: SamplingParams
    output_token_ids: list[int]       # 已生成的 token
    block_ids: list[list[int]]        # 当前分配的物理块
    generator: torch.Generator | None # 随机数生成器（用于可复现采样）
设计要点 CPU pinned memory 的 token_ids_cpu 和 block_table_cpu 支持 DMA 异步传输到 GPU，避免 Python 侧每步分配新 Tensor 的开销。GPU 侧常驻张量则进一步消除显存分配延迟。

6. CUDAGraph 优化

6.1 为什么需要 CUDAGraph

没有 CUDAGraph 的 CPU 开销 Python 侧循环逐个 Kernel Launch，每个 Kernel Launch ≈ 5–20 μs CPU 开销。一个 Transformer 层有 10+ Kernel，32 层模型 ≈ 320 Kernel × 10 μs = 3.2 ms CPU 开销，严重限制小 batch decode 的吞吐。
使用 CUDAGraph 后 第一次执行时"录制"，之后只需 cudaGraphLaunch() 一次。单次 Launch ≈ 5–10 μs，无论模型多大。对 Decode 阶段（batch 小，GPU 计算快）收益最大。

6.3 CUDAGraph 执行

def _run_with_cudagraph(
    self,
    batch_descriptor: BatchDescriptor,
    input_ids: torch.Tensor,
    positions: torch.Tensor,
    attn_metadata: AttentionMetadata,
):
    """使用预捕获的 CUDAGraph 执行推理"""

    # 1. 将实际数据拷贝到 Graph 的静态输入缓冲区
    input_buffers = batch_descriptor.input_buffers
    input_buffers["input_ids"][:num_tokens].copy_(input_ids)
    input_buffers["positions"][:num_tokens].copy_(positions)
    # 注意：kv_cache 张量本身就是静态的，不需要拷贝

    # 2. replay CUDAGraph（一次 API 调用完成所有 Kernel）
    batch_descriptor.graph.replay()

    # 3. 从静态输出缓冲区读取结果
    logits = batch_descriptor.output_buffers["logits"]
    hidden_states = batch_descriptor.output_buffers["hidden_states"]

    return logits, hidden_states

6.4 何时使用 CUDAGraph

条件模式说明
所有请求都在 Decode 阶段（每请求 1 个 token） CUDAGraph batch_size 需在预捕获范围内
有 Prefill 请求（序列长度可变） Eager 形状不固定，无法 replay
Chunked Prefill（混合 Prefill+Decode） Eager 形状动态变化
batch_size 超出预捕获范围 Eager 找不到匹配 graph
强制禁用（enforce_eager=True） Eager 调试/兼容性用途

# GPUModelRunner 异步调度模式
def execute_model(self, scheduler_output):
    if self.use_async_scheduling:
        # 1. 先用上一步采样的 token（GPU 上缓存）
        #    不等待 CPU 侧的 token 确认

        # 4. bookkeeping 在下一步的 prepare_inputs 中做
        #    本步不等待 CPU 同步
        return ModelRunnerOutput(use_async=True, ...)

    else:
        # 同步路径...

8.1 Prefill vs Decode Attention 的区别

阶段Q shapeKV 来源Mask实现
Prefill（新请求 prompt tokens） [seq_len, num_heads, head_size] 当前序列内（自注意力） 下三角因果 mask FlashAttention
Decode（生成新 token） [1, num_heads, head_size] KV Cache（历史，非连续分页） 无（全历史注意力） FlashInfer / Paged Decode Kernel
混合批次处理 同一个 batch 可能同时包含 Prefill 和 Decode 请求（Chunked Prefill）。attn_metadata 中通过 query_start_loc 精确区分每个请求的 token 范围，后端负责对不同类型的请求分别调用对应的 Attention Kernel。

9.2 SamplingMetadata 批量化

为避免逐请求循环（Python 慢），采样参数被批量化为张量：

class SamplingMetadata:
    """所有请求的采样参数，批量化为张量形式"""

    temperature: torch.Tensor    # [num_reqs] 每个请求的 temperature
    top_p: torch.Tensor          # [num_reqs]
    top_k: torch.Tensor          # [num_reqs]
    min_p: torch.Tensor          # [num_reqs]

    do_top_p_top_k: bool         # 是否有请求需要 top-p/top-k 过滤
    do_temperatures: bool        # 是否需要 temperature 缩放

    generators: dict[int, torch.Generator]  # req_idx → 随机数生成器
批量化的意义 将 N 个请求的采样参数预先向量化为 Tensor，使得 temperature 缩放、top-k/top-p 过滤等操作可以对整个 batch 执行单次 GPU 向量运算，而非 Python 循环 N 次。这对于大 batch size 时的采样性能至关重要。

11.1 CUDAGraph Replay 时序对比

无 CUDAGraph（Eager 模式，混合 Prefill+Decode） [prepare inputs] → [CPU→GPU transfer] → [FW Att.] → [Layer 0] → [Layer 1] → ... → [Layer N] → [logits] → [sample] 每个 Kernel 单独 Launch，CPU 开销随层数线性增长。
有 CUDAGraph（Decode-only 批次） [prepare inputs] → [memcpy to static buffers] → [Graph Replay（单次 cudaGraphLaunch，所有 Kernel 一起执行）] CPU 开销极低（约 5–10 μs），GPU 执行完全重叠，Decode 吞吐大幅提升。
