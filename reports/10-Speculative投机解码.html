1. 投机解码概述

1.1 核心原理

投机解码 (Speculative Decoding) 是一种利用"猜测-验证"范式加速自回归生成的技术。其核心思想是：使用一个轻量的 draft model（草稿模型）快速生成多个候选 token，然后由完整的 target model（目标模型）一次性并行验证这些候选 token 的正确性。

flowchart LR subgraph "传统自回归解码" A1[Token 1] --> A2[Token 2] --> A3[Token 3] --> A4[Token 4] --> A5[Token 5] end subgraph "投机解码" B1[Draft: 猜测 5 个 Token
1 次 forward] --> B2[Target: 并行验证
1 次 forward] B2 --> B3[接受 3 个 + Bonus Token
= 4 个有效 Token] end style B1 fill:#1a3a4a,stroke:#39d0d8,color:#e6edf3 style B2 fill:#2a1a3a,stroke:#bc8cff,color:#e6edf3 style B3 fill:#1a2a1a,stroke:#3fb950,color:#e6edf3
加速原理：如果 draft model 的预测准确率较高，一次"猜测+验证"可以在 2 次 forward 中生成 N+1 个 token（N 个被接受 + 1 个 bonus token），而传统方法需要 N+1 次 forward。由于 LLM 推理是 memory-bound 的，draft model 的额外计算开销很小，因此可以获得显著的加速。

def draft_forward(self, forward_batch):
    score_list, token_list, parents_list = [], [], []

    for i in range(self.speculative_num_steps):
        # 1. 选择 top-k token 并构建树信息
        input_ids, hidden_states, scores, tree_info = select_top_k_tokens(
            i, topk_p, topk_index, hidden_states, scores, self.topk
        )
        score_list.append(tree_info[0])   # 累积分数
        token_list.append(tree_info[1])   # token ids
        parents_list.append(tree_info[2]) # 父节点索引

        if i == self.speculative_num_steps - 1:
            break  # 最后一步不需要 forward

        # 2. 设置 forward batch 并运行 draft model
        forward_batch.input_ids = input_ids
        forward_batch.out_cache_loc = out_cache_loc[i]
        forward_batch.positions.add_(1)
        logits_output = self.draft_model_runner.forward(forward_batch)

        # 3. 获取下一步的 topk
        probs = torch.softmax(logits_output.next_token_logits, dim=-1)
        topk_p, topk_index = fast_topk(probs, self.topk, dim=-1)
        hidden_states = logits_output.hidden_states

    # 4. 整理结果并构建验证树
    parent_list, top_scores_index, draft_tokens = organize_draft_results(
        score_list, token_list, parents_list, self.speculative_num_draft_tokens
    )

def organize_draft_results(score_list, token_list, parents_list, num_draft_token):
    # 1. 将多步的分数展平
    score_list = torch.cat(score_list, dim=1).flatten(1)  # (bs, total_candidates)

    # 2. 选择 top num_draft_token-1 个最高分候选
    top_scores = torch.topk(score_list, num_draft_token - 1, dim=-1)
    top_scores_index = torch.sort(top_scores.indices).values

    # 3. 根据索引收集 draft tokens
    draft_tokens = torch.gather(ss_token_list, index=top_scores_index, dim=1)

    # 4. 构建父节点列表 (不包含最后一步)
    parent_list = torch.cat(parents_list[:-1], dim=1)

    return parent_list, top_scores_index, draft_tokens

4.2 Tree Mask 构建

build_tree_kernel_efficient 使用 sgl_kernel 的 CUDA 内核高效构建树注意力掩码和检索索引：

Tree Mask 模式

模式掩码形状说明
FULL_MASK(seq_lens_sum * N + N^2 * bs)完整掩码，包含 prefix + draft 部分
QLEN_ONLY(N * bs * N)仅 query 长度部分的掩码
QLEN_ONLY_BITPACKING(N * bs) 按位打包位打包优化，节省显存

检索索引结构

  • retrive_index: (bs, num_verify_tokens) -- 每个位置在验证结果中的索引
  • retrive_next_token: (bs, num_verify_tokens) -- 树中每个节点的第一个子节点
  • retrive_next_sibling: (bs, num_verify_tokens) -- 树中每个节点的下一个兄弟节点
  • positions: (bs * num_verify_tokens) -- 每个 draft token 的位置编码
positions 计算示例：如果每个 draft token 的深度为 [0, 1, 1, 2]，prompt 长度为 7，则 positions = [7, 8, 8, 9]。深度相同的 token 共享相同的位置编码。

6. Overlap Worker (V2 架构)

6.1 V2 设计理念

V2 Worker（如 EAGLEWorkerV2、StandaloneWorkerV2）支持 overlap scheduler，允许 draft 和 verify 阶段与调度器的其他操作重叠执行。

sequenceDiagram participant S as Scheduler participant T as Target Worker participant D as Draft Worker Note over S,D: V1: 串行执行 S->>D: draft() D-->>S: draft tokens S->>T: verify() T-->>S: accepted tokens Note over S,D: V2: 重叠执行 S->>D: draft() [异步] S->>S: 调度其他任务 D-->>S: draft tokens [回调] S->>T: verify() [与下一个 draft 重叠] T-->>S: accepted tokens S->>D: draft_extend_after_decode() [与调度重叠]

V2 架构的关键特性：

  • FutureIndices：延迟的索引结构，允许在验证完成前就开始准备下一轮
  • verify_done Event：CUDA Event 用于同步验证完成状态
  • 支持 multi-layer eagle：MultiLayerEagleWorkerV2 将多层 draft model 与 overlap 结合

    def _prepare_draft_tokens(self, batch):
        # 1. 同步缓存
        self.ngram_cache.synchronize()

        # 2. 为每个请求构建查询 token
        batch_tokens = []

        (max_num_token, hidden_size), dtype=model_runner.dtype
    )
    self.topk_p = torch.zeros((max_num_token, topk), dtype=torch.float32)
    self.topk_index = torch.zeros((max_num_token, topk), dtype=torch.int64)

10. 采样策略

10.1 贪心验证 (Greedy Verification)

贪心验证是最简单的策略：target model 的 argmax 预测必须与 draft token 完全匹配才接受。

def verify_tree_greedy_func(predicts, accept_index, accept_token_num,
                            candidates, retrive_index, retrive_next_token,
                            retrive_next_sibling, target_predict):
    # sgl_kernel 实现的高效树贪心验证
    # 使用 retrive_next_token/retrive_next_sibling 进行 DFS 遍历
    verify_tree_greedy(
        predicts=predicts,           # 输出: 最终预测的 token
        accept_index=accept_index,   # 输出: 接受的 token 索引
        accept_token_num=accept_token_num,  # 输出: 每个序列接受的 token 数
        candidates=candidates,       # 输入: draft tokens
        retrive_index=retrive_index, # 输入: 树检索索引
        retrive_next_token=retrive_next_token,  # 输入: 子节点指针
        retrive_next_sibling=retrive_next_sibling,  # 输入: 兄弟指针
        target_predict=target_predict,  # 输入: target 预测
    )

10.2 随机采样验证 (Stochastic Verification)

对于非贪心采样（temperature > 0），使用概率接受/拒绝机制：

# 非贪心验证核心逻辑
# 1. 应用温度缩放
expanded_temperature = torch.repeat_interleave(
    sampling_info.temperatures, draft_token_num, dim=0
)
target_probs = F.softmax(logits / expanded_temperature, dim=-1)

# 2. 应用 top-k 和 top-p 截断
target_probs = top_k_renorm_prob(target_probs, top_ks)
target_probs = top_p_renorm_prob(target_probs, top_ps)

# 3. 随机接受/拒绝
coins = torch.rand_like(candidates, dtype=torch.float32)
coins_for_final = torch.rand((bs,), dtype=torch.float32)

tree_speculative_sampling_target_only(
    predicts=predict,
    accept_index=accept_index,
    accept_token_num=accept_length,
    candidates=candidates,
    uniform_samples=coins,
    uniform_samples_for_final_sampling=coins_for_final,
    target_probs=target_probs,
    draft_probs=draft_probs,  # 全零 (target-only)
    threshold_single=speculative_accept_threshold_single,
    threshold_acc=speculative_accept_threshold_acc,
    deterministic=True,
)
接受阈值：threshold_single 和 threshold_acc 控制单个 token 和累积的接受阈值，允许在接受率和生成质量之间权衡。

投机解码的 KV Cache 管理是最复杂的部分之一，因为需要处理树结构的分配和释放。

大页面 + 多 topk 场景

当 page_size > 1 且 topk > 1 时，需要特殊处理：

  1. 最后一个部分页面需要复制：每个 topk 分支需要自己的部分页面副本

  3. 释放多余页面：使用 get_target_cache_loc 确定需要释放的 slot

12. MoE 投机支持

SGLang 的投机解码完整支持 MoE 模型，通过专门的上下文管理器切换 MoE 后端：

# 在 draft forward 前切换 MoE 后端
with speculative_moe_backend_context(), speculative_moe_a2a_backend_context():
    # Draft model 使用独立的 MoE runner 后端
    # 例如 draft 可以用 triton 后端，target 用 deep_gemm 后端
    self.forward_draft_extend(batch, hidden_states, next_token_ids, seq_lens_cpu)

speculative_moe_backend_context

临时将全局 MOE_RUNNER_BACKEND 切换为 SPECULATIVE_MOE_RUNNER_BACKEND：

@contextmanager
def speculative_moe_backend_context():
    global MOE_RUNNER_BACKEND
    original = MOE_RUNNER_BACKEND
    try:
        MOE_RUNNER_BACKEND = get_speculative_moe_runner_backend()
        yield
    finally:
        MOE_RUNNER_BACKEND = original

speculative_moe_a2a_backend_context

类似地切换 A2A 通信后端，并禁用 FP4 allgather（因为 MTP 层通常未量化）。

SGLang 的投机解码系统具有以下核心优势：

  1. 多算法支持：EAGLE / EAGLE-3 / Standalone / N-Gram 四种算法，覆盖不同场景需求。EAGLE 适合通用加速，N-Gram 适合无模型场景，Standalone 支持任意模型对。
  2. 树结构投机：使用 Token Tree 而非线性链，通过 topk 分支在有限验证预算内探索更多路径。build_tree_kernel_efficient 使用 CUDA 内核高效构建树掩码和检索索引。
  3. 高效验证：支持贪心和随机采样两种验证策略。贪心使用 verify_tree_greedy CUDA 内核，随机使用 tree_speculative_sampling_target_only 内核，均通过 DFS 遍历树结构。
  4. CUDA Graph 加速：Draft 和 Draft Extend 阶段均支持 CUDA Graph 捕获，消除多步推理的 kernel launch 开销。支持按 batch size 分桶捕获。
  5. Overlap 调度：V2 Worker 支持与 overlap scheduler 配合，将 draft/verify/extend 操作与调度器的批次管理重叠执行。
  6. 精细的 KV Cache 管理：支持 page_size > 1 和 topk > 1 的复杂场景，通过 Triton 内核（assign_draft_cache_locs、get_src_tgt_cache_loc 等）高效管理树结构的 KV Cache 分配和释放。
  7. MoE 兼容：通过 context manager 机制无缝支持 MoE 模型的投机解码，允许 draft model 和 target model 使用不同的 MoE 后端。
  8. Grammar 约束：验证阶段支持结构化输出的语法约束（vocab_mask），通过在 tree 验证过程中应用语法掩码确保输出合法性。
性能关键路径：在典型的 EAGLE 配置下（topk=4, steps=3, draft_tokens=16），一次 draft+verify 循环可以在 2 次模型 forward（1 次 draft + 1 次 target verify）中生成平均 3-5 个有效 token。CUDA Graph 的使用进一步将 draft 阶段的延迟降低到接近单次 decode 的水平。
