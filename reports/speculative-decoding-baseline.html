Speculative Decoding

Feature Deep Dive — Draft Model + Verification Mechanism

LLM Inference Latency Optimization 2024 — 2025 State of Art

1. Introduction

Speculative decoding (投机解码) 是目前最重要的 LLM 推理加速技术之一。 它由 Chen et al. (DeepMind, 2023) 与 Leviathan et al. (Google Brain, 2023) 几乎同时独立提出，核心思想是： 用一个廉价的 draft model 先猜测多个 token，再用昂贵的 target model 一次性并行验证， 从而在不改变输出分布的前提下大幅提升 throughput 和降低延迟。

  • Speculative Sampling (DeepMind) — Chen et al. 2023. [arXiv 2302.01318]
  • Medusa — Cai et al. (LMSYS). 2024. [arXiv 2401.10774]
  • EAGLE — Li et al. 2024. [arXiv 2401.15077]; EAGLE-2 [arXiv 2406.16858]

  • SpecInfer — Miao et al. (CMU). 2023. [arXiv 2305.09781]

2. Motivation & Memory-Bandwidth Bottleneck

Autoregressive 推理的固有瓶颈

大语言模型的标准推理是 autoregressive decoding：每次前向传播只产生 1 个 token， 然后将其拼接到上下文重新输入。对于一个参数量 N 的模型，每次前向传播需要从 HBM 加载约 2N bytes（fp16），而实际计算量仅 2N FLOPs（batch-size=1 时）。 现代 GPU 的算力/带宽比（arithmetic intensity ceiling）远高于 1 FLOP/byte， 因此单 token 推理完全是 memory-bandwidth bound。

Time per token ≈ 2N bytes / HBM_bandwidth
e.g. A100 80G: 2TB/s → 70B fp16 model ≈ 70ms/token

即使算力有大量剩余，也必须等待参数从显存搬运完毕。提高 batch size 可以摊薄带宽开销， 但会增加延迟并引入 KV-cache 压力。Speculative decoding 提供了另一条路： 在单次"大模型前向"中验证多个 draft token，相当于同时生成多个 token， 从而提高算力利用率，降低每 token 的等效延迟。

3. Draft + Verify Loop — 核心机制

Speculative decoding 的一个 完整迭代由以下步骤构成：

Draft Phase（草稿阶段）

用 draft model Mq（小模型）对当前上下文自回归生成 γ 个 token（称为 speculative tokens）： x̃1, x̃2, …, x̃γ， 同时记录每步的概率分布 q(x | context)。 Draft model 通常比 target model 小 10–100×， 因此这 γ 步的开销可以忽略不计，或者与一次 target model 前向相当。

Verification Phase — Parallel Forward Pass（并行验证阶段）

将原始上下文 + γ 个 draft token 拼接成一个长度为 L + γ 的序列，送入 target model Mp 一次前向传播（causal attention 天然支持此操作）。 target model 在位置 i 的输出给出了 p(x | x1..i-1)，即对位置 i 的 token 的 "真实"概率分布。注意这 γ+1 个位置的概率分布是并行计算的， 只需 一次 target model 前向，计算代价约等于生成 1 个 token。

Token-by-Token Acceptance（逐 token 接受/拒绝）

对每个 draft token x̃i，计算接受概率：

Corrected Token Sampling（修正采样）

若第 k 个 draft token 被拒绝，则从 修正分布 norm(max(0, p(x) − q(x))) 中采一个新 token 作为位置 k 的最终输出， 并丢弃之后所有 draft token，开始下一轮 speculate。 若全部 γ 个 draft token 都被接受，则额外从位置 γ+1 的 p 分布采一个 bonus token， 共获得 γ+1 个新 token。

αi = min(1, p(x̃i) / q(x̃i))

以概率 αi 接受 x̃i；否则从 norm(max(0, p − q)) 采样新 token
Speculative Sampling acceptance rule (Chen et al. 2023)
关键性质
每次迭代，无论接受几个 draft token（最少 1 个，最多 γ+1 个）， 最终输出序列的分布 与直接用 target model autoregressive 采样完全等价。 Speculative decoding 是 lossless 的加速方法。

4. Acceptance Sampling 数学推导

接受概率的直觉

若 draft model 在某位置给出 q(x)，target model 给出 p(x)， 则对每个具体 token 值 x，接受概率为：

α(x) = min(1, p(x) / q(x))

这是 Rejection Sampling 的标准结果。当 p 与 q 完全相同时， 接受率为 1（所有 draft token 均被接受，相当于速度提升 γ 倍）； 当两者差异极大时，接受率趋近于 0（每轮只能接受 0–1 个 token，退化为普通推理）。

期望接受 token 数

E[accepted tokens per step] = γ · ᾱ + 1（bonus token）

其中 ᾱ = E[α] = Σx min(p(x), q(x)) （= 1 − TV(p,q) total variation 距离的补）
Leviathan et al. 2023, Theorem 1

因此 ᾱ（平均 token-level 接受率）是核心指标，直接决定加速比。 当 ᾱ = 0.9, γ = 4 时，期望每轮接受 4×0.9 + 1 ≈ 4.6 个 token， target model 前向次数相对减少到原来的 4.6×，但每次前向的 compute 代价也略有上升。

5. Output Distribution Guarantee（无损证明）

Speculative decoding 的无损性（lossless guarantee）由以下定理保证：

定理（非正式）

设 τ 为位置 i 最终输出的 token（经接受/修正采样）。 则对所有 token 值 x：

Pr[τ = x] = p(x)

即最终输出分布恒等于 target model 的分布，与 draft model 无关。

证明思路： 令 A 为接受事件。
Pr[τ = x] = Pr[accept] · Pr[τ = x | accept] + Pr[reject] · Pr[τ = x | reject]

  • 接受时：τ = x̃，且 Pr[accept, x̃ = x] = q(x) · min(1, p(x)/q(x)) = min(p(x), q(x))
  • 拒绝时：τ 从修正分布 norm(max(0, p − q)) 采样，对 x 的贡献为 max(0, p(x) − q(x)) 乘以归一化常数
  • 两项之和 = min(p(x), q(x)) + max(0, p(x) − q(x)) = p(x) ∎

这意味着即使使用 greedy 采样的 target model，只要 draft 与 target 使用同一温度参数， 最终结果和纯 target model 生成 逐 token 等价（即在相同随机数序列下产生完全相同的输出）。

6. 理论加速比分析

Wallclock 加速比公式

设每次 target model 前向耗时为 Tp，draft model 生成 γ 个 token 耗时 Tq(γ)，一轮 speculative 迭代期望产出 β = γᾱ + 1 个 token， 则 wallclock 加速比为：

Speedup = β / (Tq(γ) + Tp) × Tp

= (γᾱ + 1) / (1 + Tq(γ) / Tp)
理想化公式，忽略 KV-cache 操作开销

当 Tq ≪ Tp（draft 极快）且 ᾱ → 1（几乎全接受）时， 加速比趋近 γ + 1。实际中一般可达 2–4×。

影响加速比的关键因素

接受率 ᾱ

最核心指标。受 draft model 质量、任务难度、采样温度影响。通常需要 ᾱ > 0.7 才有明显收益。

γ（Draft 长度）

越大理论收益越高，但也放大了拒绝的影响。需根据 ᾱ 动态调整（EAGLE-2 实现了动态 γ）。

Draft 模型速度比

Draft model 开销需远小于 target model。通常 draft 参数量 < target 的 1/10。

Batch Size 效应

batch size 增大时 target model 变得 compute-bound，speculative decoding 的相对收益下降。

KV Cache 管理

拒绝时需要回滚 KV-cache。PagedAttention 等机制需要正确处理分支撤销，增加实现复杂度。

Target 模型大小

模型越大，每次 target 前向越慢，speculative decoding 的绝对收益也越大。

8. 主要 Draft Model 变体详解

8.1 标准独立 Draft Model

最原始的做法：使用一个完整的小模型（如同族 7B 对应 68M/160M 的 draft）作为推测器。 Draft model 和 target model 需要：

  • 共享同一个 tokenizer（词表完全相同）
  • draft model 的分布与 target model 高度对齐（最好是 target 蒸馏而来）
  • 部署时 draft 和 target 共享同一 GPU 或位于不同 GPU 上（pipeline 方式）
局限性
需要维护两个模型权重，显存开销较大。Draft model 需要特别训练以对齐 target， 在 target 模型更新时 draft 也需要同步更新。

8.2 Medusa — 多头并行推测

Medusa（Cai et al., 2024）避免了独立 draft model 的部署复杂性， 在 target LLM 的最后一层隐状态上添加 K 个额外的 prediction heads （每个 head 是一个简单 MLP + softmax），分别预测未来第 2、3、…、K+1 个 token。 

# Medusa 核心思路（伪代码）
class MedusaModel(LLM):
    def forward(self, x):
        hidden = self.base_lm(x)           # 主模型隐状态
        logits_0 = self.lm_head(hidden)    # 标准下一 token 预测
        # Medusa heads：预测 t+1, t+2, ..., t+K
        logits_k = [head(hidden) for head in self.medusa_heads]
        return logits_0, logits_k

# 验证时构建 token tree，使用 tree attention mask
# 一次前向验证多个候选路径

Medusa 使用 Tree Attention：将多个候选 token 组合成一棵树， 用一次前向传播（带特殊 attention mask）验证所有树节点， 选择最长的被接受前缀作为输出。

Medusa-1（heads 对齐冻结 base）

只训练 K 个额外 head，base model 权重冻结。部署极简，适合资源受限场景。

Medusa-2（联合微调）

在 RLHF/SFT 阶段同时训练 base + heads，接受率更高，但需要重训练。

Medusa 在代码生成（HumanEval）上可实现 2.2–3.6× 加速。 其弱点是每个 head 独立预测，不能捕捉多步 token 间的自回归依赖关系。

8.3 EAGLE / EAGLE-2 — 特征级自回归 Draft

EAGLE（Li et al., 2024）的关键洞察：LLM 最后一层的 feature vector 包含了比 token logits 更丰富的上下文信息。 EAGLE 用一个仅 1 层 transformer 的轻量 draft model， 以 target model 的上一步特征向量 + 当前 token embedding 作为输入，自回归地预测未来 token 的特征向量（再经 target 的 LM head 得 logits）。

f̃t+1 = DraftTransformer([ft, emb(xt)])
logitst+1 = LM_head(f̃t+1)
EAGLE draft 前向（目标是预测 target 模型的 feature，而非直接 logits）

由于 draft 直接对齐了 target 的 feature space（而非仅 token 概率）， 接受率显著高于 Medusa，在 MT-Bench 上达到 ᾱ ≈ 0.8–0.9， 实测加速比 3–4×（单 GPU，batch=1）。

EAGLE-2（Li et al., 2024）引入了 adaptive draft tree： 根据 draft 自身的置信度（entropy 或 top-1 probability）动态决定是否扩展树节点， 优先在高置信度路径上展开更深的猜测，在低置信度路径上提前截止。 这使得 γ 得以动态调整，在困难生成步骤避免无谓猜测，进一步提升效率 （相比 EAGLE-1 再提升约 20–30%）。

8.4 Self-Speculative Decoding（自推测）

LayerSkip（Meta AI, 2024）和 Draft & Verify 等方法 无需任何额外模型，而是让同一个 LLM 先用 前 L' 层（早退）产生 draft， 再用完整 L 层验证。适合 MoE 架构或层间差异较大的模型。

约束
通常需要在训练阶段加入早退正则化（exit loss），使浅层输出分布接近最终层， 否则 draft 质量过差导致加速比不理想。

8.5 Lookahead Decoding — Jacobi 迭代

Lookahead Decoding（Fu et al., 2024）基于 Jacobi 迭代的思想： 将自回归生成看作求解非线性方程组，并行地更新多个位置的 token 猜测， 直到收敛。它不需要任何 draft model，通过维护一个 "lookahead window" 来并行生成候选， 再通过 n-gram 复用来提高接受率（"n-gram pool"）。

9. 哪些场景下有效

Speculative decoding 的收益主要体现在以下场景：

高收益场景

  • 代码生成：函数签名、样板代码、关键词高度可预测，ᾱ > 0.85，加速比 3–4×
  • 格式化输出：JSON schema、HTML 模板、CSV，draft 极易命中
  • 低温度 / greedy 采样：分布集中，draft 和 target 分布更接近
  • Batch size = 1（online serving）：单请求 latency 优化效果最显著
  • 大模型（>70B）target：每次 target 前向越贵，speculation 相对成本越低
  • 领域内对话：客服、法律、医疗等专业领域，句式固定，草稿命中率高
  • Translation（机器翻译）：句式结构相对可预测
  • Prompt 内容 copy：RAG、摘要场景，输出大量复制 prompt 中的词汇

低收益 / 无效场景

  • 高温度创意写作：分布熵高，draft 频繁被拒，退化为正常速度
  • 大 batch 推理（offline）：batch=32+ 时 target 已 compute-bound，speculation 收益消失
  • 短序列生成：输出 < 20 tokens，推测带来的 warmup/overhead 不合算
  • Draft 与 target 分布差异大：不同训练方法/数据的模型组合效果差
  • MoE 大模型（激活稀疏）：实际 FLOP 少，target 前向已较快，margin 更小
  • CPU 推理：通常 compute-bound，带宽优化意义有限
  • 多模态任务：视觉 token 预测不确定性极高，接受率低
  • Chain-of-Thought 自由推理：reasoning token 多样性高，draft 难以提前预测

核心判断标准：接受率与 batch size

实践经验法则
  • 若 ᾱ < 0.6，speculative decoding 收益有限，甚至可能因为 overhead 而变慢
  • 若 batch_size × γ > compute_bound_threshold，target 前向从 memory-bound 转为 compute-bound，speculation 失效
  • EAGLE/Medusa 等方法因 draft 与 target 对齐程度高，通常 ᾱ 比独立 draft model 高出 0.1–0.2

10. 哪些场景下无效或有害

Batch Size 临界点

当请求 batch size 增大时，目标模型从 memory-bandwidth-bound 转变为 compute-bound。 设转变点为 Bcrit = compute_flops / (memory_bandwidth × 2N/FLOP per token)， 当 batch_size > Bcrit 时，每增加 γ 个 draft token 的验证都需要实质性额外计算， speculation 的时间效益趋近于零甚至为负（因为增加了序列长度，attention quadratic 代价上升）。

注意
在高并发生产环境中（大 batch），speculative decoding 可能会 降低 throughput。 vLLM 等框架会在 batch size 超过阈值时自动降级到标准 decoding。

KV Cache 管理复杂性

当 draft token 被拒绝时，需要 回滚（roll back）这些 token 对应的 KV-cache entry。 在使用 PagedAttention 的系统中，这意味着需要释放对应的 page 或维护 "speculative KV-cache"， 增加了内存管理的复杂度，也是各框架实现的主要工程难点。

投机生成的额外显存消耗

同时维护 draft model 和 target model（独立 draft model 方案）需要约 2× target 显存 （若 draft 和 target 大小相近则更多），这在 GPU 显存紧张时不可行。 Medusa/EAGLE 等 "attached head" 方案的额外显存开销只有 <1%，更适合生产部署。

12. 框架实现 — vLLM

vLLM（v0.4+ 起引入 speculative decoding）是目前实现最完整的开源推理框架之一。 其 speculative decoding 实现涵盖：

  • 独立 draft model（ngram + small LLM）
  • Medusa heads
  • Prompt Lookup Decoding（PLD）
  • MLPSpeculator（IBM 提出的轻量 MLP draft）

核心架构

# vLLM speculative decoding 核心调用链（简化）
class SpecDecodeWorker:
    def execute_model(self, execute_model_req):
        # Step 1: Draft phase
        proposals = self.proposer.get_proposals(
            execute_model_req,
            num_speculative_tokens=self.num_speculative_tokens
        )
        # Step 2: Score with target model（并行验证）
        proposal_scores = self.scorer.score_proposals(
            execute_model_req, proposals
        )
        # Step 3: 按接受拒绝规则选出最终 tokens
        accepted_token_ids, target_logprobs = self.rejection_sampler(
            proposal_scores.probs,   # target 概率
            proposals.probs,         # draft 概率
            proposals.token_ids
        )
        return accepted_token_ids

关键工程细节

  • Bonus token：全部接受后额外从 target top-1 或 sampling 得到第 γ+1 个 token
  • KV Cache 回滚：使用 free_request_blocks 删除被拒绝的 speculative slot
  • Dynamic speculation length：根据最近 N 步的接受率动态调整 γ（vLLM v0.6+）
  • Batch fallback：当 batch size 超过设定阈值时自动切换回标准 decoding
  • Chunked prefill：speculation 与 chunked prefill 兼容，支持 continuous batching

13. 框架实现 — TensorRT-LLM

NVIDIA TensorRT-LLM 在硬件层面深度优化了 speculative decoding， 提供了目前最高效的工程实现之一，支持：

  • Draft model（独立小模型）
  • Medusa + tree decoding（含 tree attention kernel）
  • EAGLE（从 v0.9 起原生支持）
  • ReDrafter（循环推测）
  • Lookahead decoding

TRT-LLM 的工程亮点

  • Fused Tree Attention Kernel：专为 speculative token 树实现了 custom CUDA kernel， 相比 naive multi-pass attention 快 2–3×
  • In-flight batching with spec decode：在连续批处理场景下， 不同请求可有不同 speculation 长度，通过 padded batch 实现
  • KV cache 预分配：为 speculative token 预留 KV slot， 避免动态分配开销
  • FP8 量化兼容：draft 和 target 均可使用 FP8 量化，进一步降低带宽开销

14. 框架实现 — SGLang

SGLang（v0.3+）针对结构化生成场景优化了 speculative decoding， 其特点是 EAGLE-2 的原生一流支持，且与 RadixAttention（prompt KV-cache 共享）深度集成：

  • EAGLE / EAGLE-2 集成，含 adaptive draft tree 构建
  • RadixAttention + speculation：即使在带 prefix caching 的情况下， draft token 的 KV-cache 仍可部分复用前缀
  • 支持 multi-step scheduling：一次 scheduler 调度执行多步 draft，减少调度开销
  • benchmark 显示 SGLang + EAGLE-2 在单卡 LLaMA-3-70B 上可达 4.0× tokens/s

SGLang 中的 EAGLE 实现片段

# SGLang EAGLE draft step（简化）
class EAGLEDraftRunner:
    def draft(self, hidden_states, token_ids, num_steps):
        draft_tokens = []
        h = hidden_states  # target 上一步特征向量
        for _ in range(num_steps):
            # 1-layer transformer: [h_prev, emb(x)] → h_next
            h = self.draft_layer(
                torch.cat([h, self.embed(token_ids)], dim=-1)
            )
            logits = self.lm_head(h)
            # tree expansion: sample top-k candidates
            top_ids = logits.topk(self.expand_factor).indices
            draft_tokens.append(top_ids)
            token_ids = top_ids[..., 0]  # greedy path for next step
        return draft_tokens

15. 框架实现 — DeepSpeed-FastGen

Microsoft DeepSpeed-FastGen（MII）通过 Dynamic SplitFuse 协议结合 speculative decoding：将 speculative verification pass 嵌入 SplitFuse 的 "fill" phase，与新请求的 prefill 合并处理，减少 GPU 空闲时间。 支持：

  • 独立 draft model（需手动配置）
  • 与 ZeRO++ 量化通信兼容（多 GPU 场景）
  • Tensor Parallelism 下的 speculative decoding（TP 分片的 draft 与 target 共用 AllReduce）

16. 框架实现 — HuggingFace TGI & Transformers

HuggingFace transformers 库原生支持 speculative decoding（v4.38+）， 通过 generate() 的 assistant_model 参数启用： 

from transformers import AutoModelForCausalLM, AutoTokenizer

target = AutoModelForCausalLM.from_pretrained("meta-llama/Llama-2-70b-chat-hf")
draft  = AutoModelForCausalLM.from_pretrained("meta-llama/Llama-2-7b-chat-hf")

outputs = target.generate(
    input_ids,
    assistant_model=draft,     # speculative decoding 启用
    num_speculative_tokens=4,  # γ = 4
    do_sample=True,
    temperature=0.7
)

TGI（Text Generation Inference）服务端同样支持， 通过配置 --speculate 4 和 --draft-model-id 启用。 TGI 的 speculative decoding 实现采用 continuous batching 兼容方式， 不同请求可分别启用/禁用 speculation。

17. 实践调优指南

关键参数

参数推荐范围说明
num_speculative_tokens (γ) 3–8 大多数场景 γ=4 是较好的起点。EAGLE-2 等自适应方法可设置最大值。
temperature 0–0.7 最佳 低温度接受率高，speculative decoding 收益更大
draft_model_size target 的 1/10–1/20 draft 太大则验证 overhead 增加；太小则接受率低
max_batch_size_for_spec 1–8（单 GPU） 超过此值自动关闭 speculative decoding，改用标准推理

Profiling 建议

  1. 先用小样本集测量 ᾱ（acceptance rate）。若 ᾱ < 0.65，考虑换 draft model 或使用 EAGLE/Medusa。
  2. 对比有/无 speculative decoding 的 tokens/sec（而非 ms/step），以实际吞吐为准。
  3. 关注 GPU utilization：理想情况下 target model 前向仍然应该高度占用 SM。
  4. 在 online serving 场景中，分别测试 p50/p90/p99 TTFT 和 TPOT，而不是只看平均值。
  5. 使用 --speculate-acceptance-rate-threshold（vLLM）动态关闭低接受率序列的 speculation。
