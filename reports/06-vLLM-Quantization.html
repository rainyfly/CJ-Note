vLLM 量化支持深度解析

TYPE: REPO ANALYSIS DATE: 2026-04-10 TOPIC: Quantization / CUDA Kernels / KV Cache BASE: vLLM v1

TL;DR

  • vLLM 通过统一的 LinearMethodBase 接口支持 FP8、AWQ、GPTQ/Marlin、BitsAndBytes、GGUF 多种量化方案，模型层无感知切换。
  • FP8 W8A8 是 H100/H200 Hopper GPU 的首选量化策略，权重 2× 压缩，算力提升 2×，精度损失极小。
  • Marlin GEMM 内核针对 Ampere+ GPU 将 GPTQ/AWQ INT4 推理提速 4× 以上，利用 INT4 TensorCore 指令与 Persistent Kernel 技术。
  • KV Cache FP8 量化（reshape_and_cache 内核）可将 KV Cache 显存压缩 2×，对长序列大 batch 场景效果显著。
  • 量化方案选择取决于 GPU 代际：Hopper 优选 FP8、Ampere 优选 AWQ/GPTQ+Marlin、边缘设备可选 BitsAndBytes NF4 或 GGUF。

1. 量化概述

量化通过降低模型权重/激活值的数据精度来减少显存占用和加速推理。以 7B 参数模型为例，下面展示了三种典型配置下的显存需求：

BF16 基准（7B）
~30 GB
W4A16 AWQ/GPTQ（7B）
~20 GB
W8A8 FP8（7B）
~15 GB
BF16 基准（7B模型）:
  权重显存: 7B × 2 bytes = 14 GB
  KV Cache: ~16 GB（batch=32, seq=2048）
  总计: ~30 GB（需要 A100 40GB）

W4A16 量化（AWQ/GPTQ）:
  权重显存: 7B × 0.5 bytes = 3.5 GB（4× 压缩）
  KV Cache: ~16 GB（不变）
  总计: ~20 GB（可用 RTX 3090 24GB）

W8A8 FP8 量化:
  权重显存: 7B × 1 byte = 7 GB（2× 压缩）
  KV FP8: ~8 GB（2× 压缩）
  总计: ~15 GB（可用 RTX 4090 24GB）

2. 量化框架架构

文件路径：vllm/model_executor/layers/quantization/

vLLM 使用注册表模式管理量化方案，每种量化类型对应一个 QuantizationConfig 子类，在模型构建时将标准 Linear 层替换为量化版本。

量化注册表与模型构建流程

graph TD A[model_config.quantization = "fp8"] --> B[QuantizationConfig.get_quant_cls] B --> C{量化注册表} C -->|fp8| D[Fp8Config] C -->|awq| E[AWQConfig] C -->|gptq| F[GPTQConfig] C -->|marlin| G[MarlinConfig] C -->|bitsandbytes| H[BitsAndBytesConfig] C -->|gguf| I[GGUFConfig] D --> J[build_model] E --> J F --> J G --> J H --> J I --> J J --> K[ColumnParallelLinear → Fp8LinearMethod] J --> L[RowParallelLinear → Fp8LinearMethod]

2.1 量化 Linear 层的接口

所有量化方法实现同一个抽象基类，确保模型层代码无需感知底层量化细节：

class LinearMethodBase(ABC):
    """量化 Linear 的基类"""

    @abstractmethod
    def create_weights(
        self, input_size, output_size, ...
    ) -> dict[str, torch.Tensor]:
        """创建量化权重张量（INT4/INT8/FP8 等格式）"""

    @abstractmethod
    def apply(
        self, layer: torch.nn.Module, x: torch.Tensor, ...
    ) -> torch.Tensor:
        """执行量化矩阵乘法"""

3. FP8 量化

文件路径：vllm/model_executor/layers/quantization/fp8.py

3.1 W8A8 FP8（推荐方案）

推荐场景 FP8 W8A8 是 H100/H200 Hopper GPU 的生产首选。原生 FP8 TensorCore 使算力达到 BF16 的 2×，同时 E4M3 格式动态范围足够保持精度。 
FP8 格式: E4M3（范围更大，适合激活值）/ E5M2（范围更大）

W8A8:
  权重: FP8 存储（~1 byte/param）
  激活: FP8 量化（动态量化，每次前向计算时缩放）

支持硬件: NVIDIA H100/H200/Blackwell（原生 FP8 TensorCore）
          NVIDIA A100（模拟 FP8，略慢）

优势:
  - 权重 2× 压缩
  - H100 FP8 TensorCore 是 BF16 的 2× 算力
  - 精度损失极小（E4M3 动态范围足够）

3.2 FP8 量化内核

FP8 GEMM 执行流程（csrc/quantization/fp8.cu）

graph LR A["input_bf16 [M,K]"] --> B["动态量化\nscale_x = max(abs(x)) / FP8_MAX"] B --> C["input_fp8 [M,K]"] D["weight_fp8 [K,N]"] --> E["CUTLASS FP8 GEMM"] C --> E E --> F["output_fp32 [M,N]"] F --> G["× scale_w × scale_x"] G --> H["output_bf16 [M,N]"]

3.3 Per-tensor vs Per-channel 量化

Per-tensor 量化:
  整个 weight 矩阵用一个 scale 值
  简单，但对权重分布不均匀时精度差

Per-channel 量化（FP8 recommended）:
  weight.shape = [out_features, in_features]
  scale.shape = [out_features, 1]  ← 每个输出通道一个 scale
  精度更好，内核稍复杂

Channelwise activation scaling:
  动态计算每个 token 的激活 scale
  scale_x = max(abs(x)) per token

4. AWQ 量化

文件路径：vllm/model_executor/layers/quantization/awq.py

4.1 原理

AWQ（Activation-aware Weight Quantization）是激活感知的 W4A16 量化。其核心洞察是：约 1% 的权重通道对激活值中"显著"通道具有支配性影响，直接量化这些通道会造成较大误差。

AWQ 量化校准与推理流程

graph TD A["分析激活值分布"] --> B["识别重要通道\n（激活幅度大的 ~1%）"] B --> C["对重要通道权重做缩放\nscale = activation_scale"] C --> D["对缩放后权重做 INT4 量化\n误差更均匀"] D --> E["存储 weight_int4 + scale + zero"] E --> F{"推理时"} F --> G["weight_dequant = weight_int4 × scale\n反量化到 FP16"] G --> H["GEMM: output = input_fp16 @ weight_fp16"]

4.2 AWQ 推理内核

AWQ 量化 Linear 执行:
  weight_int4 [out, in/8]  ← 每个 uint8 存 2 个 INT4
  weight_scale [out, in/group_size]  ← 每组（默认128）一个缩放因子
  weight_zero [out, in/group_size]   ← 零点

  CUTLASS AWQ GEMM (csrc/quantization/awq/):
    1. 反量化: w_fp16 = (weight_int4 - zero) × scale
    2. GEMM: output = input_fp16 @ w_fp16

    注意: Marlin GEMM 是 AWQ 在 Ampere+ GPU 上的高性能实现

5. GPTQ / Marlin 量化

5.1 GPTQ 原理

GPTQ（Generative Pre-Trained Transformers Quantization）:
  - 逐层量化，每层独立校准
  - 使用 Hessian 矩阵指导量化顺序（误差最小化）
  - 支持 INT4/INT8

Marlin（GPTQ 的高性能 CUDA 内核）:
  - 针对 Ampere/Hopper GPU 优化的量化 GEMM
  - 比原始 GPTQ 内核快 4× 以上
  - 支持: W4A16, W4A8, W8A16

5.2 Marlin 执行流程

Marlin GEMM 内核执行流程（csrc/quantization/marlin/）

graph TD subgraph offline["离线预处理（一次性）"] A["weight_int4（原始 GPTQ 格式）"] --> B["重新排列为 Marlin 内存布局\n优化 GPU L2 Cache 命中率"] end subgraph online["推理（每次前向）"] C["读取 Marlin 格式 weight_int4"] --> D["反量化\nweight_fp16 = dequant(weight_int4, scale, zero)"] D --> E["超高速 GEMM\noutput = input @ weight_fp16"] end B --> C subgraph opts["关键优化"] F["INT4 TensorCore 指令\n(NVIDIA Ampere)"] G["Persistent Kernel\n长驻 SM，减少 launch 开销"] H["复用输入 TensorCore 矩阵\n减少 DRAM 带宽"] end

6. BitsAndBytes（NF4/INT8）

BitsAndBytes 支持:
  INT8: W8A8，简单量化，适合边缘场景
  NF4: 4-bit NormalFloat，QLoRA 使用的格式

特点:
  - 不需要量化校准数据
  - 支持动态量化（推理时逐层量化）
  - 精度低于 AWQ/GPTQ（无校准优化）

使用方式:
  LLM(model="...", quantization="bitsandbytes", load_in_4bit=True)
注意 BitsAndBytes 无需校准数据，使用门槛低，但由于缺乏激活感知优化，推理精度通常低于 AWQ/GPTQ。适合资源受限的边缘场景或快速实验，不建议用于生产高精度服务。

7. GGUF 量化

GGUF（llama.cpp 格式）:
  支持 Q4_0, Q4_K_M, Q5_K_M, Q6_K, Q8_0 等格式

  Q4_K_M（流行格式）:
    大部分权重: 4-bit + 独立量化缩放因子
    关键层（attention、embedding）: 6-bit

vLLM GGUF 支持 (csrc/quantization/gguf/):
  直接加载 .gguf 文件
  使用专用 GGUF dequant 内核

  限制:
    不支持 LoRA
    推理速度略低于 CUTLASS 量化方案
使用场景 GGUF 主要用于兼容 llama.cpp 生态。如果模型已以 GGUF 格式发布，vLLM 可直接加载，无需转换。但若追求最高推理吞吐，建议转换为 AWQ/GPTQ 格式使用 CUTLASS/Marlin 内核。

8. KV Cache 量化

文件路径：vllm/model_executor/layers/quantization/kv_cache.py

KV Cache FP8 量化写入/读取流程

sequenceDiagram participant A as Attention 计算层 participant B as reshape_and_cache 内核 participant C as KV Cache (FP8 存储) participant D as FlashAttention/FlashInfer A->>B: key_fp16, val_fp16 Note over B: k_scale = max(abs(key)) / FP8_MAX
v_scale = max(abs(val)) / FP8_MAX B->>C: key_fp8, val_fp8, k_scale, v_scale C->>D: key_fp8 + k_scale Note over D: 内核内部反量化
key_fp8 → BF16/FP16 D->>A: 注意力输出 
KV Cache FP8 量化流程:

写入时（reshape_and_cache 内核）:
  key_fp16 → key_fp8（使用动态 scale）
  val_fp16 → val_fp8

  k_scale = max(abs(key)) / FP8_MAX  ← 全局 scale（运行时统计）
  v_scale = max(abs(val)) / FP8_MAX

读取时（FlashAttention/FlashInfer 内核内部）:
  key_fp8, k_scale → 反量化到 FP16/BF16
  然后正常计算注意力

显存节省:
  BF16 → FP8: 2× 压缩
  对于长序列/大 batch，KV Cache 可能占总显存的 50%+
  FP8 KV Cache 可显著提升系统容量

9.2 内核选择逻辑

vLLM 在运行时根据 GPU 架构自动选择最优内核，以 FP8 为例：

# 根据 GPU 架构选择最优内核

class Fp8LinearMethod(LinearMethodBase):
    def apply(self, layer, x):
        if current_platform.is_hopper():
            # H100/H200: 使用原生 FP8 TensorCore
            return cutlass_fp8_gemm(x, weight, scale_x, scale_w)
        elif current_platform.is_ampere():
            # A100: 模拟 FP8（先转 FP16 再 GEMM）
            w_fp16 = dequant_fp8_to_fp16(weight, scale_w)
            return torch.mm(x, w_fp16)
        else:
            # 回退到 FP16
            return torch.mm(x, dequant(weight))
设计亮点 这种基于 current_platform 的运行时分发，使同一量化配置可在不同 GPU 世代上取得最佳性能，用户无需手动调整。Hopper 原生 FP8 算力是 Ampere 模拟路径的约 2×。
