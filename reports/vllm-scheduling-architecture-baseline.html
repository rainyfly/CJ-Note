1. 概述与背景

vLLM（Virtual Large Language Model）是由 UC Berkeley Sky Computing Lab 于 2023 年 6 月开源的 高性能 LLM 推理框架，核心论文为 "Efficient Memory Management for Large Language Model Serving with PagedAttention" （Kwon et al., SOSP 2023）。 它在学术界和工业界迅速成为 LLM 推理服务的事实标准参考实现。

核心创新点

  • PagedAttention：借鉴操作系统虚拟内存分页思想，消除 KV Cache 内存碎片
  • Continuous Batching：迭代级（iteration-level）动态调度，大幅提升 GPU 利用率
  • Block Manager：统一管理 GPU/CPU 上的物理 KV cache 块，支持 copy-on-write
  • 前缀共享（Prefix Caching）：相同前缀请求复用 KV cache 物理块
  • Tensor Parallelism / Pipeline Parallelism：多卡分布式推理

系统定位

vLLM 定位为在线推理服务（Online Serving）场景，目标是最大化服务吞吐（requests/s）同时 保持合理的请求延迟（TTFT / TPOT）。它通过 OpenAI 兼容 API 对外暴露服务。

核心挑战 LLM 推理的瓶颈不是计算（FLOPs），而是内存带宽和 KV Cache 管理。 一个 13B 参数模型在 FP16 下，每 token 的 KV cache 约需 800KB（取决于层数、头数、hidden dim）。 在 batch_size=32、sequence_length=2048 的场景下，KV cache 需要约 50GB 显存。

2. Continuous Batching（连续批处理）

2.1 动机：Static Batching 的局限

传统 LLM 推理服务采用 Static Batching（静态批处理）： 等待固定时间窗口内的请求凑成一批，然后一起处理，直到批内所有请求都完成才释放批槽。

Static Batching 问题示意： 时间轴 → 请求 A: [prefill][decode][decode][decode][decode][decode] (长序列) 请求 B: [prefill][decode] (短序列，已完成) 请求 C: 等待 ... 等待 ... 等待 (等 A 完成才能加入) 请求 D: 等待 ... 等待 ... 等待 GPU 利用率: ████████████████░░░░░░░░░░░░░░░░░░░░░░░░░░░ ~50% (批后半段大量槽位空闲，但 GPU 仍在运行 A)

这造成两个核心问题：

  • 队列等待（Head-of-line blocking）：新请求必须等待当前批最长请求完成
  • GPU 空泡（Bubble）：批内短序列完成后，对应的 KV cache slot 空置但无法被新请求利用

2.2 设计原理

vLLM 采用的 Continuous Batching（也称 Iteration-level Scheduling， 由 Orca 论文 [Yu et al., OSDI 2022] 首次提出）核心思想是： 在每一次 forward pass（每个 decode step）结束后重新进行调度决策， 而不是等一个完整请求 batch 结束。

Continuous Batching 示意： Step 1: Batch = [A(prefill), B(prefill), C(prefill)] → B 完成 prefill，进入 decode Step 2: Batch = [A(decode), B(decode1), C(decode)] → B decode 序列很短，第 2 步后 B 生成 EOS Step 3: Batch = [A(decode), D(prefill+decode1), C(decode)] ↑ 立即将排队的 D 填入 B 释放的槽位！ Step 4: Batch = [A(decode), D(decode2), C(decode)] ... GPU 利用率: █████████████████████████████████████████ ~95%

关键设计决策：

  • Prefill 与 Decode 混合：新请求的 prefill phase 和已有请求的 decode phase 可以在同一 batch 内并行运行（但二者计算模式不同，需要特殊处理）
  • 迭代级调度粒度：调度器在每个 decode step 介入，而非请求级别
  • 抢占与重调度：内存压力时可以抢占低优先级请求，释放 KV cache 块
效果 Continuous Batching 将 GPU 利用率从传统方法的 50–60% 提升到 90%+， 在长尾延迟分布场景下尤为显著。

2.3 工程实现

vLLM 的调度在 vllm/core/scheduler.py 中实现， 每次 forward step 调用 scheduler.schedule() 返回一个 SchedulerOutputs 对象，包含：

@dataclass
class SchedulerOutputs:
    scheduled_seq_groups: List[ScheduledSequenceGroup]
    # 本次 step 参与推理的序列组列表

    prompt_run: bool
    # True = 本次 batch 中包含 prefill 请求

    num_batched_tokens: int
    # 本次 batch 中总 token 数（prefill tokens + decode tokens）

    # CPU -> GPU KV cache 块映射（用于换入）

    blocks_to_swap_out: Dict[int, int]
    # GPU -> CPU KV cache 块映射（用于换出）

    # 需要 copy-on-write 的块映射

调度主循环（伪代码）

def schedule(self) -> SchedulerOutputs:
    # 1. 处理已完成/出错的序列，释放资源
    self._schedule_finished_seqs()

    # 2. 尝试将 SWAPPED（CPU上）的序列换回 GPU
    swapped_out = self._schedule_swapped()

    # 3. 尝试推进正在运行（RUNNING）的序列的 decode step
    running = self._schedule_running()

    # 4. 从等待队列（WAITING）中取入新请求进行 prefill
    if budget_remaining:
        waiting = self._schedule_prefills()

    # 5. 合并结果，生成 SchedulerOutputs
    return SchedulerOutputs(
        scheduled_seq_groups=running + waiting,
        blocks_to_swap_in=...,
        blocks_to_swap_out=...,
        ...
    )

3. PagedAttention（分页注意力）

3.1 动机：KV Cache 碎片化问题

在传统实现中，每个请求的 KV cache 被分配为一块连续的显存空间， 大小等于 max_seq_len × num_layers × 2 × num_heads × head_dim × dtype_size。

这带来三类内存浪费：

内部碎片（Internal Fragmentation）
预分配 > 实际
为应对最长序列而预分配 max_seq_len 空间， 实际生成短于预分配时浪费大量显存
外部碎片（External Fragmentation）
不连续空洞
不同长度请求完成后，留下大量无法利用的不连续碎片空间
预留开销（Reservation Overhead）
未知序列长度
服务启动时无法提前知道每个请求的最终输出长度， 保守估计导致过多预留
量化：传统方法的内存利用率 SOSP 2023 论文测量显示，传统 KV cache 管理在典型负载下， 实际利用率仅有 20–40%，即超过 60% 的显存被浪费在碎片和预留上。

3.2 核心设计：分页虚拟内存

PagedAttention 借鉴操作系统分页虚拟内存（Paged Virtual Memory）的思想：

  • 将 KV cache 切分为固定大小的 物理块（Physical Block）， 每块包含 block_size 个 token 的 KV（默认 block_size = 16）
  • 每个序列维护一个 逻辑块表（Block Table）， 将逻辑块号映射到物理块号
  • 物理块按需动态分配，不需要连续；释放时立即回收
PagedAttention 内存映射示意： 序列 A（15 tokens）逻辑视图： Logical Block: [ 0 ][ 1 ] Block 0: tokens 0-7（8 tokens） Block 1: tokens 8-14（7 tokens，最后 1 slot 空置） 序列 B（9 tokens）逻辑视图： Logical Block: [ 0 ][ 1 ] Block 0: tokens 0-7（8 tokens） Block 1: token 8（1 token，7 slots 空置，但这是最后一块，暂时允许） 物理内存布局（GPU HBM）： ┌─────────┬─────────┬─────────┬─────────┬─────────┐ │Phys #0 │Phys #1 │Phys #2 │Phys #3 │Phys #4 │ │(A 逻辑0)│(B 逻辑0)│(A 逻辑1)│(B 逻辑1)│ 空闲 │ └─────────┴─────────┴─────────┴─────────┴─────────┘ Block Table: 序列 A: logical[0] → physical[0], logical[1] → physical[2] 序列 B: logical[0] → physical[1], logical[1] → physical[3]

Copy-on-Write (CoW) 机制

vLLM 支持 Beam Search 和并行采样。多个序列（候选）可以共享同一个物理块 （引用计数 > 1）。当某个序列需要写入新 token 时， 若目标物理块被多个序列引用，则触发 Copy-on-Write： 

# block_manager.py 中的 CoW 逻辑（简化）
def append_slots(self, seq: Sequence) -> List[Tuple[int, int]]:
    """为序列追加新 token 的 KV cache 槽位，返回需要 CoW 的块对"""
    cows = []  # (src_block, dst_block) pairs

    logical_blocks = seq.logical_token_blocks
    block_table = self.block_tables[seq.seq_id]

    if len(block_table) < len(logical_blocks):
        # 需要分配新物理块
        new_block = self.gpu_allocator.allocate()
        block_table.append(new_block)
    else:
        # 当前最后一个物理块
        last_block = block_table[-1]
        if last_block.ref_count > 1:
            # 共享块 -> 触发 CoW
            new_block = self.gpu_allocator.allocate()
            self.gpu_allocator.free(last_block)  # 减引用计数
            block_table[-1] = new_block
            cows.append((last_block.block_number, new_block.block_number))

    return cows

前缀缓存（Prefix Caching）

vLLM v2 引入 Automatic Prefix Caching（APC）： 对每个物理块计算其内容的哈希值，若新请求的前缀块与已有请求完全一致， 则直接复用已有物理块，无需重新计算这些 token 的 KV。 

# 前缀缓存键：块内所有 token id 的哈希 + 父块哈希（链式）
block_hash = hash(tuple(token_ids_in_block) + (parent_block_hash,))

3.3 CUDA Kernel 实现

标准的 Flash Attention 要求 KV cache 连续，无法直接用于 PagedAttention。 vLLM 实现了专用的 CUDA kernel：

Paged Attention Kernel

// vllm/attention/backends/paged_attn.py 调用的 CUDA kernel 签名
void paged_attention_v1(
    torch::Tensor& out,           // [num_seqs, num_heads, head_size]
    torch::Tensor& query,         // [num_seqs, num_heads, head_size]
    torch::Tensor& key_cache,     // [num_blocks, num_heads, head_size/x, block_size, x]
    torch::Tensor& value_cache,   // [num_blocks, num_heads, head_size, block_size]
    int64_t num_kv_heads,
    double scale,
    torch::Tensor& block_tables,  // [num_seqs, max_num_blocks_per_seq]
    torch::Tensor& context_lens,  // [num_seqs] - 每个序列已有多少 token
    int64_t block_size,
    int64_t max_context_len,
    const c10::optional<torch::Tensor>& alibi_slopes
);

Kernel 的核心计算流程：

  1. 每个 CUDA block 负责一个 (query_head, sequence) 对
  2. 从 block_tables 中读取该序列的物理块映射
  3. 按物理块逐块读取 K、V，计算局部 attention 分数
  4. 使用在线 softmax（Flash Attention 风格）合并结果
FlashAttention vs PagedAttention Kernel 的差异 FlashAttention 要求 Q/K/V 在序列维度连续存储，适合 prefill phase（一次处理整个 prompt）。 PagedAttention kernel 则设计用于 decode phase，支持非连续物理块的高效访问。 vLLM 根据 phase 自动选择对应的 kernel。

KV Cache 物理布局

# Key cache 的张量形状（以 block_size=16, head_dim=128 为例）
key_cache shape:
  [num_blocks,       # 总物理块数
   num_kv_heads,     # KV head 数量
   head_dim // x,    # x = 8 for FP16（向量化访问优化）
   block_size,       # 16
   x]                # 8

# 实际访问时：
# key[block_idx, head_idx, :, token_idx_in_block, :] 取一个 token 的 key vector

这种 layout 将同一 block 内相邻 token 的 key 向量在内存中排列紧密， 利于 CUDA 的 warp-level 合并访问（coalesced memory access）， 显著提升内存带宽利用率。

4. 调度器架构

4.1 整体架构

vLLM 的调度器实现于 vllm/core/scheduler.py， 与 LLM Engine（vllm/engine/llm_engine.py）配合工作。

序列状态机

每个序列（Sequence）在调度器中经历以下状态：

┌─────────┐ add_request() ┌─────────┐ │ │ ─────────────────→ │ │ │ 初始化 │ │ WAITING │ │ │ │(等待队列)│ └─────────┘ └────┬────┘ │ 获得足够 KV cache 块 │ schedule() 选中 ▼ ┌─────────┐ 内存压力/抢占 ┌─────────┐ │ │ ─────────────────→ │ │ │ RUNNING │ │ SWAPPED │ │(运行队列)│ ←───────────────── │(换出队列)│ └────┬────┘ 内存充足/换回 └─────────┘ │ 生成 EOS token 或 │ 达到 max_tokens ▼ ┌─────────┐ │FINISHED │ │(释放资源)│ └─────────┘

4.2 调度策略与优先级

vLLM 默认采用 FCFS（First-Come-First-Served） 策略， 同时支持优先级调度。调度时遵循以下原则：

  1. 优先保证 RUNNING 请求：已在运行的请求优先获得 KV cache 块， 避免中断已部分完成的推理
  2. 其次处理 SWAPPED 请求：将从 CPU 换回的请求恢复执行
  3. 最后接入 WAITING 请求：在剩余预算内进行新请求的 prefill

4.3 抢占机制

当 GPU 内存不足以容纳所有 RUNNING 序列时，调度器需要抢占（Preempt） 部分序列，释放其 KV cache 块。vLLM 支持两种抢占策略：

策略 实现方式 优点 缺点
Swap（换出） 将 GPU 上的 KV cache 块搬移到 CPU RAM 序列状态完整保留，恢复时无需重新 prefill PCIe 带宽是瓶颈，换入/换出有延迟
Recompute（重计算） 直接丢弃 KV cache 块，序列重入 WAITING 队列 无 CPU 内存压力，实现简单 需要重新 prefill，浪费计算 
# 抢占时的选择策略（默认选最后加入 RUNNING 的序列，即 LIFO）
def _preempt(self, seq_group: SequenceGroup,
             blocks_to_swap_out: Dict[int, int]) -> PreemptionMode:
    if self.user_specified_preemption_mode is None:
        # 启发式：如果序列只有 1 个序列（非 beam search），用 recompute
        # 否则（beam search），用 swap
        if seq_group.get_max_num_running_seqs() == 1:
            return PreemptionMode.RECOMPUTE
        else:
            return PreemptionMode.SWAP
    return self.user_specified_preemption_mode

5. Block Manager 与 KV Cache 管理

vllm/core/block_manager.py 是 vLLM 内存管理的核心， 实现了类似 OS 内存管理器的功能：

核心数据结构

class BlockAllocator:
    """管理一类设备（GPU 或 CPU）上的物理块"""
    def __init__(self, device: Device, block_size: int, num_blocks: int):
        self.free_list: BlockTable = []   # 空闲物理块列表
        self.device = device
        self.block_size = block_size

    def allocate(self) -> PhysicalTokenBlock:
        if not self.free_list:
            raise OutOfMemoryError("KV cache out of memory")
        block = self.free_list.pop()
        block.ref_count = 1
        return block

    def free(self, block: PhysicalTokenBlock) -> None:
        block.ref_count -= 1
        if block.ref_count == 0:
            self.free_list.append(block)  # 真正释放

class BlockSpaceManagerV1:
    """统一管理 GPU/CPU 上的块分配，维护每个序列的 block table"""

    gpu_allocator: BlockAllocator   # GPU HBM 上的块
    cpu_allocator: BlockAllocator   # CPU RAM 上的块（用于 swap）

    # seq_id -> List[PhysicalTokenBlock]
    block_tables: Dict[int, BlockTable]

    def can_allocate(self, seq_group) -> AllocStatus:
        """检查是否有足够的 GPU 块来容纳一个新请求"""
        num_required_blocks = seq_group.get_num_required_blocks()
        num_free_blocks = self.gpu_allocator.get_num_free_blocks()

        if num_free_blocks >= num_required_blocks:
            return AllocStatus.OK
        elif num_free_blocks > 0:
            return AllocStatus.LATER   # 等内存释放
        else:
            return AllocStatus.NEVER   # 请求本身就超出最大容量

Swap 操作流程

def swap_out(self, seq_group: SequenceGroup) -> Dict[int, int]:
    """GPU -> CPU，返回 {gpu_block_id: cpu_block_id} 映射"""
    mappings = {}
    for seq in seq_group.get_seqs(SequenceStatus.RUNNING):
        new_block_table = []
        for gpu_block in self.block_tables[seq.seq_id]:
            # 为每个 GPU 块在 CPU 上分配对应块
            cpu_block = self.cpu_allocator.allocate()
            mappings[gpu_block.block_number] = cpu_block.block_number
            new_block_table.append(cpu_block)
            self.gpu_allocator.free(gpu_block)
        self.block_tables[seq.seq_id] = new_block_table
    return mappings  # Worker 用此映射执行实际的显存到内存拷贝

BlockManagerV2（vLLM 2.x）

较新版本引入 BlockSpaceManagerV2，重构了块分配逻辑以支持：

  • Automatic Prefix Caching（APC）：基于内容哈希的前缀共享
  • Sliding Window Attention：只保留窗口内的 KV cache
  • 更灵活的块大小：不同 layer 可以有不同配置
  • Speculative Decoding 集成：Draft model 与 target model 的 KV cache 协同管理

6. 分布式推理架构

Tensor Parallelism（张量并行）

vLLM 使用 Megatron-LM 风格的张量并行，将 Attention 层和 FFN 层按 head/hidden dim 切分到多卡：

Tensor Parallel (TP=4) for Attention: GPU 0: Q_heads[0..7], K_heads[0..7], V_heads[0..7] GPU 1: Q_heads[8..15], K_heads[8..15], V_heads[8..15] GPU 2: Q_heads[16..23], K_heads[16..23], V_heads[16..23] GPU 3: Q_heads[24..31], K_heads[24..31], V_heads[24..31] 每卡独立计算本地 heads 的 attention → AllReduce 合并结果（只在层输出时做一次通信） 每卡存储 1/TP 的 KV cache，总 KV cache 容量 = 单卡容量 × TP

Pipeline Parallelism（流水线并行）

PP 将模型层按顺序分配到不同 GPU，vLLM 使用微批流水线减少 bubble：

Pipeline Parallel (PP=2): Stage 0 (GPU 0,1 with TP=2): Layer 0-15 Stage 1 (GPU 2,3 with TP=2): Layer 16-31 Micro-batch 流水线（减少 idle 时间）： Time: t1 t2 t3 t4 GPU0: [B1] [B2] [B3] [B4] GPU2: [B1] [B2] [B3] ↑点对点通信（activation 传递）

  • cuDSMEM / cuMemcpy：同节点 GPU 间直接显存拷贝（NVLink/PCIe）
  • 异步 swap：KV cache swap 操作在独立 CUDA stream 中执行， 与计算重叠

Latency 分析

vLLM 中的延迟组成：

  • TTFT（Time To First Token）： = 调度延迟 + prefill 计算时间。 continuous batching 下，若 prefill 与 decode 共存于同一 batch， 大 prompt 的 prefill 会阻塞 decode 序列的当步推进（chunked prefill 可缓解）
  • TPOT（Time Per Output Token）： ≈ decode 步耗时 / batch_size。 memory-bound 场景下，扩大 batch 不增加延迟（直到 compute-bound 转换点）
  • 队列等待时间：WAITING 队列中等待调度的时间， 受当前并发量和 KV cache 可用量影响

请求生命周期（完整流程）

Step 1

请求接入

AsyncLLMEngine.generate() 接收请求，tokenize，创建 SequenceGroup，加入 WAITING 队列

Step 2

调度决策

Scheduler.schedule() 检查 RUNNING/SWAPPED/WAITING 队列， 调用 BlockManager 分配 KV cache 块，生成 SchedulerOutputs

Step 3

输入准备

ModelRunner.prepare_model_input() 根据 SchedulerOutputs 构建 attention metadata（block tables, context lengths, slot mappings）

Step 4

前向传播

模型 forward pass，Attention 层调用 PagedAttention kernel 读写 KV cache 物理块

Step 5

采样

Sampler 对 logits 进行 temperature/top-p/top-k 采样，生成下一 token

Step 6

输出处理

检查是否生成 EOS/达到 max_tokens，更新序列状态，释放完成序列的 KV cache 块， 将结果 token 流式返回给客户端

Step 7

循环

回到 Step 2，执行下一个 decode step（continuous batching 的核心）

重要配置参数

参数 默认值 说明
block_size 16 每个 KV cache 物理块包含的 token 数，影响内存碎片和 kernel 效率
gpu_memory_utilization 0.90 GPU 显存中用于 KV cache 的比例（其余留给模型权重和激活值）
max_num_seqs 256 同时运行的最大序列数（受显存和 batch 矩阵大小限制）
max_num_batched_tokens auto 每次 forward 的最大 token 数，影响 prefill 吞吐和延迟
swap_space 4 GB CPU swap 空间大小，用于 SWAPPED 状态序列的 KV cache 存储
enable_prefix_caching False 启用 Automatic Prefix Caching（APC），相同前缀复用 KV cache
chunked_prefill_enabled False 将长 prefill 分块与 decode 交替执行，降低 TTFT 抖动

9. 总结与展望

核心贡献总结

创新点 解决的问题 技术手段
PagedAttention KV cache 内存碎片（60-80% 浪费） OS 分页思想 + 定制 CUDA kernel
Continuous Batching GPU 空泡、长尾延迟、HOL blocking 迭代级调度，step 级 batch 重组
CoW 共享块 Beam Search / 并行采样的 KV 冗余 引用计数 + 写时复制
Prefix Caching 重复前缀的重复计算 块级内容哈希，跨请求复用
Swap 机制 内存峰值抢占恢复 GPU↔CPU 块迁移 + 状态机管理

局限性与未来方向

已知局限
  • Prefill-Decode 干扰：Continuous batching 中 prefill 会占用大量 token budget， 导致同 batch 的 decode 请求延迟上升（Chunked Prefill 可缓解但增加系统复杂度）
  • 调度器单点：调度器是单线程 Python 进程，在极高 QPS 下可能成为瓶颈
  • PCIe 带宽限制 Swap：CPU-GPU swap 速度受限于 PCIe 带宽（~32 GB/s）， 频繁抢占会显著降低吞吐
  • Block 碎片（最后一块）：每个序列的最后一个物理块可能只填了少量 token， 该部分显存暂时无法复用

社区发展方向

已实现

Chunked Prefill

将 prefill 分成多个 chunk 与 decode 交替执行，减少 TTFT 抖动， 改善 SLA 保证（vLLM v0.4+）

已实现

Speculative Decoding

用小 draft model 预测多步，target model 验证，提升吞吐（vLLM v0.3+）

已实现

Disaggregated Prefill

将 prefill 和 decode 分配到不同 GPU 集群，彻底隔离两阶段互相干扰

进行中

vLLM V1 架构重构

更模块化的架构，更好的异步流水线，支持更多硬件后端（Intel GPU、AMD、TPU）

研究方向

KV Cache 压缩/卸载

结合量化（KVQuant）、稀疏化（Scissorhands）、NVMe 卸载（InfiniGen） 进一步扩展有效 KV cache 容量
