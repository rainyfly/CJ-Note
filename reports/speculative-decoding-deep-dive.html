Speculative Decoding Deep Dive

TYPE: FEATURE DEEP DIVE DATE: 2026-04-10 TOPIC: Decoding / Throughput / Latency

TL;DR

  • Speculative decoding 用一个小的 draft 模型快速生成 γ 个候选 token，再用大模型并行验证，输出分布与纯大模型等价（无损）。
  • 加速比取决于 draft 命中率 α：当 α 接近 1 时理论加速可达 γ+1 倍；当 α 接近 0 时反而比 naive 更慢。
  • 适合低并发/对话场景（memory-bandwidth bound），不适合高并发批量推理（compute bound）；draft 模型选型是关键工程决策。
  • 主流框架（vLLM, TRT-LLM, SGLang, HuggingFace）均已实现，但各有不同的 draft 策略：外部 draft 模型、同模型草稿头（Medusa）、n-gram lookup、EAGLE 等。
  • 没有免费的午餐：speculative decoding 引入了 draft 模型的内存开销和 KV cache 管理复杂度，并发数上升后收益快速消失。

1. Background — 为什么 LLM 推理是 Memory-Bandwidth Bound

自回归语言模型推理时，每生成一个 token 都需要把模型的全部参数从 HBM 加载进 SRAM/寄存器，进行一次前向传播，然后输出一个 token。这个"加载权重 → 产出一个 token"的过程是 memory-bandwidth bound 的：GPU 的算力（FLOP/s）远超实际利用率，因为带宽才是瓶颈。

以 Llama-3 70B（BF16，~140GB 参数）为例，在 batch=1 下每个 token 需要读取 ~140GB 参数，H100 SXM5 上大约需要 42ms，也就是约 24 tokens/s 上限，远未触及算力天花板。

核心观察 解码阶段每步只产出 1 个 token，但每步都要完整地过一次大模型。如果我们能在"每次大模型前向"中产出更多 token，就能把带宽利用率摊薄，提升有效吞吐。
Before — 标准自回归

每次大模型前向 → 产出 1 token → 下次前向。N 个 token 需要 N 次完整前向，带宽利用率低。

After — Speculative Decoding

小 draft 模型先快速生成 γ 个 token → 大模型一次并行验证全部 γ+1 个位置 → 平均每次大模型前向产出 >1 个 token。

Speculative Decoding 最早由两篇工作同期独立提出：Chen et al. 2023（Google DeepMind，"Accelerating Large Language Model Decoding with Speculative Sampling"，arXiv 2302.01318）和 Leviathan et al. 2023（Google，"Fast Inference from Transformers via Speculative Decoding"，ICML 2023，arXiv 2211.17192）。两者核心思想一致，后者给出了更严格的理论证明。

2. 核心机制：Draft + Verify

设大模型为 M（target model），小模型为 m（draft model）。推理分两阶段循环：

Phase 1 — Draft（起草）

使用 draft model m 自回归地生成 γ 个 token（γ 通常取 4–8）。Draft 模型很小（参数量通常是 target 的 1/10 至 1/100），速度很快，cost 可以忽略。

# Draft phase (simplified pseudocode)
draft_tokens = []
x = current_prefix  # already generated tokens
for i in range(gamma):
    q_i = draft_model.forward(x)         # logits from draft model
    t_i = sample(q_i)                    # sample a draft token
    draft_tokens.append(t_i)
    x = concat(x, t_i)                   # extend prefix
# Now draft_tokens = [t_1, t_2, ..., t_γ]

Phase 2 — Verify（验证）

将原始 prefix + 所有 draft token 拼接成长度为 n+γ 的序列，一次性送入 target model 做前向，得到每个位置的概率分布 p_1, p_2, …, p_γ, p_{γ+1}。这是关键：target model 可以利用 attention 并行处理所有位置，而不是顺序执行 γ 次。

# Verify phase (simplified pseudocode)
full_sequence = concat(prefix, draft_tokens)
# One forward pass of target model across all γ+1 positions
target_logits = target_model.forward(full_sequence)  # shape: [γ+1, vocab]
p = [softmax(target_logits[i]) for i in range(gamma + 1)]
q = draft_logits  # probabilities used during draft phase

接受后的序列更新

设前 k 个 draft token 全部接受（k ≤ γ），第 k+1 个被拒绝。则：

  • 接受 t_1 … t_k（直接加入输出序列）
  • 从 target model 在位置 k+1 的修正分布中再采样一个 token t'
  • 输出序列增加 k+1 个 token（而不是 1 个）
  • 如果全部 γ 个都接受，还可以从 p_{γ+1} 多采一个，共产出 γ+1 个
关键：每轮至少产出 1 个 token 即使所有 draft token 全被拒绝（最坏情况），target model 仍会从 p_1 采样一个有效 token。所以 speculative decoding 永远不会比标准解码产出更少。

3. 接受采样算法与等价性证明

这是 speculative decoding 最精妙的部分。朴素想法是"如果 draft 模型预测的 token 和 target 模型预测一致就接受"——但这会改变输出分布。Leviathan 等人给出了正确的接受准则，保证输出分布完全等价于 target 模型的原始分布。

接受准则（Modified Rejection Sampling）

对于第 i 个位置的 draft token t_i：

Accept t_i with probability min(1, p_i(t_i) / q_i(t_i) ) 其中 p_i = target model 在位置 i 的分布，q_i = draft model 在位置 i 的分布

如果 t_i 被拒绝，则从如下修正分布中重新采样：

p'_i(x) = normalize( max(0, p_i(x) − q_i(x)) ) 修正分布：target 分布中去掉 draft 已覆盖的部分，再归一化

等价性直觉

考虑对某个 token x，经过接受/拒绝采样后它被输出的概率是多少？

  • Draft 采样到 x 且被接受：概率 = q(x) × min(1, p(x)/q(x)) = min(q(x), p(x))
  • Draft 采样到某个被拒绝的 y，然后从 p' 采样到 x：概率 = [Σ_y q(y)·max(0, 1−p(y)/q(y))] × p'(x)

对两种情况求和，结果恰好等于 p(x)。这就是为什么这个算法能保证输出与 target 模型完全等价——不是近似，而是精确的分布等价。

无损性（Lossless Property） Speculative decoding 是一种精确加速方法：对于任何输入，其生成分布与 target model 单步自回归生成的分布在统计上完全相同。这与知识蒸馏、量化等有损方法本质不同。

Temperature / Top-p 的处理

如果 target 模型使用了 temperature t 或 top-p 采样，则接受准则中的 p_i 应替换为经过对应变换后的分布。实现时需要确保 draft 阶段和验证阶段使用相同的采样参数，否则等价性被破坏。这是实现上容易出 bug 的地方。

4. 理论加速分析

平均每轮产出的 token 数

设每个 draft token 被接受的概率为 α（假设各位置独立同分布，这是一个近似），则：

E[accepted tokens per round] = (1 − α^(γ+1)) / (1 − α) 这包括了全接受时的 bonus token；当 α→1 时趋于 γ+1，当 α→0 时趋于 1

加速比

设 target model 单次前向时间为 T_big，draft model γ 次前向总时间为 T_draft，则理论加速比为：

Speedup ≈ E[accepted tokens] / (1 + T_draft/T_big) 当 T_draft ≪ T_big（draft 模型极快）且 α 较高时，加速比趋近于 E[accepted tokens]

关键参数 α 的含义

α 不是固定值，它随输入内容动态变化：

场景 典型 α 说明
代码补全（函数体） 0.80–0.95 小模型擅长局部语法和重复模式，命中率高
翻译、摘要 0.70–0.85 对话指令下结构较规律，命中率中等偏高
创意写作 0.40–0.65 高 temperature、多样性要求高，命中率低
数学推理（多步） 0.60–0.80 推理链结构规律但关键 token 难以预测
Greedy decoding (T=0) ~0.90+ 确定性输出，draft 更容易命中
注意：α 对实际加速比的敏感性 当 γ=4 时：α=0.9 → 平均产出 3.8 tokens；α=0.6 → 平均产出 2.1 tokens；α=0.3 → 平均产出 1.3 tokens。加速比非线性下降，低 α 时 speculative decoding 几乎无收益。

5. 什么场景有效 vs 无效

有效的场景

场景 原因 预期加速
实时对话（batch=1–4） Memory-bandwidth bound，每次前向浪费大量算力；draft 验证的"并行化"收益最大 2–3x latency reduction
代码补全（GitHub Copilot 类） α 高（语法重复性强），小 draft 模型（如 7B→70B）命中率极好 2–4x
Greedy / low-temperature 生成 输出确定性强，draft 命中率高 2.5–3.5x
Prefix 固定的批量任务（小 batch） KV cache 热，draft 开销低 1.5–2.5x
长序列生成（报告、文章） 大部分 token 是"可预测的填充"，draft 命中率高 1.5–2x

无效或有害的场景

场景 原因 建议
高并发批量推理（batch > 32） 大 batch 下已经是 compute-bound，额外 draft 模型占用资源反而降低整体吞吐 关闭 speculative decoding，用 continuous batching 替代
高 temperature / top-p 创意生成 α 低（~0.4），每轮平均产出接近 1，加速比接近 0 甚至为负 考虑 n-gram draft 或直接 baseline
极短序列（<20 tokens） warm-up 开销（KV cache setup）占比高，speculative 收益不显著 不启用
内存资源紧张 Draft 模型额外占用 GPU 内存，可能导致 target 模型 KV cache 被压缩 使用 n-gram draft（无额外模型参数）
Draft 和 target 词表不同 需要对齐词表，实现复杂且可能引入误差 确保同 tokenizer；或用 EAGLE/Medusa 同架构 draft
最大误区 很多人以为 speculative decoding 对所有推理场景都有效。事实上，它的核心假设是"带宽瓶颈 + 低并发 + draft 命中率高"。高并发服务场景中关闭 speculative decoding 往往是更好的选择。

External Draft Model（经典方案）

使用独立的小语言模型作为 draft。例如用 Llama-3-8B draft Llama-3-70B，或用 CodeGemma-2B draft Gemma-7B。优点：实现简单，draft 模型可替换；缺点：占用额外 GPU 内存，词表必须匹配。

Medusa（2024）

Cai et al. 在 target 模型顶层加 K 个额外的预测头（Medusa heads），每个头负责预测未来第 i 个 token（i=1..K），所有头共享 target 模型的 backbone。推理时用 tree attention 并行验证多个候选路径。

  • 优点：不需要单独的 draft 模型，内存开销小（heads 只有几百 MB）；tree attention 可以同时验证多个 token 路径，增大有效 γ
  • 缺点：需要微调 Medusa heads（数据依赖）；无法适配任意新模型；heads 间无自回归依赖，预测能力有限
# Medusa heads: K heads on top of target LM backbone
# During inference (simplified):
hidden = target_model.get_last_hidden(input_ids)
# head[0] predicts token at pos n+1, head[1] predicts n+2, etc.
medusa_logits = [head(hidden) for head in medusa_heads]  # K × vocab
# Build candidate tree, run tree attention verification
candidates = build_tree(medusa_logits, top_k=5)
accepted = target_model.tree_verify(candidates)  # parallel tree attn

EAGLE / EAGLE-2（2024）

EAGLE（Extrapolation Algorithm for Greater Language-model Efficiency）是目前效果最好的 speculative decoding 方案之一。核心改进：draft head 是一个轻量的自回归 transformer layer，输入是 target 模型的 feature（hidden states），因此 draft 预测能充分利用大模型的语义表示，α 显著高于 Medusa。

  • EAGLE-2 进一步引入 adaptive draft tree：根据每个候选 token 的置信度动态展开树，避免浪费验证次数
  • 实测加速比：Vicuna-7B 上约 3x，Vicuna-33B 上约 3.5x（greedy decoding）

n-gram / Prompt Lookup Decoding

最简单的 draft 策略：在已有输入（prompt）中查找与当前 context 末尾匹配的 n-gram，将其后续内容作为 draft token。不需要任何模型，几乎零开销。

  • 适合 prompt 与生成内容高度重叠的场景：如 RAG（生成引用原文）、文档摘要、代码注释生成
  • vLLM 通过 --speculative-model=[ngram] 启用；HuggingFace 通过 PromptLookupDecodingLogitsProcessor 实现

LayerSkip（Meta 2024）

Self-speculative decoding 的代表：用目标模型的前 L' 层（Early Exit）生成 draft，再用完整 L 层验证。不需要额外模型，只需要 target 模型在训练时加入 early exit 正则化（Layer Dropout + 共享 LM Head）。对内存不足的场景特别友好。

vLLM 实现深度解析

vLLM 的 speculative decoding 实现位于 vllm/spec_decode/ 目录，核心文件如下：

文件 职责
spec_decode_worker.py SpecDecodeWorker：协调 draft worker 和 target worker，管理 proposal 和 verification 循环
batch_expansion.py BatchExpansionTop1Scorer：将 draft 生成的 proposals 展开成 target 模型的批次输入
metrics.py 记录 acceptance rate、系统吞吐等统计，用于监控和调优
top1_proposer.py 用单个 draft 模型生成 top-1 proposals（标准 speculative decoding）
ngram_worker.py 基于 n-gram lookup 的无模型 draft 方案
medusa_worker.py Medusa heads draft，需要加载配套 head 权重
multi_step_worker.py 让 draft worker 执行多步前向，支持 pipeline parallelism 场景

vLLM 的关键设计决策：

# vllm/spec_decode/spec_decode_worker.py (概念简化版)
class SpecDecodeWorker:
    def execute_model(self, execute_model_req):
        # Step 1: Draft phase
        proposals = self.proposer.get_spec_proposals(
            execute_model_req, k=self.num_speculative_tokens)
        # proposals: [batch_size, k] draft token ids + probs

        # Step 2: Expand batch for target verification
        # Each sequence becomes k+1 positions for target to verify in parallel
        target_input = self.scorer.expand_batch(execute_model_req, proposals)

        # Step 3: Target model forward (one pass, processes k+1 positions per seq)
        target_logits = self.scorer.score_proposals(target_input, proposals)

        # Step 4: Rejection sampling — produce accepted tokens
        accepted_tokens, bonus_tokens = self.rejection_sampler.forward(
            target_probs=target_logits.probs,
            bonus_token_ids=target_logits.bonus_tokens,
            draft_probs=proposals.probs,
            draft_token_ids=proposals.token_ids,
        )
        return accepted_tokens

TensorRT-LLM 实现要点

TRT-LLM 将 speculative decoding 分为两种引擎配置：

  • Draft-model mode：分别编译 draft engine 和 target engine，通过 runtime 层协调；draft engine 共享 target 的 KV cache pool 分配器，避免内存碎片
  • Medusa mode：将 Medusa heads 融合进 target engine，通过 custom CUDA kernel 实现 tree attention（使用 causal mask 的变体，允许树形 attention 模式）

关键 CUDA 优化：TRT-LLM 为 tree attention 实现了专用的 flash attention 变体，mask 不再是下三角矩阵而是树形 mask，需要修改 attention kernel 的 mask 生成逻辑（参见 cpp/tensorrt_llm/kernels/speculativeDecoding/）。

HuggingFace Assisted Generation

HuggingFace 的实现最为简洁，核心在 generate_utils.py 中的 _assisted_decoding 函数。使用示例：

from transformers import AutoModelForCausalLM, AutoTokenizer

# Load target and draft models
target = AutoModelForCausalLM.from_pretrained("meta-llama/Llama-3-70B")
draft  = AutoModelForCausalLM.from_pretrained("meta-llama/Llama-3-8B")

# Speculative decoding via assisted_model parameter
outputs = target.generate(
    input_ids,
    assistant_model=draft,
    do_sample=True,
    temperature=0.8,
    max_new_tokens=200,
    # HF automatically handles rejection sampling + correction
)

# n-gram / Prompt Lookup (no draft model needed)
from transformers import PromptLookupDecodingLogitsProcessor
proc = PromptLookupDecodingLogitsProcessor(num_speculative_tokens=5)
outputs = target.generate(input_ids, logits_processor=[proc])
HuggingFace 的 heuristic 调整 HuggingFace 实现了动态调整 γ（num_speculative_tokens）的逻辑：如果当前接受率较低，减少 γ 避免浪费；如果接受率高，增加 γ 以获得更多收益。这是实际部署中减少"bad case"影响的有效策略。

8. 工程权衡与实践建议

内存开销

External draft model 需要额外加载到 GPU 内存。以 Llama-3 系列为例：

组合 Draft 内存 总内存需求 推荐硬件
Llama-3-8B → 70B ~16GB (BF16) ~156GB 2× H100 80GB
Medusa heads → 70B ~200MB ~140GB 2× H100 80GB
n-gram → 70B 0 ~140GB 2× H100 80GB
Llama-3-1B → 8B ~2GB (BF16) ~18GB 1× A10G 24GB

KV Cache 管理复杂性

Speculative decoding 引入了 KV cache 管理的新挑战：

  • Draft model 生成的 token 序列是"临时性的"——如果被拒绝，对应的 KV cache entry 需要回滚
  • vLLM 通过 block table 的"provisional blocks"机制处理这个问题：draft token 对应的 block 先标记为 provisional，验证通过后才转为 committed
  • tree attention（Medusa/EAGLE）更复杂：多条候选路径共享前缀的 KV，需要 tree 结构的 KV cache，实现难度远高于线性序列

Chunked Prefill 与 Speculative Decoding 的冲突

vLLM 的 chunked prefill 和 speculative decoding 在 v0.5 之前无法同时启用——chunked prefill 会将 prefill 和 decode 混在同一 batch，破坏了 speculative decoding 假设的"纯 decode 阶段"。vLLM v0.6+ 通过异步 speculative decoding 机制解决了这个问题。

监控哪些指标

启用 speculative decoding 后，应持续监控：

  • acceptance_rate（α）：低于 0.5 说明 draft 模型不匹配，应考虑换 draft 或关闭功能
  • mean_accepted_tokens：应接近 γ×α，如果与理论值偏差大，检查实现是否有 bug
  • actual speedup vs baseline：用 end-to-end latency 衡量，警惕"理论加速 2x 但实际只有 1.2x"的情况（往往是 draft 模型开销被低估）
  • GPU memory usage：Draft 模型是否压缩了 KV cache 空间，导致更多 cache eviction

9. Takeaways

什么是 Speculative Decoding 真正聪明的地方

它的精妙不在于"用小模型猜"，而在于修正拒绝采样（Modified Rejection Sampling）保证了数学等价性。这让它成为一种可以安全部署的无损加速方案——不像量化或蒸馏，你不需要重新评估质量影响。

它解决的本质问题

Autoregressive decoding 的根本瓶颈是"每步只产出 1 token 但要读取全部参数"。Speculative decoding 用并行验证把这个比值提高到"每次大模型前向产出 1~γ+1 个 token"，本质是提高了模型参数的计算复用率。

它的局限性

  • 需要一个合适的 draft 模型（或 Medusa/EAGLE heads），选型本身是额外工程工作
  • 加速比对 α 极度敏感，生产环境需要持续监控
  • 高并发场景下完全失效，会降低整体吞吐
  • Speculative decoding + continuous batching 的组合实现复杂，各框架成熟度不一
  • 多 GPU tensor parallel 场景下，draft 模型的通信开销会进一步蚕食收益

业界应用现状（截至 2025-2026）

  • Google 在 Gemini serving 中使用自家的 speculative decoding 方案，据报告 latency 显著降低
  • Meta 发布了 LayerSkip（ICLR 2024），提供了自推测解码的开源实现
  • NVIDIA 在 TRT-LLM 中将 speculative decoding 作为一等公民特性支持，并在 Hopper GPU 上针对 tree attention 做了专项优化
  • vLLM 社区持续改进，已支持 EAGLE-2，且与 prefix caching 的集成正在进行中
  • 实际生产中最常见的部署是：小 batch 实时对话服务 + external draft model（n-gram 作为 fallback）

未来方向

  • Draft 质量提升：EAGLE-2 的 adaptive tree 方向是正确的；基于强化学习训练 draft 模型（奖励为 acceptance rate）是有前景的方向
  • 与其他加速技术融合：speculative decoding + prefix caching + continuous batching 的三合一实现是业界正在攻克的硬工程问题
  • 多模态扩展：VLM 场景中 image token 通常是重复的，n-gram draft 天然适合；但 visual draft model 的设计尚未成熟
