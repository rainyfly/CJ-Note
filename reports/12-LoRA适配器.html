1. 架构总览

SGLang 的 LoRA 子系统实现了论文 S-LoRA: Serving Thousands of Concurrent LoRA Adapters 和 Punica: Multi-Tenant LoRA Serving 的核心思想，能够在单个基座模型之上同时服务数千个不同的 LoRA 适配器。其核心设计包括：

  • 统一的 LoRA 内存池 (LoRAMemoryPool) -- 预分配 GPU 显存，按槽位管理多个适配器权重
  • Segment GEMM 计算核心 -- 通过 Triton 内核实现异构 rank 的批量矩阵乘法
  • LRU/FIFO 驱逐策略 -- 当活跃适配器数超过槽位时自动替换
  • Tensor Parallel 切分 -- LoRA 权重按 TP 维度正确分片

2. LoRA 配置与适配器

2.1 LoRAConfig -- 配置解析

LoRAConfig 负责从 HuggingFace PEFT 格式的 adapter_config.json 中加载适配器元信息：

属性来源说明
radapter_config.jsonLoRA 秩 (rank)
lora_alphaadapter_config.json缩放因子 alpha
target_modulesadapter_config.json应用 LoRA 的模块列表
lora_added_tokens_sizeadded_tokens.json额外词表大小

配置支持两种加载方式：从本地/远程路径加载 (snapshot_download) 或从字典直接构造 (from_dict)。

2.2 LoRAAdapter -- 权重管理

LoRAAdapter 是单个适配器的权重容器。每个适配器包含：

  • 按层组织的 LoRALayer 列表（与基座模型层数一致）
  • embedding_layers -- embed_tokens 和 lm_head 的 LoRA 权重
  • added_tokens_embeddings -- 额外词汇的嵌入权重
权重归一化关键逻辑：SGLang 将独立的 q_proj / k_proj / v_proj 权重合并为统一的 qkv_proj，将 gate_proj / up_proj 合并为 gate_up_proj。若适配器只有 q_proj 和 v_proj（无 k_proj），系统自动补零。对于已经 stacked 的权重（如 qkv_proj），LoRA_A 重复 3 倍以匹配维度。 
# 归一化: q/k/v -> qkv
weights[qkv_name] = torch.cat((weights[q_name], k_proj_weight, weights[v_name]), 0)

# 归一化: gate/up -> gate_up
weights[gate_up_name] = torch.cat((weights[weight_name], weights[up_name]), 0)

3.2 动态加载与卸载

load_lora_adapter() 的完整流程：

  1. 解析配置 (LoRAConfig) 并校验兼容性（rank、target_modules）
  2. 创建 LoRAAdapter 实例并加载权重到 CPU
  3. 若启用 overlap loading，将权重 pin 到 CPU 固定内存以加速 H2D 传输
  4. 校验 pinned adapter 数量不超过 max_loras_per_batch - 1（防止饥饿）

4. 层级封装与 TP 切分

SGLang 通过装饰器模式将基座模型的线性层替换为带 LoRA 的版本。get_lora_layer() 是工厂函数，根据基座层类型返回对应的 LoRA 包装层：

基座层类型LoRA 包装层应用场景
QKVParallelLinearQKVParallelLinearWithLoRA注意力 QKV 投影
MergedColumnParallelLinearMergedColumnParallelLinearWithLoRAMLP gate_up 投影
ColumnParallelLinearColumnParallelLinearWithLoRA通用列并行线性层
RowParallelLinearRowParallelLinearWithLoRAo_proj / down_proj
VocabParallelEmbeddingVocabParallelEmbeddingWithLoRA嵌入层
ParallelLMHeadParallelLMHeadWithLoRA语言模型头

4.1 前向传播公式

对于线性层：output = base_output + scaling * (x @ A^T) @ B^T

对于嵌入层：output = base_embedding(x) + scaling * lora_B @ lora_A[x]

对于 LM Head：output = hidden @ W^T + scaling * (hidden @ A^T) @ B^T

5.2 槽位管理

数据结构说明
uid_to_buffer_id适配器 UID -> 缓冲槽位 ID
buffer_id_to_uid缓冲槽位 ID -> 适配器 UID（EmptySlot 表示空闲）

prepare_lora_batch() 负责将当前批次需要的适配器加载到 GPU：

  1. 标记所有当前批次的 UID 为"已使用"（用于 LRU 追踪）
  2. 遍历 UID 列表，未在池中的分配空闲槽位
  3. 若无空闲槽位，触发驱逐
  4. 从 CPU 缓存复制权重到 GPU 缓冲区（copy_(weight, non_blocking=True)）

5.3 驱逐策略

驱逐策略通过抽象基类 EvictionPolicy 实现，内置两种策略：

策略实现选择逻辑
LRULRUEvictionPolicy基于 OrderedDict，每次使用移到末尾，驱逐最老的
FIFOFIFOEvictionPolicy基于 OrderedDict，仅记录插入顺序，不更新访问
驱逐优先级：优先驱逐非 pinned 的 LoRA 适配器，仅当批次全为 LoRA 请求且无其他候选时才驱逐基座模型 (uid=None)。Pinned 适配器永远不会被驱逐。

6. Triton 计算后端

SGLang 提供多种 LoRA 计算后端：triton（默认）、torch、chunked。Triton 后端通过自定义 Triton 内核实现高效的 Segment GEMM。

6.3 CUDA Graph 兼容

Triton 后端在初始化阶段预创建 cuda_graph_batch_info，包含固定大小的 seg_indptr、weight_indices 等张量。在 CUDA Graph 捕获期间，仅通过 copy_(non_blocking=True) 更新这些张量的内容，避免重新分配。

7.2 异构 Rank 支持

不同 LoRA 适配器可以有不同的 rank。内存池分配 max_lora_rank 大小的缓冲区，小 rank 的适配器仅使用前 r 行/列。Triton 内核通过 lora_ranks 数组动态确定每个适配器的实际计算范围。

8.1 权重加载路径

  1. 首次加载：HuggingFace checkpoint -> LoRAAdapter.initialize_weights() -> CPU 内存
  2. Pin Memory（可选）：enable_lora_overlap_loading 时 pin 到固定内存
  3. On-Demand：批次执行前，按需从 CPU 复制到 GPU LoRAMemoryPool
  4. 驱逐回收：GPU 槽位满时，LRU/FIFO 选择受害者，新权重覆盖旧槽位
