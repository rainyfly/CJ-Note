1.1 路由机制 (Router)

sglang/srt/layers/moe/router.py

SGLang 实现了两种高效的融合路由内核，通过 FusedMoeRouter 类统一管理：

自适应路由选择策略：当批量较大（bs >= 512）或专家数量较多（> 8 个）时，使用 TensorCore 内核进行矩阵乘法加速；否则使用 CudaCore 内核进行逐元素计算。

CudaCore 路由内核

使用 Triton JIT 编译，每个 program 处理一个 token。核心步骤：

  1. 路由矩阵乘法：计算 logits = hidden_states @ router_weight.T
  2. Softcapping：tanh(logits/cap) * cap，防止 logits 过大
  3. Correction Bias：添加纠正偏置（DeepSeek V3 使用 sigmoid + bias）
  4. TopK 选取：迭代选择 top-k 专家并计算 softmax 权重

TensorCore 路由内核

使用分块矩阵乘法（BLOCK_SIZE_M x BLOCK_SIZE_N x BLOCK_SIZE_K），利用 TensorCore 的 tl.dot 指令加速路由计算。支持批量处理多个 token，适合高吞吐场景。

# 路由内核自动选择逻辑
if (bs >= 512 or num_experts > 8) and hidden_dim % BLOCK_SIZE_K == 0:
    # TensorCore 路径 - 大批量/多专家
    return fused_moe_router_tensorcore(...)
else:
    # CudaCore 路径 - 小批量/少专家
    return fused_moe_router_cudacore(...)

分组 TopK 算法 (DeepSeek V2/V3)

DeepSeek 系列使用分层选择策略：

  1. 将专家分为 num_expert_group 组
  2. 对每组取最大分数，选出 topk_group 个最佳组
  3. 在选中的组内选择 top-k 个专家
  4. 可选地添加 correction_bias 进行偏置校正

SGLang 提供多种加速实现：

  • FlashInfer fused_topk_deepseek：针对 DeepSeek 优化的融合内核
  • sgl_kernel moe_fused_gate：SGLang 自研的融合门控内核
  • kimi_k2_moe_fused_gate：针对 384 专家的 Kimi K2 特化内核
  • AIter biased_grouped_topk：AMD ROCm 平台优化

2.2 实现细节

Token 排序与对齐

moe_align_block_size 函数将 token 按专家分组并对齐到 BLOCK_SIZE_M 的倍数，返回：

  • sorted_token_ids：排序后的 token 索引
  • expert_ids：每个 block 对应的专家 ID
  • num_tokens_post_padded：填充后的总 token 数

分块计算策略

# 大批量分块处理，避免显存溢出
CHUNK_SIZE = 64 * 1024  # 每块最多 64K token
for chunk in range((num_tokens // CHUNK_SIZE) + 1):
    # 1. GEMM1: 上投影 (hidden_size -> intermediate_size * 2)
    invoke_fused_moe_kernel(hidden, w1, intermediate_cache1, ...)

    # 2. 激活函数 (SiLU-and-Mul 或 GeLU-and-Mul)
    silu_and_mul(intermediate_cache1, intermediate_cache2)

    # 3. GEMM2: 下投影 (intermediate_size -> hidden_size)
    invoke_fused_moe_kernel(intermediate_cache2, w2, intermediate_cache3, ...)

    # 4. 加权求和
    moe_sum_reduce(intermediate_cache3, out_hidden_states, scaling_factor)

Inplace vs Outplace 模式

模式内存使用适用场景
inplace_fused_experts原地修改输入标准推理，节省显存
outplace_fused_experts分配新输出EP 模式下的 no_combine

TMA 优化 (Tensor Memory Accelerator)

对于 Blackwell+ 架构，下投影 GEMM 支持 TMA 描述符加速数据传输。down_moe_use_tma 标志控制是否启用 TMA，可显著减少数据搬运延迟。

3.2 架构设计

MoeRunner 采用策略模式，核心组件：

  • RunnerCore：实际的计算核心（TritonRunnerCore / DeepGemmRunnerCore / TritonKernelsRunnerCore）
  • FusedOpPool：融合算子注册池，允许将 A2A 通信与计算融合
  • PermuteMethodPool：pre_permute / post_permute 方法池，处理不同 dispatch 格式和 runner 格式之间的数据转换
class MoeRunner:
    def run(self, dispatch_output, quant_info) -> CombineInput:
        if self.fused_func is not None:
            # 融合路径：A2A + 计算一体化
            return self.fused_func(dispatch_output, quant_info, self.config)

        # 分步路径：排列 -> 计算 -> 反排列
        runner_input = self.pre_permute_func(dispatch_output, quant_info, ...)
        runner_output = self.runner_core.run(runner_input, quant_info, ...)
        return self.post_permute_func(runner_output, quant_info, ...)

4.2 Token 分发策略

在 Expert Parallelism 模式下，每个 GPU 只持有部分专家权重。Token 需要通过 All-to-All 通信发送到持有对应专家的 GPU：

# Normal 模式分发核心
def _dispatch_core(self, x, topk_ids, topk_weights, previous_event):
    buffer = self._get_buffer()
    # 1. 计算分发布局
    layout = buffer.get_dispatch_layout(topk_ids, self.num_experts, ...)
    # 2. 执行 All-to-All 分发
    recv_x, recv_topk_ids, recv_topk_weights, num_recv, handle, event = \
        buffer.dispatch(x, topk_idx=topk_ids, topk_weights=topk_weights, ...)
    return recv_x, recv_topk_ids, recv_topk_weights, num_recv, event

# Low-Latency 模式
def _dispatch_core(self, hidden_states, topk_ids):
    buffer = self._get_buffer()
    packed_recv, packed_count, handle, event, hook = \
        buffer.low_latency_dispatch(hidden_states, topk_ids, ...)

6.1 EPLB Manager

sglang/srt/eplb/eplb_manager.py

EPLBManager 使用 Python 生成器实现协程式的重平衡调度：

class EPLBManager:
    def _entrypoint(self):
        while True:
            # 等待 N 次前向传播
            for _ in range(self._rebalance_num_iterations):
                yield
            # 执行重平衡
            yield from self.rebalance()

    def rebalance(self):
        # 1. 获取专家分布统计
        logical_count = recorder.dump_record()["logical_count"]
        # 2. 检查是否需要重平衡（GPU 利用率阈值）
        if not self._check_rebalance_needed(utilization_rate):
            return
        # 3. 计算新的专家位置
        metadata = ExpertLocationMetadata.init_by_eplb(logical_count)
        # 4. 分块更新专家权重
        for chunk in update_layer_ids_chunks:
            yield  # 让出控制权，允许推理继续
            self._model_runner.update_expert_location(metadata, chunk)

7. CUTLASS MoE

  1. 多路由策略：支持 softmax/sigmoid TopK、分组 TopK、带偏置分组 TopK 等多种路由方式，覆盖 DeepSeek、Mixtral、Qwen3、Kimi K2 等主流模型
  2. 融合内核优化：Triton 融合内核将排序、GEMM、激活函数、加权求和全部融合，支持 FP8/INT8/INT4 等多种量化精度
  3. 灵活的执行后端：MoE Runner 框架统一管理 Triton、DeepGemm、CUTLASS、FlashInfer、Marlin 等多种后端
  4. 高效的专家并行：基于 DeepEP 的 Token Dispatcher 支持 Normal/Low-Latency 两种模式，配合两阶段流水线和 TBO 重叠优化
  5. 动态负载均衡：EPLB 系统支持运行时专家重分布，采用 DeepSeek、层级、向量化等多种重平衡算法
  6. 冗余专家支持：通过 physical/logical 专家映射，支持专家冗余部署和就近路由
