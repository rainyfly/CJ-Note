1. 概述

采样系统是 LLM 推理引擎中将模型输出 logits 转化为最终 token 的关键环节。SGLang 的采样子系统设计了一套高度模块化、批量化的架构，支持丰富的采样策略和自定义扩展能力。

采样参数体系

SamplingParams 定义了 temperature、top-k、top-p、min-p 等完整参数集，并支持约束解码、自定义处理器等高级功能。

3. 采样流水线

3.1 LogitsProcessor — Logits 提取

LogitsProcessor 是一个 nn.Module，负责从模型的 hidden states 中提取最终的 logits。它处理张量并行、DP Attention、logit softcapping 等底层细节。

LogitsProcessorOutput 数据结构

一个 @dataclass，承载从 logits 提取到采样后的全部结果：

  • Part 1 (LogitsProcessor 填充)：next_token_logits [#seq, vocab_size]，hidden_states
  • Part 2 (Sampler 填充)：next_token_logprobs，next_token_top_logprobs_val/idx
  • Part 3 (Prefill 填充)：input_token_logprobs，input_top_logprobs_val/idx
  • Part 4 (Diffusion LLM)：full_logits
  • Part 5 (自定义)：customized_info

4. 采样策略详解

4.1 Temperature 温度缩放

温度是最基础的采样参数，通过缩放 logits 控制概率分布的锐度：

logits.div_(sampling_info.temperatures)  # 原地除法，shape: [batch, vocab]
probs = torch.softmax(logits, dim=-1)    # 转为概率分布
  • temperature = 1.0：保持原始分布
  • temperature < 1.0：分布变锐，高概率 token 更突出 — 生成更确定
  • temperature > 1.0：分布变平，低概率 token 获得更多机会 — 生成更多样
  • temperature = 0：等价于 greedy decoding（内部转换为 top_k=1）

4.2 Top-K 采样

Top-K 保留概率最高的 K 个 token，将其余 token 的概率置零：

# PyTorch 实现
probs_sort, probs_idx = probs.sort(dim=-1, descending=True)
probs_sort[arange >= top_ks.view(-1, 1)] = 0.0  # 超出 K 的置零

FlashInfer 后端提供了专用的 top_k_renorm_prob CUDA 内核，在 GPU 上高效执行 Top-K 过滤并重新归一化。

4.3 Top-P (Nucleus Sampling)

Top-P 按照累积概率从高到低排列，保留累积概率刚好超过 p 的最小集合：

probs_sum = torch.cumsum(probs_sort, dim=-1)
probs_sort[(probs_sum - probs_sort) > top_ps.view(-1, 1)] = 0.0

当同时使用 Top-K 和 Top-P 时，FlashInfer 后端通过 top_k_top_p_sampling_from_probs 将两者合并为单次操作（filter_apply_order="joint"），避免多次排序。

4.4 Min-P 采样

Min-P 是一种动态阈值采样方法。它以最高概率 token 的概率值乘以 min_p 系数作为动态阈值，过滤低于该阈值的所有 token：

min_p_thresholds = probs_sort[:, 0] * min_ps  # 最高概率 * min_p
probs_sort[probs_sort < min_p_thresholds.view(-1, 1)] = 0.0

Min-P 的优势在于阈值随分布自适应：当模型对某个 token 非常确信时，阈值自然升高，过滤更多；当模型不确定时，保留更多候选。

5.2 频率惩罚 (Frequency Penalty)

频率惩罚按照 token 在已生成输出中的出现次数进行线性惩罚：

工作原理

数学公式：logits[token] -= frequency_penalty * count(token)

  • 累积阶段：每生成一个 token，通过 scatter_add_ 在频率矩阵对应位置累加惩罚值
  • 应用阶段：将累积的频率惩罚矩阵直接从 logits 中减去
# 累积：每生成 token 时
self.cumulated_frequency_penalties.scatter_add_(
    dim=1, index=output_ids.unsqueeze(1), src=self.frequency_penalties
)
# 应用：采样前
logits.sub_(self.cumulated_frequency_penalties)

5.3 存在惩罚 (Presence Penalty)

存在惩罚关注 token 是否出现过（二元判断），而非出现次数：

工作原理

数学公式：logits[token] -= presence_penalty * (1 if token appeared else 0)

  • 累积阶段：使用 scatter_（非 scatter_add_），多次出现只覆盖不累加
  • 应用阶段：同频率惩罚，直接减去
# 关键区别：scatter_ (覆盖) vs scatter_add_ (累加)
self.cumulated_presence_penalties.scatter_(
    dim=1, index=output_ids.unsqueeze(1), src=self.presence_penalties
)
频率 vs 存在惩罚的本质区别 — 频率惩罚使用 scatter_add_ 累加，token 每出现一次惩罚增大一次；存在惩罚使用 scatter_ 覆盖，token 无论出现多少次惩罚恒定。两者可同时使用，效果叠加。

5.4 最小新 Token 惩罚 (MinNewTokens)

最小新 token 惩罚确保模型至少生成指定数量的 token 后才能触发停止 token：

工作原理

  • 准备阶段：构建停止 token 惩罚矩阵 stop_token_penalties，将所有停止 token（EOS + 用户指定）对应位置设为 -inf
  • 累积阶段：仅计数已生成的 token 数 len_output_tokens += 1
  • 应用阶段：当 len_output_tokens < min_new_tokens 时，将 -inf 加到停止 token 的 logit 上，有效阻止它们被采样
mask = (self.len_output_tokens < self.min_new_tokens).expand_as(logits)
logits[mask] += self.stop_token_penalties[mask]  # 停止 token 位置变为 -inf
Pad 处理技巧 — 由于不同请求的停止 token 数量不同，系统使用 pad_sequence 将它们填充到同一长度。填充值设为 vocab_size（超出词表范围），然后分配 vocab_size + 1 列的张量，最后裁剪掉最后一列，巧妙避免了越界问题。

6.2 序列化与传输机制

自定义处理器需要从客户端传输到服务端，SGLang 使用 dill 序列化：

# 客户端：序列化
@classmethod
def to_str(cls) -> str:
    return json.dumps({"callable": dill.dumps(cls).hex()})

# 服务端：反序列化（带 LRU 缓存）
@lru_cache(maxsize=None)
def _cache_from_str(json_str: str):
    data = orjson.loads(json_str)
    return dill.loads(bytes.fromhex(data["callable"]))
处理器合并优化 — 在 SamplingBatchInfo.from_schedule_batch() 中，相同类型的自定义处理器会被合并。系统为每种处理器创建一个布尔掩码 tensor，标记批次中哪些请求使用该处理器。应用时通过 batch_mask.nonzero() 获取索引，只对相关请求执行处理，避免了逐请求处理的开销。

6.3 思考预算控制 (ThinkingBudgetLogitProcessor)

针对支持"思考"功能的模型（如 Qwen3、DeepSeek-R1、GLM-4），该处理器可以限制思考阶段的 token 数量：

工作流程

  1. 检查当前序列中是否已经进入思考阶段（存在 THINKING_START_TOKEN_ID 且未结束）
  2. 计算思考开始后已生成的 token 数量
  3. 当达到 thinking_budget 时：
    • 如果上一个 token 不是换行符：强制输出换行符（所有其他 token 设为 -inf）
    • 如果上一个 token 是换行符：强制输出思考结束 token

6.4 N-Gram 去重 (DeepseekOCRNoRepeatNGramLogitProcessor)

专为 DeepSeek-OCR 设计的滑动窗口 N-Gram 重复检测处理器：

算法流程

  1. 从请求的 custom_params 获取 ngram_size 和 window_size
  2. 在滑动窗口 [len(seq) - window_size, len(seq)] 内搜索所有 N-Gram
  3. 如果当前的 (n-1)-gram 前缀与窗口中某个 N-Gram 的前缀匹配，则将该 N-Gram 的最后一个 token 加入禁止列表
  4. 支持 whitelist_token_ids 豁免特定 token
  5. 将禁止 token 的 logit 设为 -inf

7. 批量管理机制

SamplingBatchInfo 支持调度器所需的全部批量操作。

7.1 过滤 (filter_batch)

当部分请求完成时，需要从批次中移除它们：

def filter_batch(self, keep_indices, keep_indices_device):
    # 1. 过滤惩罚器状态
    self.penalizer_orchestrator.filter(keep_indices_device)
    # 2. 过滤自定义处理器掩码
    if self.has_custom_logit_processor:
        self._filter_batch_custom_logit_processor(keep_indices, keep_indices_device)
    # 3. 过滤采样参数张量
    for item in ["temperatures", "top_ps", "top_ks", "min_ps", "sampling_seed"]:
        setattr(self, item, value[keep_indices_device])
    # 4. 过滤 logit_bias
    if self.logit_bias is not None:
        self.logit_bias = self.logit_bias[keep_indices_device]

8. 确定性采样

SGLang 支持通过 sampling_seed 参数实现确定性推理，使相同输入始终产生相同输出。

multinomial_with_seed 算法

确定性采样通过 Gumbel-Max 技巧实现，避免了 torch.multinomial 的不确定性：

  1. 基于 seed 和 position 计算确定性的 step_seed：
    step_seed = (seed * 19349663) ^ (positions * 73856093)
  2. 通过哈希函数生成均匀分布的伪随机数：
    hashed = (seed_expanded * 8589934591) ^ (col_indices * 479001599)
  3. 转换为 Gumbel 噪声：gumbel = -log(-log(uniform))
  4. 加到 log 概率上：perturbed = log(probs) + gumbel_noise
  5. 取 argmax 即为采样结果
Gumbel-Max 技巧的优势 — 这种方法将随机采样转化为确定性的 argmax 操作。通过精心设计的哈希函数和大素数乘法，每个 (seed, position, token_index) 三元组产生唯一的伪随机数，保证了跨运行的可复现性。

9.3 设计亮点

批量化一切

所有采样参数和惩罚状态都以批量张量形式存储和操作，最大化 GPU 利用率。标志位（is_all_greedy 等）快速跳过不必要的计算分支。

惩罚器编排模式

Orchestrator 模式解耦了惩罚器的创建、准备和应用。惩罚器惰性初始化，只在实际需要时分配内存。弱引用避免循环引用。

Overlap 优化

通过 copy_for_forward() 预计算惩罚缓冲区，使惩罚计算可以与 GPU 模型前向重叠执行，减少端到端延迟。

多后端灵活性

FlashInfer CUDA 内核用于生产加速，PyTorch 后端用于调试和兼容，Ascend 后端支持国产 NPU。注册工厂函数支持第三方扩展。
