vLLM 整体设计综述

TYPE: REPO ANALYSIS DATE: 2026-04-10 基于 vLLM v1 架构

TL;DR

  • vLLM 的核心创新是 PagedAttention，借鉴 OS 虚拟内存分页管理 KV Cache，彻底消除内存碎片，GPU 利用率可提升 2–4×。
  • v1 架构将引擎拆分为前端（LLMEngine）与后端（EngineCore）两个独立进程，通过 ZMQ + msgpack 通信，前端负责 Tokenize/Detokenize，后端专注调度与执行。
  • 调度器原生支持 Continuous Batching 和 Chunked Prefill，统一以 num_computed_tokens 追踪每个请求状态，不再区分 Prefill/Decode 两阶段。
  • 框架支持 TP / PP / DP / EP / CP 五种并行范式，以及 FP8、AWQ、GPTQ、Marlin 等主流量化方案，覆盖 NVIDIA、AMD、Intel、TPU 多种硬件。
  • 通过 200+ 模型支持（Decoder-Only、MoE、MLA、SSM、VLM）、推测解码（EAGLE/Medusa/N-gram）、P/D 分离、结构化输出等丰富 Feature 矩阵，vLLM 已成为生产级 LLM 服务的事实标准。

1. 项目定位与核心价值

vLLM 是一个面向大语言模型（LLM）推理的高吞吐量、低延迟开源服务框架，由 UC Berkeley 主导开发。 其核心技术贡献是 PagedAttention，通过借鉴操作系统虚拟内存的分页思想来管理 KV Cache， 彻底解决了传统推理框架中 KV Cache 的内存碎片化和静态预分配问题。

核心技术贡献

技术 解决的问题 效果
PagedAttention KV Cache 碎片化、静态预分配浪费 GPU 利用率提升 2-4×
Automatic Prefix Caching (APC) 相同前缀重复计算 System Prompt 计算消除
Chunked Prefill Prefill 阻塞 Decode，延迟尖刺 TTFT/TPOT 均衡
Continuous Batching 静态批处理 GPU 空转 吞吐量最大化
推测解码 (Spec Decode) Decode 阶段内存带宽瓶颈 2-3× 速度提升

6. 请求全链路概览

请求全链路时序（sequenceDiagram）

sequenceDiagram participant User as 用户 participant IP as InputProcessor participant ECC as EngineCoreClient participant EC as EngineCore participant SCHED as Scheduler participant EXEC as Executor participant GMR as GPUModelRunner participant OP as OutputProcessor User->>IP: llm.generate(["Hello World"], SamplingParams) IP->>IP: Tokenize / 多模态特征提取 IP->>ECC: EngineCoreRequest ECC->>EC: ZMQ PUSH (msgpack) EC->>SCHED: step() → schedule() SCHED-->>EC: SchedulerOutput EC->>EXEC: execute_model(SchedulerOutput) EXEC->>GMR: collective_rpc GMR->>GMR: 1. _update_states GMR->>GMR: 2. _prepare_inputs GMR->>GMR: 3. model.forward() GMR->>GMR: 4. sampler() GMR-->>EXEC: ModelRunnerOutput EXEC-->>EC: ModelRunnerOutput EC->>SCHED: update_from_outputs() SCHED-->>EC: EngineCoreOutput EC-->>ECC: ZMQ PULL (msgpack) ECC->>OP: EngineCoreOutput OP->>OP: Detokenize / 停止条件检测 OP-->>User: RequestOutput (outputs[0].outputs[0].text)
Note 每个 EngineCore.step() 调用处理一个调度周期：Scheduler 决定当前 batch 中哪些请求参与执行（包括 Chunked Prefill 切片和 Decode token），KVCacheManager 分配/释放物理块，Executor 驱动所有 GPU Worker 并行执行一次 forward pass。

7. 技术演进：v0 → v1

方面 v0 (旧架构) v1 (现行架构)
请求状态 Prefill/Decode 两阶段 num_computed_tokens 统一视图
引擎结构 单进程 前端/后端 进程分离
IPC 协议 无（同进程） ZMQ + msgpack
Chunked Prefill 有限支持 原生支持，统一调度
APC 基于 Sequence 哈希 基于 Block 哈希，更细粒度
配置系统 EngineArgs VllmConfig（统一容器）
Worker 通信 复杂的 ray/multiproc 混用 collective_rpc 统一抽象
设计哲学 v1 的核心思想是"关注点分离"：前后端进程分离使得 CPU 密集的 Tokenize/Detokenize 不阻塞 GPU 调度循环；num_computed_tokens 的统一状态视图消除了 Prefill/Decode 分支逻辑，让 Chunked Prefill 成为一等公民；Block 级哈希 APC 比 Sequence 级哈希具有更细的复用粒度，尤其对长 System Prompt 场景效果显著。
