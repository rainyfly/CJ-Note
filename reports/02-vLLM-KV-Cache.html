vLLM KV Cache 管理：PagedAttention 与 APC 深度解析

TYPE: REPO ANALYSIS DATE: 2026-04-10 基于 vLLM v1 架构

TL;DR

  • vLLM 以 Block（默认 16 tokens）为单位动态分配 KV Cache，彻底消除传统静态预分配的显存浪费，理论内存利用率接近 100%。
  • PagedAttention 类比操作系统虚拟内存：逻辑 KV 槽位通过 block_table 映射到非连续物理块，CUDA 内核通过 slot_mapping 完成间接寻址。
  • Automatic Prefix Caching (APC) 使用链式哈希索引已满块，前缀相同的请求（如共享 System Prompt 的 RAG 场景）可复用 KV Cache，TTFT 最多降低 90%+。
  • BlockPool 维护 LRU 双向链表：普通空闲块插队首（优先驱逐），APC 缓存块插队尾（LRU 延迟驱逐），确保热点 prefix 尽量留在 GPU。
  • 当 GPU 块不足时，支持 KV Cache Offloading 到 CPU Pinned Memory，以及 P/D 分离场景下通过 KVConnector（NCCL/RDMA）跨节点传输 KV。

1. KV Cache 管理概述

KV Cache 管理是 vLLM 最核心的创新。传统框架为每个请求预先分配最大序列长度的连续显存，导致严重浪费：

传统方案（静态预分配）:
  请求 A: 实际 200 tokens，预分配 2048 个 KV 槽位 → 90% 浪费
  请求 B: 同样浪费
  结果: GPU 内存利用率极低，并发数受限

vLLM PagedAttention:
  以 Block（默认 16 tokens）为单位动态分配
  按需分配，按需释放，内存利用率接近 100%

2.3 Block Table 与 Slot Mapping

# block_table[req_idx, block_slot] = physical_block_id
# 形状: [num_reqs, max_blocks_per_req]
block_table = tensor([
    [7, 23, 5, -1, -1, ...],  # 请求 A

    # 请求 B
    7*16+0, ...,                     # token 0-15 → 块7（共享）
    ...
])

3. 核心数据结构

3.1 KVCacheBlock

class KVCacheBlock:
    """代表一个物理 KV Cache 块"""
    block_id: int              # 物理块编号（GPU 显存索引）
    ref_count: int             # 引用计数（共享时 > 1）
    block_hash: BlockHash | None  # 内容哈希（仅当块已满时）
    num_hashed_tokens: int     # 参与哈希的 token 数

    # 在 free_block_queue（LRU deque）中的节点引用
    # ref_count=0 时加入此队列
    _prev_free_block: KVCacheBlock | None
    _next_free_block: KVCacheBlock | None

3.2 BlockHash

class BlockHash(NamedTuple):
    """用于 APC 的块哈希"""
    hash_value: int        # 数字哈希值（xxHash/SHA256）
    token_ids: tuple[int]  # 块内 token ID（用于哈希碰撞验证）

    # 哈希输入：
    # - 父块的 hash_value（链式哈希）
    # - 本块的 token_ids
    # - 多模态内容哈希（如有图像/音频）
    # - extra_hash（如 LoRA adapter id）

3.3 BlockPool

class BlockPool:
    """管理所有物理块的池"""

    # 物理块数组（固定大小）
    blocks: list[KVCacheBlock]     # index = block_id

    # 空闲块 LRU 队列
    # 队首 = 最久未用（优先驱逐）
    # 队尾 = 最近释放（优先复用）
    free_block_queue: BlockFreeQueue

    # APC 索引：hash → KVCacheBlock
    # 注意：cached_blocks 中的块 ref_count 可能 = 0（可被驱逐但尚未驱逐）
    cached_blocks: dict[BlockHash, KVCacheBlock]

    # 统计
    num_gpu_blocks: int
    num_free_blocks: int           # = len(free_block_queue) + 引用为0的缓存块

4. KVCacheManager

文件：vllm/v1/core/kv_cache_manager.py

4.1 核心接口

class KVCacheManager:

    def get_computed_blocks(
        self, request: Request
    ) -> tuple[list[KVCacheBlock], int]:
        """
        查找请求 prompt 的 APC 命中块。
        返回: (命中的 KVCacheBlock 列表, 已计算的 token 数)
        """

    def allocate_slots(
        self,
        request: Request,
        num_tokens: int,
        computed_blocks: list[KVCacheBlock] | None = None,
    ) -> KVCacheBlocks | None:
        """
        为请求分配 KV 块。
        返回: KVCacheBlocks（含 block_id 列表），None 表示分配失败
        """

    def free(self, request: Request) -> None:
        """释放请求的所有 KV 块"""

    def free_block(self, block: KVCacheBlock) -> None:
        """释放单个块（引用计数 -1，归零则加入 free_block_queue）"""

4.3 free() 释放流程

free(request)
       │
       ├── 获取 request 的所有 KV 块
       │
       for block in request.blocks:
       │
       ├── block.ref_count -= 1
       │
       └── if block.ref_count == 0:
               # 块不再被任何请求使用
               if block.block_hash is not None:
                   # APC 块：加入 LRU 尾部（不立即删除 cached_blocks 中的条目）
                   # 等到其他请求需要块时才真正驱逐
                   free_block_queue.append(block)
               else:
                   # 普通块（最后一块，通常未满）：
                   # 直接加入 LRU 头部（优先被复用/驱逐）
                   free_block_queue.appendleft(block)

5. BlockPool（物理块池）

文件：vllm/v1/core/block_pool.py

5.1 LRU 驱逐队列

BlockPool 使用双向链表（deque）实现 LRU 队列：队首放最久未用的块（优先驱逐），队尾放最近释放的块（优先复用）。

free_block_queue（队首 = 最久未用，队尾 = 最近释放）:

  ┌──────┐  ┌──────┐  ┌──────┐  ┌──────┐  ┌──────┐
  │Block │←→│Block │←→│Block │←→│Block │←→│Block │
  │  #3  │  │  #17 │  │  #9  │  │  #42 │  │  #8  │
  │hash=A│  │hash=B│  │hash=C│  │hash=D│  │hash=E│
  └──────┘  └──────┘  └──────┘  └──────┘  └──────┘
  队首（oldest）                           队尾（newest）
  ↑ 分配新块时从这里取，如有 cached_hash 则驱逐 APC 条目
                                          ↑ 释放块时加到这里

5.2 块的生命周期

KVCacheBlock 状态机

6. Automatic Prefix Caching (APC)

6.1 工作原理

APC 使得前缀相同的请求可以共享 KV Cache。以 RAG 场景为例（多请求共享相同 System Prompt）：

T1: 请求 A 处理
  Prompt: "System: You are helpful.\nUser: Hello, what is AI?"
  Block 0 (tokens 0-15): "System: You are helpful.\n"
    → 块满，计算哈希: hash_0 = H(seed, token_ids[0:16])
    → cached_blocks[hash_0] = block #7

  Block 1 (tokens 16-31): "User: Hello, what"
    → 块满，计算哈希: hash_1 = H(hash_0, token_ids[16:32])
    → cached_blocks[hash_1] = block #23

T2: 请求 B 处理（相同系统提示）
  Prompt: "System: You are helpful.\nUser: What is machine learning?"

  get_computed_blocks(request_B):
    检查 hash_0 → 命中! block #7 (ref_count 0→1, 从LRU移除)
    检查 hash_1 → 未命中（"User: What is" ≠ "User: Hello, what"）

  → num_computed_tokens = 16（前16个 token 不需要重新计算）
  → 只需分配第2块开始的新块

6.2 哈希链（Chain Hash）

链式哈希确保前缀唯一性：每个块的哈希依赖父块哈希，形成不可伪造的前缀依赖关系。

链式哈希结构

8.1 FlashAttention 后端布局

kv_cache Tensor:
  shape = [num_layers, 2, num_blocks, block_size, num_kv_heads, head_size]

访问示例：

  kv_cache[3, 0, 7, 5, 2, 64]

写入新 token（在 execute_model 时通过 slot_mapping）:
  slot_id = block_id × block_size + position_in_block
  # CUDA 内核: cache_ops.reshape_and_cache(key, value, key_cache, value_cache, slot_mapping)

8.2 FlashInfer 后端布局

kv_cache Tensor:
  shape = [num_blocks, 2, block_size, num_kv_heads, head_size]
  (与 FlashAttention 的差异：num_layers 维度在外层分开存储)

8.3 MLA（DeepSeek）布局

MLA（Multi-head Latent Attention）将 KV 压缩为低秩表示，相比标准 MHA 节省约 4× 显存：

布局 Tensor Shape 说明
标准 MHA KV Cache [num_layers, 2, num_blocks, block_size, num_kv_heads, head_size] 基准
MLA c_kv（压缩 KV） [num_layers, num_blocks, block_size, kv_lora_rank] kv_lora_rank=512
MLA k_pe（RoPE key） [num_layers, num_blocks, block_size, qk_rope_head_dim] 额外位置信息
MLA 显存节省 kv_lora_rank (512) << num_kv_heads × head_size (2048)，约 4× KV Cache 压缩比，DeepSeek 系列模型因此能在相同 GPU 上支持更长上下文。

11.2 块驱逐顺序（LRU）

free_block_queue 队首（最先被分配/驱逐）的块：
  1. ref_count=0 的普通块（最后一块，通常未满，无 APC 价值）
  2. 最久未被命中的 APC 块（LRU 顺序）

驱逐时:
  block = free_block_queue.popleft()
  if block.block_hash:
      del cached_blocks[block.block_hash]  # 清除 APC 索引
  # 块被重新分配给新请求
APC 与抢占的交互 被抢占的请求释放块后，若这些块有 APC hash 且尚未被驱逐，则重新调度时仍能命中 APC 缓存，无需完整重新计算 prompt。这是 vLLM RECOMPUTE 策略相比 SWAP 策略的一个有趣优化点。

13.2 APC 命中对调度性能的影响

场景：多请求共享 System Prompt（RAG 场景）

请求 1（无 APC）：System Prompt 1024 tokens + User 100 tokens
Prefill 计算量：1124 tokens（全部），KV 分配：71 块

请求 2（APC 命中 System Prompt）：System Prompt 1024 tokens + User 80 tokens
APC 命中 64 块（1024 tokens），Prefill 只需计算 80 tokens，仅需 5 新块（+64 共享块 ref_count++）

效果：计算量减少 93%（80 vs 1124 tokens）| 共享块被 N 个请求共用 | TTFT 大幅降低
