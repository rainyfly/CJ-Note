vLLM Engine 层与 Scheduler 深度解析

TYPE: REPO ANALYSIS DATE: 2026-04-10 基于 vLLM v1 架构

TL;DR

  • vLLM v1 将前端（Tokenize/Detokenize）与后端（GPU 推理）分离到独立进程，通过 ZMQ + msgpack 通信，绕过 GIL 实现真正的 CPU/GPU 并行。
  • LLMEngine（同步）和 AsyncLLM（异步）共享同一套 EngineCoreClient 抽象，后者用每请求 asyncio.Queue 实现流式并发输出。
  • EngineCore 的 step() 在等待 GPU 结果时同步编译结构化输出的语法 bitmask，将两者执行时间重叠，降低额外延迟。
  • 调度器用单一整数 num_computed_tokens 替代传统 Prefill/Decode 两阶段状态，Chunked Prefill、推测解码、APC 前缀命中均自然统一在这一模型下。
  • Chunked Prefill（默认 chunk_size ≤ 16384 tokens）将长 Prefill 与 Decode 请求混合调度，显著降低 TTFT 抖动并稳定 TPOT。

1. Engine 层概述

vLLM v1 的 Engine 层采用前后端进程分离架构：

Engine 层整体架构（前后端进程分离）

graph TD subgraph Frontend["前端进程 (Frontend)"] LLMEngine["LLMEngine\n(同步批推理)"] AsyncLLM["AsyncLLM\n(异步流式推理)"] OpenAI["OpenAI API Server"] InputProc["InputProcessor\n• Tokenize\n• 多模态预处理"] OutputProc["OutputProcessor\n• Detokenize\n• 停止条件检测"] ECClient["EngineCoreClient\n(ZMQ PUSH → / ← ZMQ PULL)"] LLMEngine --> InputProc AsyncLLM --> InputProc OpenAI --> InputProc InputProc --> ECClient ECClient --> OutputProc end subgraph Backend["后端进程 (EngineCore)"] EC["EngineCore\nScheduler + KVCacheManager\n+ Executor + StructuredOutput"] end ECClient -- "ZMQ ipc:// / tcp://\nmsgpack 序列化" --> EC EC -- "EngineCoreOutputs" --> ECClient

设计动机：

  • CPU 密集的 Tokenize/Detokenize（前端）与 GPU 推理（后端）真正并行，绕过 GIL
  • 进程隔离提升稳定性（Worker Crash 不影响 API Server）
  • ZMQ 通信延迟极低（< 1 μs IPC），msgpack 序列化比 pickle 快 5–10×

2.3 核心方法

add_request()

def add_request(
    self,
    request_id: str,
    prompt: PromptType,
    params: SamplingParams | PoolingParams,
    arrival_time: float | None = None,
) -> None:
    # 1. InputPreprocessor → EngineCoreRequest
    engine_core_request = self.input_preprocessor.process_inputs(
        request_id, prompt, params, arrival_time
    )
    # 2. 发送给 EngineCore
    self.engine_core.add_request(engine_core_request)

step()

def step(self) -> list[RequestOutput]:
    # 1. 从 EngineCore 拉取最新输出
    engine_core_outputs = self.engine_core.get_output()

    # 2. 处理输出（detokenize + 停止检测）
    request_outputs, requests_to_abort = \
        self.output_processor.process_outputs(engine_core_outputs)

    # 3. 中止已完成的请求
    self.engine_core.abort_requests(requests_to_abort)

    return request_outputs

3. AsyncLLM（异步引擎）

文件：vllm/v1/engine/async_llm.py

3.1 职责

AsyncLLM 是在线服务场景的异步前端，被 OpenAI API Server 使用，支持并发请求和流式输出。

3.2 核心设计

class AsyncLLM:
    # 每个请求有独立的 asyncio.Queue
    _request_outputs: dict[str, asyncio.Queue[RequestOutput | Exception]]

    # 后台 asyncio Task，持续从 EngineCore 拉取输出
    _background_loop: asyncio.Task

3.3 核心方法

generate()

async def generate(
    self,
    request_id: str,
    inputs: PromptType,
    params: SamplingParams,
    ...
) -> AsyncGenerator[RequestOutput, None]:
    # 1. 预处理
    engine_core_request = await self.input_preprocessor.process_inputs(...)

    # 2. 创建输出队列
    queue: asyncio.Queue = asyncio.Queue(maxsize=queue_size)
    self._request_outputs[request_id] = queue

    # 3. 发送给 EngineCore
    await self.engine_core.add_request(engine_core_request)

    # 4. 流式返回输出
    try:
        while True:
            output = await queue.get()
            if isinstance(output, Exception):
                raise output
            yield output
            if output.finished:
                break
    finally:
        # 清理
        del self._request_outputs[request_id]

后台循环（_run_output_handler）

async def _run_output_handler(self):
    """后台 asyncio Task，持续处理 EngineCore 输出"""
    async for engine_core_outputs in self.engine_core.get_output_async():
        # 处理 outputs
        request_outputs = self.output_processor.process_outputs(
            engine_core_outputs
        )
        # 分发到各请求的 Queue
        for req_output in request_outputs:
            queue = self._request_outputs.get(req_output.request_id)
            if queue:
                await queue.put(req_output)

4. EngineCore（推理内核）

文件：vllm/v1/engine/core.py

4.1 职责

EngineCore 是真正的推理执行核心，运行在独立进程中（或同进程单步调用）。

组件：

  • Scheduler：决定哪些请求在本步执行
  • KVCacheManager：分配/回收 KV 块
  • Executor：调度 Worker 进程执行模型
  • StructuredOutputManager：结构化输出语法编译

4.2 核心 step() 方法

def step(self) -> list[EngineCoreOutput]:
    """单步推理循环"""

    # 1. 调度：决定本步执行哪些 token
    scheduler_output = self.scheduler.schedule()

    if scheduler_output.total_num_scheduled_tokens == 0:
        return []  # 无可调度请求，空转

    # 2. 异步提交模型执行（non_block=True 立即返回 Future）
    future = self.executor.execute_model(
        scheduler_output, non_block=True
    )

    # 3. 并行获取结构化输出的语法 bitmask（与 GPU 执行重叠）
    grammar_output = self.scheduler.get_grammar_bitmask(scheduler_output)

    # 4. 等待 GPU 执行完成
    model_output: ModelRunnerOutput = future.result()

    # 5. 更新调度器状态（消费采样结果，检查停止条件）
    outputs = self.scheduler.update_from_outputs(
        scheduler_output, model_output, grammar_output
    )

    return outputs
关键优化 execute_model(non_block=True) 立即返回 Future，GPU 在后台执行的同时，CPU 并发编译语法约束 bitmask（可能耗时数毫秒）。future.result() 等待 GPU 完成时，bitmask 通常已就绪，实现零额外等待。

4.3 进程模式

同进程模式（单 GPU）

class InprocClient(EngineCoreClientBase):
    """直接调用 EngineCore，无 IPC 开销"""
    def __init__(self, vllm_config):
        self.engine_core = EngineCore(vllm_config, executor_class=...)

    def add_request(self, request):
        self.engine_core.add_request(request)

    def get_output(self) -> list[EngineCoreOutput]:
        return self.engine_core.step()

独立进程模式（多 GPU 或提高并行性）

class BackgroundEngineCoreProcess:
    """在子进程中运行 EngineCore 的 busy loop"""

    def run_busy_loop(self):
        """子进程入口"""
        self.engine_core = EngineCore(...)

        while True:
            # 处理新请求（来自 ZMQ）
            self._handle_pending_requests()

            # 执行一步
            outputs = self.engine_core.step()

            # 发送输出（ZMQ PUSH）
            if outputs:
                self._send_outputs(outputs)

7. Scheduler（调度器）

文件：vllm/v1/core/sched/scheduler.py

7.1 核心设计哲学

vLLM v1 调度器取消了 Prefill/Decode 两阶段区分，统一用 num_computed_tokens 描述请求状态：

每个请求维护:
  num_computed_tokens  ←  已有 KV Cache 的 token 数
  num_tokens           ←  prompt_len + output_len（含推测 token）

  diff = num_tokens - num_computed_tokens  ←  本步需计算的 token 数

这一设计自然支持：

  • Chunked Prefill：Prefill 可分多步完成（每步计算 diff 的一部分）
  • 推测解码：Spec token 也计入 num_tokens
  • Prefix Caching：命中时直接设 num_computed_tokens = 命中块数 × block_size

7.2 请求生命周期

请求状态机（调度器生命周期）

stateDiagram-v2 [*] --> WAITING : add_request() WAITING --> RUNNING : schedule()\n有空闲内存 RUNNING --> WAITING : 内存不足（抢占） RUNNING --> FINISHED_STOPPED : finish_reason = STOP/EOS RUNNING --> FINISHED_LENGTH : max_tokens 达到 RUNNING --> FINISHED_ABORT : abort() 用户取消 FINISHED_STOPPED --> [*] FINISHED_LENGTH --> [*] FINISHED_ABORT --> [*]

7.3 schedule() 详细流程

schedule() → SchedulerOutput
═══════════════════════════════════════════════════════

初始化 budget:
  remaining_tokens = max_num_batched_tokens   # 本步 token 预算
  remaining_reqs   = max_num_seqs             # 最大并发数

────────────────────────────────────────
阶段一：处理 RUNNING 队列（优先级高）
────────────────────────────────────────
for request in running_queue (优先级从高到低):

  num_new_tokens = request.num_tokens - request.num_computed_tokens

  # Chunked Prefill 截断
  if num_new_tokens > remaining_tokens:
      if can_do_partial_prefill(request):
          num_new_tokens = remaining_tokens
      else:
          preempt(request)  # 大 Prefill 请求暂时抢占
          continue

  # 分配 KV 块
  result = kv_cache_manager.allocate_slots(request, num_new_tokens)

  if result == NO_FREE_BLOCKS:
      # 内存不足，抢占
      preempt(request)   # 释放 KV 块，移回 WAITING
      continue

  # 成功调度
  scheduled_running.append(request)
  remaining_tokens -= num_new_tokens

────────────────────────────────────────
阶段二：处理 WAITING 队列（填充预算）
────────────────────────────────────────
for request in waiting_queue (FCFS 或 priority 排序):

  if remaining_tokens <= 0: break

  # 查找前缀缓存命中
  computed_blocks, num_computed = \
      kv_cache_manager.get_computed_blocks(request)

  # 检查是否能放入剩余预算
  num_new_tokens = min(
      request.num_tokens - num_computed,
      remaining_tokens,
      max_chunk_size
  )

  if num_new_tokens <= 0: continue

  # 检查是否有足够空闲块
  if not kv_cache_manager.can_allocate(num_new_tokens):
      break  # 内存不足，停止从 WAITING 拉取

  # 分配 KV 块并移至 RUNNING
  kv_cache_manager.allocate_slots(request, num_new_tokens, computed_blocks)
  waiting → running
  scheduled_new_reqs.append(request)
  remaining_tokens -= num_new_tokens

────────────────────────────────────────
构建 SchedulerOutput
────────────────────────────────────────
return SchedulerOutput(
    scheduled_new_reqs=scheduled_new_reqs,    # 首次调度的请求
    scheduled_cached_reqs=scheduled_cached,   # 继续执行的请求
    num_scheduled_tokens={req_id: n, ...},    # 本步各请求处理 token 数
    total_num_scheduled_tokens=sum(n),
    scheduled_spec_decode_tokens={...},       # 推测解码 token
    finished_req_ids=finished,                # 本步完成的请求
    preempted_req_ids=preempted,              # 被抢占的请求
    ...
)

7.5 调度器关键参数

参数 默认值 作用

7.6 Chunked Prefill 对 TTFT/TPOT 的影响

不启用 Chunked Prefill:
  ┌──────────────────────────────┬──┬──┬──┬──┐
  │  长 Prefill (2048 tokens)    │D1│D2│D3│D4│  ← Decode 被阻塞
  └──────────────────────────────┴──┴──┴──┴──┘
  问题：TTFT 极高（等待长 Prefill 完成），TPOT 受影响

启用 Chunked Prefill (chunk_size=512):
  ┌──────┬──┬──────┬──┬──────┬──┬──────┬──┐
  │ P512 │D1│ P512 │D2│ P512 │D3│ P512 │D4│  ← Prefill 与 Decode 混合
  └──────┴──┴──────┴──┴──────┴──┴──────┴──┘
  好处：TTFT 分摊，TPOT 稳定，GPU 利用率更均匀

7.7 抢占机制

抢占触发条件：RUNNING 中的请求 KV 块分配失败

抢占策略（RECOMPUTE 模式）:
  1. 选择最低优先级（FCFS 中最后进入的）请求
  2. 释放其所有 KV 块（ref_count-- → 如归零则加入 free_pool）
  3. 请求状态: RUNNING → WAITING
  4. 本请求保留 prompt_token_ids，下次重新从头调度

  注意：如果 APC 的前缀块未被驱逐，重调度时可命中缓存，
        只需重新计算被驱逐后的部分

8. 关键数据结构

8.1 Request（调度器内部状态）

class Request:
    request_id: str
    prompt_token_ids: list[int]          # 原始 prompt token
    sampling_params: SamplingParams
    arrival_time: float

    # 状态
    status: RequestStatus                 # WAITING/RUNNING/FINISHED_*
    num_computed_tokens: int             # 已有 KV Cache 的 token 数
    output_token_ids: list[int]          # 已生成的 token

    # KV Cache 相关
    block_ids: list[list[int]]           # [layer_idx][block_idx] → block_id
    block_hashes: list[BlockHash]        # 用于 APC 的块哈希

    # 属性（只读）
    @property
    def num_tokens(self) -> int:
        return len(self.prompt_token_ids) + len(self.output_token_ids)

    @property
    def is_prefill(self) -> bool:
        return self.num_computed_tokens < len(self.prompt_token_ids)

10. 关键设计决策

10.1 为什么用 num_computed_tokens 而非 Prefill/Decode 状态？

传统框架将请求分为 Prefill（处理 prompt）和 Decode（生成 token）两阶段。vLLM v1 改为单一整数 num_computed_tokens：

好处：

  1. Chunked Prefill 自然支持：Prefill 分块只是多次增加 num_computed_tokens
  2. 推测解码统一：Spec token 也计入 num_tokens，不需要特殊处理
  3. 代码简化：消除了大量 if is_prefill 条件分支
  4. 未来扩展性：支持 Jump Decoding 等新模式

10.2 为什么用进程隔离？

单进程方案的问题 Tokenize（CPU密集）→ 阻塞 GPU 推理；GPU 推理 → 阻塞 Tokenize；Python GIL → 真正的 CPU 并行度受限。
进程分离方案 前端进程: Tokenize/Detokenize（CPU密集）并发运行；后端进程: GPU 推理 → 两者真正并行，互不阻塞。
