SGLang Disaggregation 分离架构深度解析

Prefill-Decode 分离、KV Cache 跨机传输、NIXL/Mooncake 后端 -- 源码级分析

1. PD 分离架构概览

SGLang 的 Disaggregation（分离架构）将 LLM 推理的 Prefill（预填充） 和 Decode（解码） 阶段部署在不同的 GPU 实例上，通过高速网络传输 KV Cache，从而实现：

  • 资源异构配置：Prefill 需要高计算密度，Decode 需要高内存带宽，可独立优化
  • 弹性扩缩容：Prefill 和 Decode 实例可独立扩展
  • TTFT 优化：Prefill 实例专注首 Token 延迟，不受 Decode 影响

3.1 PrefillBootstrapQueue

负责请求的连接建立和初始化：

  1. 为每个请求创建 KVSender 实例
  2. 通过 poll_and_all_reduce() 跨 TP rank 同步引导状态
  3. 引导完成后分配 metadata_buffer_index，调用 sender.init(num_pages)
  4. 将请求移入 Waiting Queue
TP 一致性保证：poll_and_all_reduce() 使用 Gloo AllReduce 对所有 TP rank 的 poll 结果取 MIN，确保只有所有 rank 都完成引导时请求才进入下一阶段。这避免了部分 rank 开始传输而其他 rank 尚未就绪的情况。 
def poll_and_all_reduce(pollers, gloo_group):
    polls = [int(poller.poll()) for poller in pollers]
    tensor_to_reduce = torch.tensor(polls, dtype=torch.uint8, device="cpu")
    dist.all_reduce(tensor_to_reduce, op=dist.ReduceOp.MIN, group=gloo_group)
    return tensor_to_reduce.tolist()

3.2 事件循环

Prefill 调度器提供两种事件循环模式：

模式特点
event_loop_normal_disagg_prefill同步处理：接收请求 -> 调度批次 -> 执行 -> 处理结果 -> 轮询传输
event_loop_overlap_disagg_prefill重叠执行：当前批次的 forward 与上一批次的结果处理并行

3.3 Chunked KV 传输

对于长序列的 chunked prefill，KV Cache 分块传输：

  • 非最后一块：延迟最后一个不完整页，仅传输 end_idx - end_idx % page_size
  • 最后一块 (last_chunk=True)：传输剩余页并附带 metadata (output_ids, logprobs, hidden_states)
  • 通过 req.start_send_idx 追踪已发送的偏移量

4.1 DecodePreallocQueue -- 内存预分配

Decode 端的关键创新是 提前预分配 KV Cache 内存：

  • DecodeReqToTokenPool 扩展了标准的 ReqToTokenPool，增加了 pre_alloc_size 额外槽位
  • 运行中的请求受 max_running_requests 限制，但预分配请求可使用额外空间
  • 这解耦了预分配与运行的资源竞争，使 Prefill 端不被 Decode 端的内存压力阻塞

内存估算公式：

allocatable_tokens = available_size - max(
    num_reserved_decode_tokens * (running + transfer + waiting),
    need_space_for_single_req  # 保证至少一个请求能完成
)

4.2 DecodeTransferQueue -- 传输状态管理

当 KV 传输完成时，_commit_transfer_to_req() 从 metadata buffer 中提取：

数据来源说明
output_idMetadataBuffers.output_idsPrefill 生成的第一个 token
cached_tokensMetadataBuffers.cached_tokens缓存命中的 token 数
logprobsMetadataBuffers.output_token_logprobs_*token 级别 log 概率
topk_p/indexMetadataBuffers.output_topk_*投机解码的 top-k 信息
hidden_statesMetadataBuffers.output_hidden_statesEAGLE 投机解码的隐藏状态

4.3 PrebuiltExtendBatch

Decode 端不执行真正的 Prefill Forward，而是构建"预构建扩展批次"：

new_batch.prepare_for_prebuilt()    # 标记为预构建模式
new_batch.process_prebuilt(...)     # 填充元数据，跳过前向传播

4.4 KV Cache 回收 (Retraction)

当 Decode 端内存不足时，支持请求回收 (retraction)：

  1. 运行中的请求的 KV Cache 可被 offload 到 CPU
  2. 回收的请求进入 retracted_queue
  3. resume_retracted_reqs() 在内存充裕时将请求恢复

5.2 页索引转换

KV Cache 使用分页管理。传输时需要将 token 级别的 kv_indices 转换为 page 级别：

def kv_to_page_indices(kv_indices, page_size):
    if page_size == 1:
        return kv_indices
    return kv_indices[::page_size] // page_size

def kv_to_page_num(num_kv_indices, page_size):
    return (num_kv_indices + page_size - 1) // page_size

6.2 连接建立流程

  1. Decode 端注册：NixlKVReceiver 通过 ZMQ 向 Prefill 发送 KVArgsRegisterInfo（agent_metadata、kv_data_ptrs、gpu_id 等）
  2. Prefill 端添加对端：_add_remote_peer() 调用 agent.add_remote_agent() 建立 RDMA 连接
  3. 传输请求：Decode 端发送 TransferInfo（room、dst_kv_indices、dst_aux_index）
  4. Bootstrap 线程：当所有 Decode TP rank 的信息到齐时，标记 KVPoll.WaitingForInput

6.3 KV Cache 传输

send_kvcache() 的核心逻辑：

  1. 索引分组：group_concurrent_contiguous() 将连续的页索引合并为大块传输，减少 RDMA 操作数
  2. 构建描述符：计算源/目标地址（kv_ptr + index * item_len），调用 agent.get_xfer_descs()
  3. 执行传输：agent.initialize_xfer("WRITE", src, dst, peer, notif) + agent.transfer()
  4. 通知：传输完成后自动发送 notification 给 Decode 端

6.4 跨 TP 切片传输

当 Prefill 和 Decode 的 TP 大小不同时，send_kvcache_slice() 按 KV head 维度进行切片传输：

Head 分配逻辑：

  • prefill_tp > decode_tp：多个 Prefill rank 向一个 Decode rank 发送各自的 head 子集
  • prefill_tp < decode_tp：一个 Prefill rank 向多个 Decode rank 发送 head 子集
dst_heads_per_rank = num_kv_heads * prefill_tp_size // decode_tp_size
bytes_per_head_slice = dst_kv_item_len // page_size // dst_heads_per_rank

6.5 TransferStatus 追踪

Decode 端通过 TransferStatus 追踪多 PP rank 的分块传输进度：

字段说明
received_kvs_per_pp每个 PP rank 已接收的 chunk ID 集合
expected_kvs_per_pp每个 PP rank 预期的 chunk 总数（最后一个 chunk 设置）
num_pp_ranks_expected预期的 PP rank 总数
received_aux辅助数据（metadata）是否已接收

6.6 心跳与容错

Decode 端运行 heartbeat_checker 后台线程：

  • 周期性 HTTP GET /health 检查 Prefill 实例存活
  • 连续失败 max_failures 次后触发 _handle_node_failure()
  • 标记所有关联的传输为 KVPoll.Failed，清理连接池

7.3 错误处理

Mooncake 定义了专用异常类 KVTransferError，包含 bootstrap_room 和 failure_reason，用于精确定位传输失败的请求。传输线程池使用 concurrent.futures.ThreadPoolExecutor 异步执行传输，通过 FastQueue 收集结果。

ZmqEventPublisher 的关键特性：

9.2 性能优化要点

  • 连续索引合并：group_concurrent_contiguous() 将相邻页合并为单次 RDMA 操作
  • 异步传输：KV 传输在独立 CUDA Stream / RDMA 通道上执行，不阻塞计算
  • 预分配解耦：Decode 端提前分配内存，使 Prefill 可以立即开始传输
  • Overlap 模式：支持计算与传输的流水线重叠
  • 故障快速检测：心跳机制 + 超时检测，快速标记失败请求
  • TP 一致性：所有 TP rank 通过 Gloo AllReduce 同步 poll 状态
