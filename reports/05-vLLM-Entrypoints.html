vLLM Entrypoints 服务入口深度解析

TYPE: REPO ANALYSIS DATE: 2026-04-10 TOPIC: Entrypoints / API Server / gRPC / Configuration

TL;DR

  • vLLM 提供三类入口：LLM 同步离线批处理、AsyncLLM 驱动的 OpenAI 兼容 HTTP Server、以及面向微服务的 gRPC Server。
  • OpenAI API Server 基于 FastAPI + uvicorn，各端点（chat/completion/embedding）分别由独立 Serving 类处理，支持 SSE 流式和非流式两种响应模式。
  • LLM.generate() 底层循环调用 engine.step() 直至所有请求完成，适合脚本/批处理场景；AsyncLLM 则通过 async generator 实现真正的异步流式输出。
  • 所有配置收敛到 VllmConfig，由 EngineArgs.create_engine_config() 从命令行参数生成，再下发给 ModelConfig、CacheConfig、SchedulerConfig、ParallelConfig 等十余个子配置对象。
  • Tokenizer 池（TokenizerGroup）支持本地单实例和 Ray remote actor 两种模式；增量 detokenize 采用"新旧文本取差值"策略，避免逐 token 全量解码。

4.2 Chat Completions 端点

@app.post("/v1/chat/completions")
async def create_chat_completion(
    request: ChatCompletionRequest,
    raw_request: Request,
):
    # 1. 验证请求
    generator = await chat.create_chat_completion(request, raw_request)

    # 2. 流式响应
    if request.stream:
        return StreamingResponse(content=generator, media_type="text/event-stream")

    # 3. 非流式响应
    else:
        return await generator

6. gRPC Server

文件：vllm/entrypoints/grpc/

6.1 适用场景

gRPC 接口适合：

  • 微服务间内部调用（低延迟、二进制协议）
  • 需要强类型协议定义的场景
  • 高并发内部服务

6.2 接口对比

特性 OpenAI HTTP gRPC
协议HTTP/1.1 + JSONHTTP/2 + Protobuf
序列化JSON（文本）Protobuf（二进制）
流式SSEgRPC 流
延迟高（解析开销）低（二进制）
兼容性高（OpenAI 生态）低（需要 gRPC client）

8. Tokenizer 管理

文件：vllm/tokenizers/

8.1 TokenizerGroup

为了支持并发 tokenize，vLLM 使用 TokenizerGroup（tokenizer 池）：

class TokenizerGroup:
    """Tokenizer 池（可选：Ray remote actor，支持真正并行）"""

    def __init__(self, tokenizer_id, max_num_seqs, ...):
        # 本地：单个 tokenizer 实例
        self.tokenizer = get_tokenizer(tokenizer_id, ...)

        # Ray 模式：多个 tokenizer actor（CPU 并行）
        if use_ray:
            self.tokenizer_actors = [
                RayTokenizerGroupPool.remote(...) for _ in range(max_num_seqs)
            ]

    async def encode(self, prompt: str) -> list[int]:
        return self.tokenizer.encode(prompt)

    async def decode(self, token_ids: list[int]) -> str:
        return self.tokenizer.decode(token_ids)

8.2 增量 Detokenize

OutputProcessor 使用增量 detokenize 避免重复解码：

增量 Detokenize 策略

flowchart LR A["token_ids[:-1]"] -->|"decode()"| B["prev_text"] C["token_ids"] -->|"decode()"| D["curr_text"] B --> E["delta = curr_text[len(prev_text):]"] D --> E E --> F["输出增量文本 delta"] F --> G["缓存 curr_text\n供下次使用"] 
每步新 token 时:
  # 不能简单 decode([new_token])，因为 tokenizer 可能有上下文依赖
  # 正确做法：decode([last_few_tokens + new_token]) 然后取差值

  prev_text = decode(token_ids[:-1])   # 上次的完整文本
  curr_text = decode(token_ids)        # 包含新 token 的完整文本
  delta = curr_text[len(prev_text):]   # 增量部分

  # 优化：缓存上次解码结果，避免重复解码
  # 通常只需解码最后 3-5 个 token 来捕捉跨 token 边界的字符
Note 增量 detokenize 的上下文依赖问题源于 BPE / SentencePiece 中字节级别的 token 边界问题。比如 "▁Hello" 这个 token 在不同上下文下解码结果不同，因此必须携带前缀 context 一起解码再取差值，而不能孤立解码单个新 token。
