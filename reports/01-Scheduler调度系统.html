深入分析事件循环、连续批处理、Prefill/Decode调度、调度策略及多进程通信机制

1. 系统总览

SGLang 的调度系统是整个推理引擎的核心，负责管理请求的生命周期、GPU 资源分配与计算调度。调度系统由多个协作进程组成，通过 ZMQ 进行高效的进程间通信。

3.2 重叠调度模式 (event_loop_overlap)

重叠模式的核心思想是：在 GPU 执行当前批次的同时，CPU 处理上一个批次的结果，实现 CPU/GPU 计算重叠。

sequenceDiagram participant CPU as CPU调度 participant GPU as GPU计算 participant Queue as 结果队列 loop 每次迭代 CPU->>CPU: recv_requests() + process_input_requests() CPU->>CPU: get_next_batch_to_run() -> batch_N CPU->>GPU: run_batch(batch_N) [异步] GPU-->>Queue: (batch_N, result_N) 入队 CPU->>Queue: pop batch_N-1 结果 CPU->>CPU: process_batch_result(batch_N-1) CPU->>CPU: launch_batch_sample_if_needed() end

重叠调度关键约束

  • 连续两个 prefill 批次时禁用重叠（由 SGLANG_DISABLE_CONSECUTIVE_PREFILL_OVERLAP 控制）以改善首 token 延迟
  • 投机解码 + 语法约束的场景暂不支持重叠模式
  • 使用 FutureMap 管理异步前向结果

4.3 Chunked Prefill

当启用 chunked prefill 时，长序列的 prefill 会被分成多个 chunk 逐步执行，避免长序列独占 GPU。关键参数：

  • chunked_prefill_size: 每次 prefill 的最大 token 数
  • enable_mixed_chunk: 允许 chunk 与 decode 混合执行
  • enable_dynamic_chunking: 流水线并行下的动态分块

6.2 In-Batch Prefix Caching

核心优化：当等待队列中多个请求共享相同前缀但现有缓存中没有该前缀时，调度器会临时降低后续请求的优先级，仅调度其中一个请求先行 prefill，后续请求可复用新建的缓存。阈值由 IN_BATCH_PREFIX_CACHING_CHECK_THRESHOLD（默认32）和 IN_BATCH_PREFIX_CACHING_DEPRIORITIZE_THRESHOLD（默认32）控制。

关键预算计算：

7. TokenizerManager

TokenizerManager 是请求进入系统的第一个处理进程，运行在独立进程中，使用 uvloop 异步事件循环。

sequenceDiagram participant API as FastAPI Server participant TM as TokenizerManager participant S as Scheduler participant DT as DetokenizerManager API->>TM: generate_request(text, params) TM->>TM: tokenize(text) -> input_ids TM->>TM: 处理多模态数据 TM->>TM: 验证输入长度 TM->>S: send TokenizedGenerateReqInput [ZMQ] Note over TM: 创建 ReqState, 等待 event S-->>DT: BatchTokenIDOutput [ZMQ] DT-->>TM: BatchStrOutput [ZMQ] TM->>TM: 更新 ReqState TM->>API: 返回流式/完整响应

TokenizerManager 关键功能

  • 异步分词：使用 AsyncDynamicbatchTokenizer 对请求进行批量分词
  • 多模态处理：通过 AsyncMMDataProcessor 异步处理图像/视频/音频
  • 请求状态管理：ReqState 维护每个请求的中间状态和增量输出
  • 流式输出：通过 asyncio.Event 实现非阻塞等待和增量推送
  • LoRA 注册表：管理 LoRA adapter 的加载和选择

9.2 DPBudget 预算跟踪

DPBudget 维护每个 DP worker 的实时负载信息，通过 WatchLoadUpdateReq 消息持续更新。分发时会乐观地递增目标 worker 的计数器，避免短时间内重复分配到同一 worker。

10.2 new_token_ratio 动态衰减

为 decode 阶段预留内存的比例从 init_new_token_ratio 逐步衰减到 min_new_token_ratio，避免过度保守导致吞吐量下降：

# 初始化
init_new_token_ratio = SGLANG_INIT_NEW_TOKEN_RATIO * schedule_conservativeness
min_new_token_ratio = init_new_token_ratio * SGLANG_MIN_NEW_TOKEN_RATIO_FACTOR
new_token_ratio_decay = (init - min) / SGLANG_NEW_TOKEN_RATIO_DECAY_STEPS

# 每步衰减
new_token_ratio = max(new_token_ratio - decay, min_new_token_ratio)
# retract 时重置
new_token_ratio = init_new_token_ratio

12. 总结

SGLang 调度系统的核心创新

  • 连续批处理：动态合并 prefill/decode 请求，最大化 GPU 利用率
  • RadixTree 前缀缓存：通过 LPM 策略最大化 KV 缓存复用
  • In-Batch Prefix Caching：等待队列内的前缀去重优化
  • 重叠调度：CPU/GPU 计算重叠，隐藏调度开销
  • 动态 token ratio：自适应内存预留策略
  • 多级并行：支持 TP、PP、DP、DP-Attention 多种并行模式
  • Mixin 模块化：通过继承 Mixin 实现功能正交分离
  • ZMQ 进程间通信：高效的多进程协作架构
