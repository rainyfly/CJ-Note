vLLM 调度架构深度解析：Continuous Batching 与 PagedAttention

TYPE: REPO ANALYSIS + FEATURE DEEP DIVE DATE: 2026-04-10 TOPIC: Scheduling / KV Cache / Memory Management

TL;DR

  • vLLM 的核心创新是 PagedAttention：借鉴操作系统虚拟内存的分页机制，将 KV Cache 切成固定大小的 block，彻底消除内存碎片，GPU 内存利用率从 ~60% 提升到 >96%。
  • Continuous Batching（也叫 iteration-level scheduling）让调度粒度从"一个请求结束才换"降到"每次 forward pass 后都可以换"，使得短请求不再被长请求阻塞，吞吐量提升 2-24x（对比 FasterTransformer 等静态批处理系统）。
  • 调度器维护三个队列（waiting / running / swapped），每个 decode step 做一次调度决策：优先推进 running 中的序列，KV block 不够时把低优先级序列 swap 到 CPU，必要时 preempt（抢占）并丢弃 KV cache 后重新计算。
  • Block Manager 是调度与执行之间的抽象层，管理 GPU block、CPU block 与 block table，支持 copy-on-write 语义实现 beam search 的 KV cache 共享。
  • vLLM v1 重构（2024 Q4）引入了新的 Scheduler + KVCacheManager 分离架构，解耦了调度逻辑与内存分配，为 prefix caching、chunked prefill 等特性打下基础。

1. 背景：LLM 推理的内存瓶颈

LLM 推理分两个阶段：prefill（处理所有输入 token，计算初始 KV cache）和 decode（每次自回归生成一个 token）。KV cache 的作用是缓存历史 token 的 Key/Value 张量，避免每步都重算——这是 transformer 推理加速的基础。

但 KV cache 的内存占用惊人。以 LLaMA-13B 在 A100-40G 上为例：

  • 模型权重：~26 GB（float16）
  • 单个请求的 KV cache：每个 token 约 2 × num_layers × num_heads × head_dim × sizeof(dtype)，对 LLaMA-13B 约 800 KB/token（40 layers × 2 × fp16）
  • 一条 2048 token 的对话：~1.6 GB KV cache

传统静态内存分配方案（Orca、FasterTransformer 等早期系统）在请求进入时直接按 最大可能长度（max_seq_len）预分配 KV cache，导致两个严重问题：

传统方案的两大杀手
  • 内部碎片（Internal Fragmentation）：预分配了 2048 slots 但实际只用了 300 token，剩余 1748 slots 的内存被白白占用，直到请求结束才释放。
  • 外部碎片（External Fragmentation）：不同请求的 KV cache 块大小不一，释放后的空洞无法被其他请求利用，内存实际利用率通常只有 60-80%。
实测：在 OPT-13B 上，传统方案的内存浪费高达 60-80%，系统实际能并发的请求数远低于 GPU 内存的理论上限。

这两个问题直接压缩了系统的 batch size 上限，而 LLM 推理的吞吐量几乎线性正比于 batch size（只要不触发内存 OOM）。解决内存碎片问题，就是解决吞吐量问题。

2. PagedAttention：操作系统式 KV Cache 管理

核心思想：把 KV cache "分页"

PagedAttention 的灵感直接来自操作系统的虚拟内存/分页机制：不再为每个请求预分配一大块连续内存，而是把 KV cache 切分成固定大小的 block（默认 16 tokens/block），按需动态分配。每个请求有一张 block table，记录逻辑 block 编号到物理 block 编号的映射——就像页表一样。

PagedAttention 内存布局 vs 传统方案

Block 粒度的选择

默认 block size = 16 tokens。这是一个工程权衡：

  • block 越大：内部碎片越多（最后一个 block 可能大半为空），但 block table 更小、访问开销更低。
  • block 越小：碎片最小，但 block table 变大，attention kernel 的 scatter/gather 开销更高。
  • 16 tokens 对大多数模型是一个合理的中点，实验表明在 A100 上 block size 8-32 的性能差异在 5% 以内。

PagedAttention Kernel

关键问题：Attention 计算本来假设 KV cache 连续，分页后物理地址不连续，标准的 FlashAttention kernel 无法直接用。vLLM 实现了专门的 paged_attention_v1/v2 CUDA kernel，接受 block table 作为参数，在 attention 计算时按 block 粒度做 gather：

template<typename scalar_t, int HEAD_SIZE, int BLOCK_SIZE>
__global__ void paged_attention_v1_kernel(
    scalar_t* __restrict__ out,           // [num_seqs, num_heads, head_size]
    const scalar_t* __restrict__ q,       // [num_seqs, num_heads, head_size]
    const scalar_t* __restrict__ k_cache, // [num_blocks, num_kv_heads, head_size/x, block_size, x]
    const scalar_t* __restrict__ v_cache, // [num_blocks, num_kv_heads, head_size, block_size]
    const int* __restrict__ block_tables, // [num_seqs, max_num_blocks_per_seq]
    const int* __restrict__ context_lens, // [num_seqs]
    ...
) {
    // 1. 从 block_tables 取出当前序列的物理 block 编号
    // 2. 对每个 block，计算 QK^T / sqrt(d) 的部分和
    // 3. 分块 softmax + weighted sum（类似 FlashAttention 的 online softmax）
}

内存布局是关键优化点：K cache 被存为 [num_blocks, num_kv_heads, head_size/x, block_size, x]，其中最内层维度 x 是向量化宽度（fp16: x=8, bf16: x=8, int8: x=16），这样 warp 内的线程可以合并读取，避免 non-coalesced access。

Copy-on-Write：Beam Search 的 KV 共享

Beam search 会产生多个候选序列，这些序列在分叉前共享相同的 prefix KV cache。PagedAttention 通过引用计数和 copy-on-write 实现共享：

Beam Search KV Cache 共享（Copy-on-Write）

graph LR P["Prefix blocks (shared)\nref_count=3"] -->|"beam 1 写入新token"| COW["COW: 复制最后一个block\n再写入"] P --> B2["beam 2 继续共享"] P --> B3["beam 3 继续共享"] COW --> NB1["beam 1 私有block"]

实现上，BlockManager.fork() 只复制 block table（浅拷贝），物理 block 的引用计数 +1。当某个 beam 需要写入时，若 ref_count > 1，则先执行 cow（分配新 block，复制内容，将该 beam 的 block table 指向新 block，ref_count -1）。

3. Continuous Batching：迭代级调度

问题：静态批处理的低效

传统做法：把一批请求打包成一个 batch，等到 batch 中 所有请求都结束（即最长的那个结束）才释放，换入下一个 batch。这叫 request-level batching。

Request-Level Batching 的问题 设想一个 batch 中有一个请求需要生成 500 tokens，其他 7 个请求只需要 50 tokens。当 7 个请求结束后，剩余 450 个 decode step GPU 只在服务 1 个请求，batch size 实际只有 1/8，吞吐量急剧下降。这叫 head-of-line blocking（队头阻塞）。

解决方案：Iteration-Level Scheduling

vLLM 把调度粒度降到 每次 forward pass（iteration）之后。每次 decode step 结束后，调度器检查：有没有请求已经完成（遇到 EOS 或达到 max_tokens）？有没有等待中的新请求可以塞进来？基于当前可用的 KV cache block 数量，动态调整 batch 组成。

连续批处理的本质是：把请求的生命周期从「batch 粒度」解耦到「token 粒度」。这需要 PagedAttention 的支持——正因为 KV cache 是按 block 动态分配的，可以随时为新请求分配 block、随时释放已结束请求的 block，而不需要预留连续的内存区间。

Chunked Prefill（分块预填充）

一个更进一步的优化：长 prompt 的 prefill 阶段可能占用整个 GPU 几百毫秒，期间所有 decode 请求都在等待，造成尾部延迟飙升（TTFT 和 TPOT 都受影响）。

Chunked Prefill 把一个长 prompt 切成多个 chunk（如 512 tokens/chunk），每次 iteration 只处理一个 chunk，同时也服务正在 decode 的请求。这样 decode 请求的延迟不会被 prefill 请求的 "大计算" 阻断。代价是该请求的 TTFT（首 token 时延）增加，但 TPOT（每 token 生成时间）和 P99 延迟更平滑。

4. 调度器架构与状态机

三队列模型

vLLM 调度器维护三个队列，对应请求的三种状态：

队列含义KV Cache 状态
waiting 已到达但尚未开始 prefill 的请求 无 KV cache（尚未分配）
running 正在 GPU 上执行 prefill 或 decode KV blocks 在 GPU 显存
swapped 因显存不足被抢占，KV cache 换到 CPU KV blocks 在 CPU 内存

请求状态机与调度决策

stateDiagram-v2 [*] --> waiting : 新请求到达 waiting --> running : GPU blocks充足 → prefill running --> running : decode继续（每次iteration） running --> swapped : GPU blocks不足 → swap out（KV移到CPU） running --> waiting : GPU blocks严重不足 → preempt（丢弃KV cache，重新排队） swapped --> running : GPU blocks释放 → swap in（KV移回GPU） swapped --> waiting : 长时间未能swap in → recompute策略 running --> [*] : 生成EOS或达到max_tokens

调度循环（每次 iteration）

核心调度逻辑在 vllm/core/scheduler.py 的 Scheduler.schedule() 方法：

# vllm/core/scheduler.py（概要，v0.x 架构）
def schedule(self) -> Tuple[List[SequenceGroup], SchedulerOutputs]:
    # 1. 尝试 swap in：把 swapped 队列中的请求移回 GPU
    #    条件：能为该请求的所有 blocks 分配 GPU 内存
    blocks_to_swap_in = {}
    for seq_group in reversed(self.swapped):
        alloc = self.block_manager.can_swap_in(seq_group)
        if alloc == AllocStatus.OK:
            self.block_manager.swap_in(seq_group)
            self.running.append(seq_group)
            self.swapped.remove(seq_group)
        elif alloc == AllocStatus.NEVER:  # 永远不够，直接丢弃
            ...

    # 2. 尝试 append slot：为 running 中每个请求的下一个 token 分配 block
    #    注意：decode 阶段每个 token 可能需要分配新 block（当前 block 满了）
    blocks_to_copy = {}
    preempted = []
    for seq_group in self.running:
        while not self.block_manager.can_append_slot(seq_group):
            # 没有空闲 block，必须抢占（preempt）优先级最低的请求
            victim = self.running[-1]  # FCFS策略下是最晚到的
            if victim == seq_group:
                break
            self._preempt(victim, preempted, blocks_to_swap_out)
        if self.block_manager.can_append_slot(seq_group):
            self.block_manager.append_slot(seq_group, blocks_to_copy)

    # 3. 尝试 prefill：从 waiting 队列取新请求
    #    条件：有足够 GPU blocks 分配 prefill 所需的全部 KV
    scheduled = []
    for seq_group in self.waiting:
        if self.block_manager.can_allocate(seq_group) == AllocStatus.OK:
            self.block_manager.allocate(seq_group)
            self.running.append(seq_group)
            scheduled.append(seq_group)
            self.waiting.remove(seq_group)
        else:
            break  # FCFS：不跳过，避免 starvation

    return self.running + scheduled, SchedulerOutputs(
        scheduled_seq_groups=...,
        blocks_to_swap_in=blocks_to_swap_in,
        blocks_to_swap_out=blocks_to_swap_out,
        blocks_to_copy=blocks_to_copy,
    )

抢占策略

当 GPU 内存不足时，调度器有两种抢占方式：

Swap（换出） 把序列的 KV cache blocks 通过 cudaMemcpy 从 GPU 搬到 CPU DRAM，释放 GPU blocks。序列进入 swapped 队列等待 swap in。优点：不丢失计算结果。缺点：PCIe 传输有延迟（~10 GB/s），且占用 CPU 内存。
Recompute（重计算） 直接丢弃序列的所有 KV cache，序列退回 waiting 队列，下次调度时重新 prefill。优点：不占 CPU 内存，实现简单。缺点：重新 prefill 需要重新计算 prompt 的 KV cache（浪费算力）。

vLLM 默认使用 Swap 策略。对于很长的 prompt，Recompute 可能比 Swap 更快（prefill 速度比 PCIe 带宽快）。--preemption-mode 参数可以控制。

调度策略：FCFS

vLLM 默认使用 先来先服务（FCFS）。不跳过队列中的请求（即使后面的请求能塞进去，也不会跳过前面 "太大" 的请求），这保证了公平性、避免了饥饿（starvation）。代价是有时会因为一个大请求把小请求挡在外面。

优先级调度（priority scheduling）在 vLLM 中已有接口但生产上较少使用，更常见的是在业务层面做排队控制。

5. Block Manager 工程实现

核心数据结构

Block Manager 是调度器与执行引擎之间的内存管理抽象层，核心文件是 vllm/core/block_manager_v1.py（v0.x 架构）。

# vllm/core/block_manager_v1.py
class BlockAllocator:
    """管理 GPU 或 CPU 上的物理 block 池"""
    def __init__(self, device: Device, block_size: int, num_blocks: int):
        self.free_blocks: List[PhysicalTokenBlock] = [
            PhysicalTokenBlock(device=device, block_number=i, block_size=block_size)
            for i in range(num_blocks)
        ]

    def allocate(self) -> PhysicalTokenBlock:
        block = self.free_blocks.pop()
        block.ref_count = 1
        return block

    def free(self, block: PhysicalTokenBlock) -> None:
        block.ref_count -= 1
        if block.ref_count == 0:
            self.free_blocks.append(block)  # 引用计数归零才真正回收

class BlockSpaceManagerV1:
    def __init__(self, block_size, num_gpu_blocks, num_cpu_blocks, ...):
        self.gpu_allocator = BlockAllocator(Device.GPU, block_size, num_gpu_blocks)
        self.cpu_allocator = BlockAllocator(Device.CPU, block_size, num_cpu_blocks)
        # seq_id → List[PhysicalTokenBlock]
        self.block_tables: Dict[int, BlockTable] = {}

    def allocate(self, seq_group: SequenceGroup) -> None:
        """为 seq_group 中所有序列分配 GPU blocks（prefill 时调用）"""
        for seq in seq_group.get_seqs(SequenceStatus.WAITING):
            block_table: BlockTable = []
            for _ in range(ceil(seq.get_len() / self.block_size)):
                block = self.gpu_allocator.allocate()
                block_table.append(block)
            self.block_tables[seq.seq_id] = block_table

    def append_slot(self, seq: Sequence) -> Optional[Tuple[int, int]]:
        """decode时：若当前最后一个block已满，分配新block；
           若最后一个block被共享（ref_count>1），执行 copy-on-write"""
        block_table = self.block_tables[seq.seq_id]
        last_block = block_table[-1]

        if last_block.ref_count == 1:
            # 独占：直接复用（追加写）
            return None
        else:
            # 共享：copy-on-write，分配新 block 并复制内容
            new_block = self.gpu_allocator.allocate()
            block_table[-1] = new_block
            self.gpu_allocator.free(last_block)  # ref_count - 1
            return last_block.block_number, new_block.block_number  # GPU 内复制

Block Table 传递到 Kernel

执行时，block table 被打包成一个整型张量传给 attention kernel：

# vllm/worker/model_runner.py
def _prepare_decode(self, seq_group_metadata_list):
    block_tables = []
    for seq_group_metadata in seq_group_metadata_list:
        for seq_id, block_table in seq_group_metadata.block_tables.items():
            # block_table: List[int]，物理 block 编号列表
            block_tables.append(block_table)
    # 对齐到 max_num_blocks_per_seq，padding 用 0
    block_tables_tensor = make_tensor_with_pad(
        block_tables, max_len=max_num_blocks_per_seq, pad=0, dtype=torch.int32
    )
    return block_tables_tensor  # 形状: [num_seqs, max_num_blocks_per_seq]

Prefix Caching（前缀缓存）

v0.3+ 引入的特性：如果多个请求共享相同的 system prompt（非常常见的场景），可以把这些 prefix blocks 缓存起来，新请求到来时直接命中缓存，跳过 prefill。实现上使用 hash(token_ids) 作为 key，维护一个 LRU cache。命中时，can_allocate 返回 CACHED，直接把缓存 block 的 ref_count +1 即可。

内存利用率对比（OPT-13B，A100-40G）

为什么吞吐量提升幅度差异如此大（2x-24x）？ 提升幅度取决于请求长度分布。当 input/output 长度差异大（如 ShareGPT 数据集）时，静态批处理的低效最为严重（长请求阻塞短请求），continuous batching 收益最大（24x）。当请求长度均匀时，提升幅度接近内存利用率提升比例（~2x）。

PagedAttention Kernel 开销

非连续内存访问会不会让 attention 计算变慢？论文的测量结论：

  • Decode 阶段：PagedAttention kernel 与等效连续 attention 的速度差异 <5%，因为 decode 是 memory-bound 的，gather 操作的开销被访存延迟掩盖。
  • Prefill 阶段：vLLM 在 prefill 时仍然使用 FlashAttention（KV 是连续分配的），不受 paged 开销影响。
  • Block size 选择：过小的 block（如 1 token）会让 scatter/gather 开销显著，推荐 8-32 tokens/block。

8. 工程启示与局限

核心洞见：抽象层次的匹配

vLLM 最聪明的地方不是算法创新，而是找到了正确的抽象层次：把操作系统的内存管理范式（分页、虚拟地址、引用计数、换入换出）直接移植到 GPU KV cache 管理。这说明 好的系统设计往往是成熟范式在新约束条件下的适当映射，而不是从头发明。

调度器的优先级权衡

FCFS + continuous batching 的组合对吞吐量友好，但对延迟不友好。对于 online serving 场景，通常需要：

  • 设置 --max-num-seqs 控制并发上限，避免排队延迟无限增长
  • 用 --max-model-len 限制最大序列长度，控制每个请求的 KV cache 上限
  • 在上层（API Gateway）做基于 SLA 的优先级队列

Swap 的代价

PCIe 带宽（~32 GB/s 单向）相对 GPU 内存带宽（~2 TB/s A100）慢了 2 个数量级。如果 swap 频繁（负载远超 GPU 内存容量），系统吞吐量会因 PCIe 传输而下降，甚至比不 swap（直接 recompute）更慢。实际部署中应监控 vllm:num_preemptions_total metric，频繁 preemption 说明需要降低 max_num_seqs 或增加 GPU。

Multi-GPU 的挑战

vLLM 的 tensor parallelism 把 KV cache 分散在多个 GPU 上，block 分配需要同步所有 rank。每次 block 分配/释放都涉及跨 GPU 协调，增加了复杂度。Pipeline parallelism 下还需要管理 KV cache 在不同 pipeline stage 的 GPU 之间的一致性。

PagedAttention 的局限

已知局限
  • Block size 固定：不同模型的 head_dim 不同，最优 block size 也不同，但 vLLM 用统一配置，存在小量次优情况。
  • Prefix caching 的 hash 冲突：使用 token id 序列的 hash，极低概率的 hash 碰撞可能导致错误的 KV 复用（实际几乎不发生，但理论上存在）。
  • 碎片仍然存在：每个序列最后一个 block 的内部碎片（平均半个 block，即 block_size/2 tokens 的浪费）。这是无法完全消除的，只能通过调整 block_size 最小化。
  • 不适用于 MQA/GQA 的优化空间：GQA（Grouped Query Attention）减少了 KV head 数，block 分配可以相应减小，但 vLLM 的 block 大小计算已经考虑到 num_kv_heads，不是大问题。

未来方向

vLLM 社区正在积极推进的方向：

  • Disaggregated Prefill：把 prefill 和 decode 分离到不同机器，分别优化（Splitwise / DistServe 论文的思路）
  • Speculative Decoding：用小模型草稿 + 大模型验证，提升 decode 速度
  • Cross-request prefix caching：在请求之间共享 system prompt 的 KV cache（已有，且在 v1 默认开启）
  • Cascade inference：多层 KV cache 层级（GPU L1 → GPU L2 → CPU → NVMe），降低 large context 的代价
