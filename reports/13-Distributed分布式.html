1. 分布式架构总览

SGLang 的分布式模块（srt/distributed/）是从 vLLM 适配而来，接管了 PyTorch 原生分布式环境的控制。其核心设计原则：

  • 多后端通信：在同一个进程组中支持 NCCL、Gloo、Custom AllReduce、PyMscclpp、QuickAllReduce、TorchSymmMem 等多种通信实现
  • 自动路由：根据张量大小、CUDA Graph 状态、硬件拓扑自动选择最优通信路径
  • 惰性初始化：通信器仅在 world_size > 1 时创建
  • 设备无关：支持 CUDA、ROCm (HIP)、NPU、XPU、HPU、CPU 等多种加速器

2.3 自定义算子注册

SGLang 通过 register_custom_op 将 AllReduce 操作注册为 PyTorch 自定义算子，使其兼容 torch.compile：

@register_custom_op(mutates_args=["tensor"])
@register_split_op()
def inplace_all_reduce(tensor: torch.Tensor, group_name: str) -> None:
    group = _groups[group_name]()
    group._all_reduce_in_place(tensor)

@register_custom_op(out_shape="tensor")
def outplace_all_reduce(tensor, group_name, outplace_all_reduce_method) -> torch.Tensor:
    group = _groups[group_name]()
    return group._all_reduce_out_place(tensor, outplace_all_reduce_method)

4. PyNccl 通信器

PyNcclCommunicator 通过 ctypes 绑定直接调用 NCCL C 库，绕过 PyTorch 的 NCCL 后端，获得更精细的控制（如流管理、组操作）。

4.1 初始化过程

  1. 加载 NCCL 动态库 (NCCLLibrary)
  2. Rank 0 生成 ncclUniqueId，通过 Gloo 或 StatelessProcessGroup 广播到其他 rank
  3. 调用 ncclCommInitRank 创建通信句柄
  4. 创建独立的 CUDA Stream
  5. 执行小规模 warmup AllReduce
  6. 默认设置为 disabled=True，仅在 CUDA Graph 捕获时通过 change_state(enable=True) 启用

4.2 支持的通信操作

方法NCCL 调用说明
all_reduce()ncclAllReduce原地全归约（支持 SUM 等 ReduceOp）
all_gather()ncclAllGather / ncclBroadcast全收集（支持不等长 sizes）
reduce_scatter()ncclReduceScatter / ncclReduce归约散播（支持不等长 sizes）
send()ncclSend点对点发送
recv()ncclRecv点对点接收
broadcast()ncclBroadcast广播
group_start/end()ncclGroupStart/End组操作（合并多个通信）

4.3 流管理策略

流解析逻辑 (_resolve_stream)：
  1. 若显式提供 stream 参数，使用该 stream
  2. 若 use_current_stream=True，使用 torch.cuda.current_stream()
  3. 否则使用通信器自身的独立 stream (self.stream)
这允许在 CUDA Graph 模式下使用当前捕获流，在 eager 模式下使用独立流避免阻塞计算。

4.4 上下文并行 AllGather

cp_all_gather_into_tensor() 是专为 Context Parallelism 设计的非阻塞 AllGather，利用 PyNccl 的异步特性，允许通信与计算重叠。

5. Custom AllReduce

CustomAllreduce 利用 GPU 直接 P2P 访问（NVLink/PCIe）和 IPC 共享内存实现低延迟的 AllReduce，特别适合小张量场景（默认最大 8MB）。

5.1 设计约束

5.3 AllReduce 执行路径

def custom_all_reduce(self, input: torch.Tensor) -> Optional[torch.Tensor]:
    if self.disabled or not self.should_custom_ar(input):
        return None  # 退回到其他实现

    if self._IS_CAPTURING:
        if torch.cuda.is_current_stream_capturing():
            return self.all_reduce(input, registered=not self.tms_cudagraph)
        else:
            return torch.zeros_like(input)  # warmup 模拟
    else:
        return self.all_reduce(input, registered=False)

5.4 CUDA Graph 支持

通过 capture() 上下文管理器，在 CUDA Graph 捕获结束时调用 register_graph_buffers()，收集所有在图中使用的缓冲区地址并注册 IPC 句柄，使得图回放时能正确访问其他 rank 的缓冲区。

5.5 AMD 确定性 AllReduce

在 ROCm 平台上，当启用确定性推理（SGLANG_ENABLE_DETERMINISTIC_INFERENCE）时，使用特殊的 1-stage 内核 (deterministic_all_reduce) 确保固定顺序的归约操作，保证结果可复现。

7.1 Eager vs Graph 模式对比

AllReduce 方法Eager 模式CUDA Graph 模式
Quick AllReduce (AMD)启用启用
Custom AllReduce启用启用
PyNccl禁用启用
PyMscclpp禁用启用
TorchSymmMem禁用启用
torch.distributed启用禁用
设计原理：CUDA Graph 不兼容 torch.distributed 的隐式同步，因此在图模式下使用 PyNccl（直接操作 CUDA Stream）或 Custom AllReduce（IPC 共享内存）。而 PyMscclpp 在 eager 模式下需要预注册缓冲区导致开销过大，仅在图模式下使用。

8. CUDA Graph 集成

8.1 graph_capture 上下文管理器

GroupCoordinator.graph_capture() 负责设置 CUDA Graph 捕获期间的通信状态：

@contextmanager
def graph_capture(self, graph_capture_context=None, stream=None):
    # 1. 创建或使用指定的 CUDA Stream
    # 2. 进入 CustomAllreduce 的 capture() 上下文
    # 3. 启用 PyNccl（切换到当前流）
    # 4. 启用 PyMscclpp
    # 5. yield -- 在此期间捕获 CUDA Graph
    # 6. 退出时恢复所有通信器状态
    with maybe_ca_context, maybe_pynccl_context, maybe_pymscclpp_context:
        yield graph_capture_context

8.2 通信器状态切换

PyNccl 通信器通过 change_state() 上下文管理器在 eager/graph 模式间切换：

@contextmanager
def change_state(self, enable=None, stream=None):
    old_disable = self.disabled
    old_stream = self.stream
    self.stream = stream
    self.disabled = not enable
    yield
    self.disabled = old_disable
    self.stream = old_stream
