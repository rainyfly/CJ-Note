vLLM 推测解码（Speculative Decoding）深度解析

TYPE: REPO ANALYSIS DATE: 2026-04-10 TOPIC: Speculative Decoding / vLLM v1

TL;DR

  • 推测解码通过"轻量 Draft 猜多步 + Target 模型一次并行验证"打破 decode 的内存带宽瓶颈，在 batch=1 场景下最高可达 4× 加速。
  • vLLM v1 内置四种 Proposer：NgramProposer（零参数，适合重复文本）、EAGLE/EAGLE-2（利用 Target 隐藏状态，精度最高）、DraftModelProposer（独立小模型）、MedusaProposer（并行多头）。
  • RejectionSampler 在 GPU 上批量执行拒绝采样，数学上保证与 Target 模型输出分布完全一致，无精度损失。
  • EAGLE-2 引入动态树形 draft（而非链式），在低置信度时探索多条路径，进一步提升平均接受 token 数。
  • 推测解码仅对 small batch（内存带宽瓶颈）有效；large batch 场景下收益为零甚至负向，需按场景选型。

1. 推测解码概述

LLM Decode 阶段受内存带宽瓶颈制约——每步仅生成 1 个 token，却需要将所有模型权重从 HBM 加载到计算单元。 推测解码通过"猜测多个 token，一次性并行验证"绕开这一瓶颈：

核心直觉 批量验证 K 个 draft token 的代价与验证 1 个 token 近似（权重加载量相同）， 但如果有 α 个被接受，等效于完成了 α 步 decode，吞吐量提升 α 倍。

传统 Decode vs 推测解码流程对比

"45ms / 3 tokens" .-> result1(["普通速度"]) spec -. "16ms / 4 tokens → 2.8×" .-> result2(["加速!"]) style trad fill:#161b22,stroke:#30363d,color:#e6edf3 style spec fill:#161b22,stroke:#58a6ff,color:#e6edf3 style result2 fill:#1f6feb,stroke:#58a6ff,color:#e6edf3

推测解码收益最大的条件：

  • Draft 命中率高（代码生成、文档、RAG 引用等重复性强的文本）
  • Target 模型处于内存带宽瓶颈（即 batch_size 小）

2. 核心原理：提议 + 拒绝采样

2.1 算法框架

每个推测解码步骤分为三个阶段：

推测解码三阶段序列图

sequenceDiagram participant P as Proposer (Draft) participant T as Target Model participant R as RejectionSampler Note over P: 阶段 1 — Draft P->>P: 生成 K 个候选 token
（远低于 Target 一步的代价） P->>T: draft_tokens[1..K] + draft_probs Note over T: 阶段 2 — Verify T->>T: context + K draft tokens 送入
并行计算 K+1 个位置的 logits T->>R: target_logits[1..K+1] Note over R: 阶段 3 — Rejection Sampling R->>R: for i=1..K:
r ~ Uniform(0,1)
if r ≤ p_target[i] / p_draft[i]: accept
else: 从修正分布重采样, break R->>R: 若 K 全接受 → 额外从 K+1 位置采样 R-->>P: accepted_tokens (≥1 个)

2.2 拒绝采样数学保证

拒绝采样在数学上保证最终输出分布与 Target 模型完全一致，无精度损失。 核心公式： 

接受率 = min(1, p_target(x) / p_draft(x))

若 draft 分布 ≈ target 分布  →  接受率高  →  加速比高
若 draft 分布与 target 差异大  →  接受率低  →  无加速（但输出仍正确）

修正分布（拒绝时重采样）:
  p_corrected(x) ∝ max(0, p_target(x) − p_draft(x))
  归一化后从该分布采样 1 个 token
关键保证 无论 Draft 质量如何，最终输出分布均与单独运行 Target 模型等价。 差的 Draft 只影响加速比，不影响输出质量。

4. EAGLE 推测解码

文件：vllm/v1/spec_decode/eagle.py

4.1 EAGLE 原理

EAGLE（Extrapolation Algorithm for Greater Language-model Efficiency）利用 Target 模型 在 step N 产生的 hidden_states 作为 Draft 模型的输入，而非从头运行独立小模型：

EAGLE 隐藏状态复用机制

flowchart LR subgraph target["Target Model (Step N)"] TF["Forward Pass"] --> HS["hidden_states_N"] & LG["logits_N"] end LG --> ST["sampled_token_N"] HS --> |"缓存"| EH subgraph eagle["EAGLE Head (Step N+1 Draft)"] EH["EagleMlpFc\nLinear(2×hidden → hidden)"] --> |"concat(hidden_N, embed(token_N))"| EL["EagleLlamaLayer\n单层 Transformer"] EL --> LMH["LM Head (共享)"] --> DT["draft_token_N+1"] EL --> NH["predicted_hidden_N+1"] end NH --> |"继续自回归"| EH2["下一步 EAGLE Head"] EH2 --> |"× K 步"| DKT["draft_tokens[1..K]"] style target fill:#161b22,stroke:#30363d,color:#e6edf3 style eagle fill:#1a2332,stroke:#1f6feb,color:#e6edf3

4.2 EAGLE Head 结构

EAGLE Head 结构:
  EagleMlpFc: Linear(2×hidden_size → hidden_size)
              输入 = concat(hidden_states_N, embed(token_N))
  EagleLlamaLayer: 单层 Transformer（Attention + MLP）
  LM Head: 共享 Target 模型的 lm_head（无额外参数）

参数量估算（以 Llama-3-8B 为例，32 层）:
  EAGLE Head ≈ Target 的 1/32 大小（单层 Transformer + MLP）
  → 极低推理代价，draft_overhead_ratio ≈ 5%

4.3 EAGLE 推理流程

EagleProposer.propose(seq_group_metadata_list, blocks_to_swap, ...)

1. 获取 Target 模型的隐藏状态缓存
   hidden_states = model_runner.get_last_hidden_states(req_ids)

2. EAGLE Head 自回归推测 K 步
   draft_tokens = []
   h = hidden_states
   for step in range(num_spec_tokens):
       # 拼接上一步的 hidden 和 token embedding
       x = concat(h, embed(prev_token))
       # EAGLE 层前向
       h_new = eagle_layer(x, positions, kv_cache)
       # 从 LM Head 采样 draft token
       draft_logits = lm_head(h_new)
       draft_token = sample(draft_logits)
       draft_tokens.append(draft_token)
       h = h_new

3. 返回 draft_tokens 给 Scheduler

4.4 EAGLE-2 改进：动态 Draft Tree

EAGLE-2 用动态树形推测替代链式推测，在置信度低时探索多条路径， 置信度高时聚焦最优路径，进一步提升平均接受 token 数。

链式（EAGLE-1）vs 树形（EAGLE-2）

flowchart LR subgraph chain["EAGLE-1 (链式)"] direction LR C0[ctx] --> C1[d1] --> C2[d2] --> C3[d3] --> C4[d4] --> C5[d5] end subgraph tree["EAGLE-2 (树形)"] direction LR T0[ctx] --> T1a[d1_a] & T1b[d1_b] T1a --> T2a[d2_a] & T2b[d2_b] T2a --> T3a[d3_a] T1b --> T2c[d2_c] end style chain fill:#161b22,stroke:#30363d,color:#e6edf3 style tree fill:#1a2332,stroke:#3fb950,color:#e6edf3

树形策略优势：高置信度 token 直接验证一条链，低置信度时探索多分支， 整体接受率更高，平均每步接受 token 数更多。

5. N-gram Proposer

文件：vllm/v1/spec_decode/ngram.py

5.1 原理

在已有的 prompt+output 文本中查找 N-gram 匹配来预测后续 token，无需任何模型参数：

示例：
  已有文本: "The quick brown fox jumps over the lazy"
  最后 2 个 token: ["the", "lazy"]

  在文本中查找 "the lazy" 的位置:
    "... jumps over THE LAZY dog ..."
    ↑ 找到匹配！

  预测接下来的 tokens:
    "the lazy" → "dog" → "." → ...

  Draft: ["dog", ".", "The"]

5.2 优势与限制

维度说明
优势 零额外模型/参数；适用于重复性强的文本（代码补全、RAG 引用、翻译）； 计算开销极低（GPU 上文本内查找）
限制 需要文本有足够重复性；无法预测从未出现过的 token 序列； 对创意写作、开放域问答等场景基本无效
最佳场景 RAG 引用原文、代码补全（重复的函数调用/关键字）、文档翻译——这些场景 N-gram 接受率极高， 且无任何模型加载开销，是成本最低的推测解码方案。

6. Draft Model Proposer

用独立的小型语言模型作为 Draft，适合同系列模型配对（如 1B 替代 8B）：

配置示例:
LLM(
    model="meta-llama/Llama-3.1-8B-Instruct",   # Target
    speculative_config={
        "model": "meta-llama/Llama-3.2-1B-Instruct",  # Draft (1B 替 8B)
        "num_speculative_tokens": 5,
        "draft_tensor_parallel_size": 1,
    }
)

执行流程:
  1. Draft 模型自回归生成 K tokens（相对 Target 模型快约 5×）
  2. Target 模型并行验证所有 K tokens

最优条件:
  Draft 与 Target 同系列（分布相近，接受率高）
  Draft 小模型速度 ≥ Target 的 1/5

7. Medusa Proposer

为 Target 模型添加多个并行预测头，每个头预测未来不同位置的 token：

Medusa 结构:
  Head 1: 预测 target+1 的 token
  Head 2: 预测 target+2 的 token
  Head K: 预测 target+K 的 token

所有 Head 并行计算，代价约等于 1 次 Target forward

配置示例:
LLM(
    model="FasterDecoding/medusa-vicuna-7b-v1.3",  # 含 Medusa 头的模型
    speculative_config={
        "method": "medusa",
        "num_speculative_tokens": 3,
    }
)
注意 Medusa 需要使用专门微调过（含 Medusa 头）的模型权重，不能直接应用于任意标准模型。

8. RejectionSampler（拒绝采样器）

文件：vllm/v1/spec_decode/rejection_sampler.py

8.1 算法实现

class RejectionSampler:

    def forward(
        self,
        spec_decode_metadata: SpecDecodeMetadata,
        draft_probs: torch.Tensor | None,   # [num_reqs, K, vocab_size]
        target_logits: torch.Tensor,         # [num_reqs, K+1, vocab_size]
        sampling_metadata: SamplingMetadata,
    ) -> SamplerOutput:
        """执行 speculative decoding 的拒绝采样。"""
        # 1. 计算 Target probabilities
        target_probs = softmax(target_logits / temperature, dim=-1)

        # 2. 获取 Draft token IDs
        draft_token_ids = spec_decode_metadata.draft_token_ids  # [num_reqs, K]

        # 3. 批量拒绝采样（GPU 上并行）
        accepted_tokens = []
        for req_idx in range(num_reqs):
            accepted = []
            for k in range(K):
                t = draft_token_ids[req_idx, k]
                p_target = target_probs[req_idx, k, t]
                p_draft = draft_probs[req_idx, k, t] if draft_probs else 1.0

                r = torch.rand(1)
                if r <= p_target / p_draft:
                    accepted.append(t)
                else:
                    # 从修正分布采样一个 token
                    corrected_prob = max(target_probs[req_idx, k] - min(target_probs, p_draft), 0)
                    corrected_prob /= corrected_prob.sum()
                    t_corrected = torch.multinomial(corrected_prob, 1)
                    accepted.append(t_corrected)
                    break  # 停止处理后续 token

            # 如果所有 K 个都接受，额外从最后位置采样
            if len(accepted) == K:
                bonus_token = torch.multinomial(target_probs[req_idx, K], 1)
                accepted.append(bonus_token)

            accepted_tokens.append(accepted)

        return SamplerOutput(sampled_token_ids=accepted_tokens)

8.2 典型接受率

不同任务下，5 个 draft tokens 的典型接受情况：

任务类型 平均接受 tokens/step 接受率 特点
代码生成（Python） 3.5 – 4.5 70 – 90% 函数调用、缩进、关键字重复性高
文档翻译 3.0 – 4.0 60 – 80% 有规律的语言模式
问答（事实性） 2.5 – 3.5 50 – 70% 结构化回答有一定规律
创意写作 2.0 – 3.0 40 – 60% 多样性高，draft 命中率低

各任务典型平均接受 tokens/step（5 个 draft tokens，EAGLE）

9. 推测解码的调度集成

9.1 Scheduler 中的 Lookahead 块分配

调度器需为 draft token 预分配 KV 块，被拒绝的 token 对应块在下一步重用：

# 调度器需要为推测 token 预分配 KV 块
num_spec_tokens = 5  # 每步最多推测 5 个 token

# 在 allocate_slots 时：
num_total_tokens = num_current_tokens + num_spec_tokens
num_blocks_needed = ceil(num_total_tokens / block_size)

# 即使推测 token 可能被拒绝，也预先分配空间
# 被拒绝的 token 对应的 KV 块在下一步会被重用（或丢弃）

9.2 SpecDecodeMetadata 传递

class SpecDecodeMetadata:
    """传递给 ModelRunner 的推测解码元数据"""
    draft_token_ids: list[list[int]]        # [req_idx][draft_idx]
    num_draft_tokens: list[int]             # 每个请求的 draft 数
    cu_num_draft_tokens: torch.Tensor       # 前缀和（GPU 上）

    # 树形推测（EAGLE-2）
    draft_tree: TreeStructure | None
调度注意事项 Lookahead 块预分配会增加内存占用。若 KV cache 紧张，建议减小 num_speculative_tokens 或在高 batch size 时动态关闭推测解码。

10. 性能分析

10.1 推测解码收益模型

加速比 ≈ (mean_accepted_tokens + 1) / (1 + draft_overhead_ratio)

其中:
  mean_accepted_tokens : 平均每步接受的 draft token 数
  draft_overhead_ratio : Draft 阶段耗时 / Target 验证耗时

示例（EAGLE + Llama-3-8B，batch=1）:
  mean_accepted     = 3.2（代码生成）
  draft_overhead    ≈ 0.05（EAGLE Head 约 Target 的 5%）
  加速比            ≈ (3.2 + 1) / (1 + 0.05) ≈ 4.0×

各场景预期加速比（EAGLE，batch=1，Llama-3-8B）

10.2 适用场景推荐

场景 推荐方法 预期加速
代码补全（Python/JS） N-gram 或 EAGLE 2 – 4×
RAG 文档引用 N-gram 2 – 3×
聊天对话 EAGLE 1.5 – 2.5×
创意写作 EAGLE 1.2 – 1.8×
批量推理（large batch） 不推荐 可能无效或负向

关键限制 推测解码仅适用于 内存带宽瓶颈（small batch）场景。 当 batch size 增大，Target 模型进入计算瓶颈，draft 验证代价无法被摊薄， 加速比趋近于 1 甚至小于 1。生产部署前务必针对实际 batch size 进行 benchmark。
