vLLM Attention Backend 与分布式并行深度解析

TYPE: REPO ANALYSIS DATE: 2026-04-10 TOPIC: Attention / Distributed Parallelism BASE: vLLM v1 架构

TL;DR

  • vLLM 通过 AttentionBackend 抽象层统一管理 FlashAttention、FlashInfer、Triton 等多种注意力后端，根据硬件平台和模型类型自动选择，NVIDIA 默认使用 FlashInfer。
  • PagedAttention 的核心是 reshape_and_cache CUDA 内核（按 slot_mapping 散写 KV）+ FlashDecoding 分块规约，支持非连续 KV Cache 块的高效 Decode。
  • TP 每层产生 2 次 AllReduce（O Proj + Down Proj），通过 Custom AllReduce / NVLink 将延迟压缩至 0.5–1 µs；PP 通信量极小（仅传激活值），适合跨节点大模型。
  • EP（Expert Parallelism）依赖 All-to-All 分发/收集 token，支持 DeepEP、FlashInfer NVLink、NIXL RDMA 等多种后端，专为 MoE 模型（DeepSeek-V3、Mixtral）设计。
  • P/D 分离（KV Transfer）通过 KVConnector 接口实现，Prefill 集群完成计算后将 KV Cache 网络传输给 Decode 集群，两者可独立扩缩容。

1. Attention Backend 概述

vLLM 通过 AttentionBackend 抽象层将注意力计算与上层推理逻辑解耦，支持多种硬件和算法实现。GPUModelRunner 统一调用 attn_backend.forward(q, k, v, kv_cache, attn_metadata)，具体后端在运行时动态选择。

1.1 后端选择逻辑

选择优先级：用户强制指定 > MLA > Mamba > 硬件自动（NVIDIA 默认 FlashInfer，AMD 默认 ROCm FA，CPU 用 CPUAttention）。

# vllm/v1/attention/selector.py

def get_attn_backend(
    head_size: int,
    dtype: torch.dtype,
    block_size: int,
    use_mla: bool = False,
    use_mamba: bool = False,
    attention_backend: str | None = None,  # 用户指定
) -> type[AttentionBackend]:

    # 用户强制指定
    if attention_backend:
        return _get_backend_by_name(attention_backend)

    # MLA（DeepSeek V2/V3）
    if use_mla:
        return MlaBackend

    # Mamba SSM
    if use_mamba:
        return Mamba1Backend or Mamba2Backend

    # 硬件自动选择
    return current_platform.get_attn_backend_cls(
        head_size, dtype, block_size
    )
    # NVIDIA: FlashInfer（默认）或 FlashAttention
    # AMD: RocmFlashAttention
    # CPU: CPUAttention

2. AttentionBackend 抽象接口

文件路径：vllm/v1/attention/backends/ 各文件。所有后端实现以下抽象基类。

class AttentionBackend(ABC):
    """注意力后端抽象基类"""

    @staticmethod
    @abstractmethod
    def get_name() -> str:
        """后端名称，如 'flash_attn', 'flashinfer'"""

    @staticmethod
    @abstractmethod
    def get_metadata_cls() -> type[AttentionMetadata]:
        """返回该后端使用的 AttentionMetadata 子类"""

    @staticmethod
    @abstractmethod
    def get_builder_cls() -> type[AttentionMetadataBuilder]:
        """返回构建 AttentionMetadata 的 Builder 类"""

    @staticmethod
    @abstractmethod
    def get_kv_cache_shape(
        num_blocks: int,
        block_size: int,
        num_kv_heads: int,
        head_size: int,
    ) -> tuple[int, ...]:
        """返回该后端所需的 KV Cache 张量形状"""

    @abstractmethod
    def forward(
        self,
        query: torch.Tensor,        # [total_tokens, num_heads, head_size]
        key: torch.Tensor,          # [total_tokens, num_kv_heads, head_size]
        value: torch.Tensor,        # [total_tokens, num_kv_heads, head_size]
        kv_cache: torch.Tensor,     # 该层的 KV Cache 张量
        attn_metadata: AttentionMetadata,
        output: torch.Tensor | None = None,
    ) -> torch.Tensor:
        """执行注意力计算，返回 attn_output [total_tokens, num_heads, head_size]"""

2.1 CommonAttentionMetadata

所有后端共用的基础元数据，包含 token 分割、KV Cache 寻址等关键信息。

class CommonAttentionMetadata:
    """所有后端共用的基础元数据"""

    # Query 起始位置（用于区分每个请求的 tokens）

    # 每个请求的完整序列长度（含历史 KV Cache）

    # 批次统计

    num_actual_tokens: int          # 本步实际处理的 token 数
    max_query_len: int

    # Prefill/Decode 分割
    num_prefills: int
    num_decode_reqs: int

    # KV Cache 寻址
    block_table: torch.Tensor       # [num_reqs, max_blocks]
    slot_mapping: torch.Tensor      # [total_tokens]，每 token 对应的 KV 槽位

3. 各 Attention 后端实现

3.1 FlashAttention 后端

使用 FlashAttention-2/3 算法（Tri Dao et al.），IO 感知：通过分块计算避免 O(N²) 显存，只需 O(N) 显存。支持 GQA。NVIDIA Hopper 上可使用 FA3 获得更高性能。

KV Cache 布局：shape = [num_layers, 2, num_blocks, block_size, num_kv_heads, head_size]

FlashAttention forward 执行流程

Prefill attention\nflash_attn_varlen_func(q_prefill, k_prefill, v_prefill)\n新 token ↔ 新 token · causal mask · 变长序列批处理"] B --> D["3. 3.2 FlashInfer 后端

比 FlashAttention 更灵活，支持更多场景（GQA、MQA）。有专用的 Paged Decode 实现，Hopper+ 上有 FP8 KV Cache 优化。需要在推理前预分配 Workspace 缓冲区（wrapper.begin_forward(...)）。

4. PagedAttention 内核实现

文件：csrc/attention/，vllm/v1/attention/ops/paged_attn.py

4.1 CUDA 内核：reshape_and_cache

每个推理步都调用此内核，将新 token 的 K/V 按 slot_mapping 散写到 KV Cache。

// csrc/attention/attention_kernels.cu
// 将新 token 的 K/V 按 slot_mapping 散写到 kv_cache

template <typename scalar_t>
__global__ void reshape_and_cache_kernel(
    const scalar_t* __restrict__ key,         // [num_tokens, num_kv_heads, head_size]
    const scalar_t* __restrict__ value,
    scalar_t* __restrict__ key_cache,         // [num_blocks, block_size, num_kv_heads, head_size]
    scalar_t* __restrict__ value_cache,
    const int64_t* __restrict__ slot_mapping, // [num_tokens]
    ...
) {
    int token_idx = blockIdx.x;
    int slot = slot_mapping[token_idx];        // 物理槽位
    int block_idx = slot / BLOCK_SIZE;
    int block_offset = slot % BLOCK_SIZE;

    // 写入 K
    key_cache[block_idx][block_offset] = key[token_idx];
    // 写入 V
    value_cache[block_idx][block_offset] = value[token_idx];
}

4.2 Paged Decode Attention

读取非连续 KV Cache 并计算注意力（Decode 阶段核心）。使用 FlashDecoding 分块计算，最后规约所有块的部分结果。

# 输入:
#   q [num_decode_reqs, num_heads, head_size]  ← 新 query
#   kv_cache [num_blocks, 2, block_size, num_kv_heads, head_size]
#   block_table [num_decode_reqs, max_blocks]  ← 逻辑→物理块映射

for each decode request r:
    seq_len = seq_lens[r]
    for block_slot in range(ceil(seq_len / block_size)):
        block_id = block_table[r, block_slot]
        k_block = kv_cache[block_id, 0, :, :, :]  # [block_size, num_kv_heads, head_size]
        v_block = kv_cache[block_id, 1, :, :, :]

        # 使用 FlashDecoding 分块计算，最后规约
        partial_attn_out, partial_lse = flash_decode_kernel(q[r], k_block, v_block)

    # 规约所有块的部分结果
    final_out = merge_attn_states(partial_outputs, partial_lses)

# 输出:
#   attn_output [num_decode_reqs, num_heads, head_size]

5. Context Parallelism（CP）Attention

当序列极长（如 128K tokens）时，单 GPU 无法处理整个 Attention，CP 将序列分割到多 GPU 并行计算。

CP Attention 计算 — 128K tokens，4 个 CP rank

graph LR subgraph rank0["CP Rank 0 · token 0~32K"] Q0["Q 本地切片"] KV0["K,V 来自所有 rank\n(AllGather)"] O0["局部 Attn Output"] end subgraph rank1["CP Rank 1 · token 32K~64K"] Q1["Q 本地切片"] KV1["K,V 来自所有 rank\n(AllGather)"] O1["局部 Attn Output"] end subgraph rank2["CP Rank 2 · token 64K~96K"] Q2["Q 本地切片"] O2["局部 Attn Output"] end subgraph rank3["CP Rank 3 · token 96K~128K"] Q3["Q 本地切片"] O3["局部 Attn Output"] end Q0 --> O0 KV0 --> O0 Q1 --> O1 KV1 --> O1 Q2 --> O2 Q3 --> O3 note["各 rank 使用 Triton Kernel\n修正 LSE 合并部分结果\nvllm/v1/attention/ops/common.py:\n_correct_attn_cp_out_kernel"]
设计要点 每个 rank 持有 Q 的一个切片，AllGather 收集所有 rank 的 K,V；Causal mask 保证每个 rank 只 attend 到 ≤ 自己位置的 KV；不需要 AllReduce（每个 rank 的 output 已是正确的局部结果），通过 Triton Kernel 修正 LSE（Log Sum Exp）合并部分结果。

6. MLA（Multi-head Latent Attention）

适用：DeepSeek-V2/V3 等使用 MLA 的模型。文件：vllm/v1/attention/backends/mla.py，csrc/attention/mla/

6.1 MLA 原理：KV Cache 压缩

核心优势 标准 MHA 每 token KV 占用：2 × num_kv_heads × head_size = 2 × 8 × 128 = 2048 floats。MLA 压缩后每 token KV 占用：kv_lora_rank + qk_rope_head_dim = 512 + 64 = 576 floats。节省比约 3.6×，KV Cache 压缩 3–4 倍。
方案KV 缓存内容per-token 大小压缩比
标准 MHAK [seq_len, num_kv_heads, head_size]
V [seq_len, num_kv_heads, head_size]		2048 floats1×
MLA压缩 KV c_kv [seq_len, kv_lora_rank]
RoPE Key k_pe [seq_len, qk_rope_head_dim]		576 floats3.6×

8. Tensor Parallelism（TP）

将每层权重矩阵按列/行切分到多个 GPU，各 GPU 并行计算，通过 AllReduce 合并结果。

8.1 列并行（ColumnParallelLinear）

用于 Q/K/V Projection、MLP Gate/Up Projection。无需通信，每个 GPU 已有完整局部输出。

# 输入 X: [total_tokens, hidden_size]  （所有 GPU 相同）
# 权重 W: [hidden_size, output_size]
# 按列切分，每个 GPU 持有 W_i: [hidden_size, output_size/TP]

GPU 0: Y_0 = X × W_0^T  → [total_tokens, output_size/TP]
GPU 1: Y_1 = X × W_1^T  → [total_tokens, output_size/TP]
...
GPU N: Y_N = X × W_N^T

结果: Y = concat(Y_0, Y_1, ..., Y_N) → [total_tokens, output_size]
# 无需通信，每个 GPU 已有完整局部输出

8.2 行并行（RowParallelLinear）

用于 O Projection、MLP Down Projection。需要 AllReduce 合并各 GPU 的部分和。

# 每个 GPU 持有 X_i: [total_tokens, input_size/TP]
# 权重按行切分，每个 GPU 持有 W_i: [input_size/TP, output_size]

GPU 0: partial_Y_0 = X_0 × W_0  → [total_tokens, output_size]  （部分和）
GPU 1: partial_Y_1 = X_1 × W_1
...

AllReduce: Y = Y_0 + Y_1 + ... + Y_N  → [total_tokens, output_size]
# 每个 GPU 最终持有完整的 Y

8.3 GQA 下的 TP

# GQA（Group Query Attention，如 Llama-3-8B: 8 KV heads，32 Q heads）
# TP=4 时:
GPU 0: Q heads 0-7,   KV heads 0-1
GPU 1: Q heads 8-15,  KV heads 2-3
GPU 2: Q heads 16-23, KV heads 4-5
GPU 3: Q heads 24-31, KV heads 6-7

每个 GPU 的 KV heads 数 = max(1, num_kv_heads / TP)

9. Pipeline Parallelism（PP）

将模型的 Transformer 层分成多个 Stage，每个 Stage 在不同 GPU 上运行。相邻 Stage 之间传递 IntermediateTensors（中间激活值），通信量为 batch_size × hidden_size，远小于 KV Cache。

PP 数据流 — 32 层模型，PP=3

sequenceDiagram participant S0 as Stage 0 (GPU 0)
Layer 0-7 participant S1 as Stage 1 (GPU 1)
Layer 8-15 participant S2 as Stage 2 (GPU 2)
Layer 16-23 (last) S0->>S0: Prefill [Layer 0-7] S0->>S1: send(IntermediateTensors) S1->>S1: [Layer 8-15] S1->>S2: send(IntermediateTensors) S2->>S2: [Layer 16-23] + logits + sample S2-->>S1: pp_broadcast(sampled_tokens) S1-->>S0: pp_broadcast(sampled_tokens)

9.1 IntermediateTensors 与 forward 逻辑

# vllm/sequence.py
class IntermediateTensors:
    """Pipeline 阶段间传递的中间激活"""
    tensors: dict[str, torch.Tensor]  # 通常包含 "hidden_states"

def forward(input_ids, ..., intermediate_tensors=None):
    if intermediate_tensors is None:
        x = embed_tokens(input_ids)      # 第一个 Stage
    else:
        x = intermediate_tensors["hidden_states"]  # 非第一 Stage

    for layer in self.layers:
        x = layer(x)

    if not is_last_stage:
        return IntermediateTensors({"hidden_states": x})  # 传给下一 Stage
    else:
        return lm_head(x)                # 最后 Stage：计算 logits

9.2 PP Broadcast 采样结果

因为只有最后一个 Stage 才有 logits，采样结果需要广播给所有 Stage：

# vllm/v1/worker/gpu/pp_utils.py

def pp_broadcast(sampled_token_ids, num_sampled, num_rejected):
    """将采样结果从最后一个 PP stage 广播给所有 stage"""
    combined = torch.cat([sampled_token_ids, num_sampled, num_rejected])
    pp_group = get_pp_group()
    torch.distributed.broadcast(combined, src=pp_group.last_rank,
                                 group=pp_group.device_group)

10. Data Parallelism（DP）

每个 DP rank 持有完整的模型副本，各自处理不同的请求子集，DP 之间无通信，是最容易线性扩展吞吐的方式。

DP=2，TP=4（共 8 GPU）配置

10.1 DP 批次协调

DP 各 rank 可能有不同数量的请求，需协调 CUDAGraph 模式确保所有 rank 使用相同的 dispatch key：

# vllm/v1/worker/dp_utils.py

def coordinate_batch_across_dp(
    num_tokens_unpadded: int,
    parallel_config: ParallelConfig,
    ...
) -> tuple[bool, torch.Tensor | None, int]:
    """在所有 DP rank 之间同步 batch 信息，确保相同 CUDAGraph dispatch key。"""
    dp_group = get_dp_group()

    # 每个 rank 广播自己的 num_tokens
    local_info = torch.tensor([num_tokens_unpadded, cudagraph_mode])
    all_info = torch.zeros(dp_group.world_size, 2)
    dist.all_gather_into_tensor(all_info, local_info, group=dp_group.device_group)

    # 协调一致的 batch padding 和 graph mode
    return should_ubatch, num_tokens_across_dp, synced_mode

11. Expert Parallelism（EP）

EP 专为 MoE（Mixture of Experts） 模型设计，如 DeepSeek-V3、Mixtral。对于每个 token，Router 选择 Top-K 个专家处理，EP 将专家分布到不同 GPU。

# MoE 层结构: N=256 个专家，EP=8
EP Rank 0: Expert 0-31
EP Rank 1: Expert 32-63
...
EP Rank 7: Expert 224-255

12. Context Parallelism（CP）

CP 针对超长序列（如 128K tokens），将序列按 token 维度切分到多 GPU。每个 rank 的 Attention 需要访问所有 token 的 KV，通过 AllGather 收集。

# 128K 序列，CP=4:
CP Rank 0: token 0 ~ 32K
CP Rank 1: token 32K ~ 64K
CP Rank 2: token 64K ~ 96K
CP Rank 3: token 96K ~ 128K

# 通信量 = 4 × local_KV_size = 全局 KV 大小
# 但通信在 Attention 计算时发生，可被 Pipeline 部分隐藏

12.1 Ring Attention（CP 优化）

Ring Attention vs 传统 CP 传统 CP：AllGather 全部 KV 再计算 → 通信 → 等待 → 计算，延迟高。Ring Attention：CP Rank 0 计算本地 QK 的同时，异步传输 KV 到相邻 rank；收到 KV 后继续计算，再发送下一块。通信与计算重叠，延迟更低。

13.2 AllReduce 选择优先级

# CudaCommunicator.all_reduce() 优先级:
def all_reduce(input_):
    # 1. NCCL 对称内存 AllReduce（最快，需要 NCCL >= 2.26 + Hopper+）
    if should_nccl_symm_mem_allreduce(world_size, input_):
        return torch.ops.vllm.all_reduce_symmetric_with_copy(input_)

    # 2. QuickAllReduce（AMD MI3xx 专用）
    if qr_comm.should_quick_allreduce(input_):
        return qr_comm.quick_all_reduce(input_)

    # 3. FlashInfer AllReduce
    if fi_ar_comm and not fi_ar_comm.disabled:
        return fi_ar_comm.all_reduce(input_)

    # 4. Custom AllReduce（小规模 TP，延迟更低）
    if ca_comm and not ca_comm.disabled:
        return ca_comm.custom_all_reduce(input_)

    # 5. PyNCCL（标准，兜底）
    return pynccl_comm.all_reduce(input_)

13.3 Custom AllReduce 原理

为什么需要 Custom AllReduce 标准 NCCL AllReduce 对于小张量（TP 时的 hidden_states 较小）延迟主要由 GPU 间协调开销决定（1–5 µs）。Custom AllReduce 利用 NVLink 直接内存访问，不通过 NCCL 主通道，延迟降低到 0.5–1 µs。仅适用于 TP ≤ 8、同节点场景。实现文件：csrc/quickreduce/（AMD QuickReduce），vllm/distributed/device_communicators/custom_all_reduce.py。

15.2 性能权衡表

并行类型 内存效益 吞吐提升 通信开销 适用场景
TP 高（权重切分） 中（受通信限制） 每层 2 次 AllReduce 单节点大模型
PP 高（权重切分） 高（流水线） 低（段间激活） 多节点大模型
DP 无（全副本） 极高（线性） 无 高吞吐服务

CP 无 中（序列分割） 高（KV AllGather） 超长序列
